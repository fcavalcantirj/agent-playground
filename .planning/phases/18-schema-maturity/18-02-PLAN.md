### Draft-2020-12 oneOf + $ref idiom

For D-01 the top-level oneOf wraps the versioned body. The body lives in $defs/v0_1. The outer schema keeps $schema, $id, title, description, then declares only:

  "type": "object",
  "oneOf": [ { "$ref": "#/$defs/v0_1" } ],
  "$defs": {
    "v0_1": { /* the full object: type, required, additionalProperties, properties, allOf */ },
    "category": { "type": "string", "enum": [ ...11 values... ], "description": "..." }
  }

This is the Kubernetes CRD versioning idiom. Adding v0.2 later = append $defs/v0_2 + add {"$ref": "#/$defs/v0_2"} to oneOf. No wire-format break.

IMPORTANT: because JSON Schema oneOf with a discriminator requires exactly-one-matches semantics, and today only v0.1 is defined, oneOf effectively reduces to $ref today. The seam is shape, not behavior. When v0.2 lands, each branch's body will still have "apiVersion": {"const": "ap.recipe/v0.1"} (resp. v0.2), so a recipe declaring v0.1 matches exactly one branch.
