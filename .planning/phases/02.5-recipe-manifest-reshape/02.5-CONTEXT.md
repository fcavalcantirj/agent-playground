# Phase 02.5: Recipe Manifest Reshape - Context

**Gathered:** 2026-04-14
**Status:** Ready for planning

<domain>
## Phase Boundary

**Replace Phase 2's hardcoded Go recipe structs with a real YAML-based recipe manifest system**, grow the catalog from 2 agents (picoclaw + hermes) to 10 reference recipes spanning every runtime family × chat_io mode × auth mechanism in the matrix, and prove the substrate generalizes via an **N recipes × N models × ≥2 providers** matrix smoke test. Everything in service of one acceptance gate: **send `whoareyou` to any recipe × any model × through OpenRouter — it must answer.**

**Concretely, this phase ships:**

1. **Recipe YAML schema v0.1.0** (`agents/schemas/recipe.schema.json`, JSON Schema Draft 2019-09) forked from the 5 battle-tested prior arts named in `.planning/research/RECIPE-SCHEMA-DRAFT.md` — Praktor (auth), Daytona (runtime/resources), Dev Containers (6-hook lifecycle + security props), MCP (launch stdio), Plandex (providers/models), Coder (frontend metadata), OpenHands (runtime_backend enum). Nothing invented; every field traces to a prior art.

2. **YAML recipe loader** in `api/internal/recipes/` that replaces the hardcoded `recipes.AllRecipes` map. Reads `agents/<id>/recipe.yaml`, validates against the JSON Schema both client-side and server-side, materializes into an internal `Recipe` struct consumed by the session-start code path.

3. **Filesystem-based template registry.** Templates live at `agents/<id>/templates/*.tmpl` — NOT in Go code. Each template mirrors the shape of the upstream agent's canonical auth/config file (picoclaw's `.security.yml`, openclaw's `auth-profiles.json`, hermes's `cli-config.yaml`, etc.) with `{{ .secrets.<name> }}` placeholders. Adding a new agent is a pure directory drop + `recipe.yaml` write — **zero Go recompile**. This is the hard contract the Phase 8 bootstrap path depends on.

4. **5 runtime base images** published to the local image cache: `ap-runtime-node`, `ap-runtime-python`, `ap-runtime-go`, `ap-runtime-rust`, `ap-runtime-zig`. Each inherits `ap-base`'s tini/tmux/ttyd/gosu pattern from Phase 2 and adds only the language toolchain. The old `ap-picoclaw:v0.1.0-c7461f9` and `ap-hermes:v0.1.0-5621fc4` Makefile targets are deleted — those images are rebuilt as runtime-base overlays.

5. **`RunWithLifecycle` in `api/pkg/docker/runner.go`** — implements all 6 Dev Containers lifecycle hooks robustly (`initializeCommand`, `onCreateCommand`, `updateContentCommand`, `postCreateCommand`, `postStartCommand`, `postAttachCommand`) with `waitFor` readiness gating and parallel array semantics per spec. No stubs. Idempotent by contract.

6. **Chat bridge extension** — Phase 2's fifo + exec_per_message bridges are preserved; Phase 02.5 adds robust implementations for `one_shot_task` (typed input form → agent writes to `/tmp/result.json` → poll) and `http_gateway` (agent runs HTTP server on internal port, bridge proxies via Go HTTP client over container netns). Total v0.1 surface = **4 modes only** (fifo, exec_per_message, one_shot_task, http_gateway). The draft's json_rpc_stdio / terminal_only / grpc_uds modes are **cut from v0.1** — no catalog agent forces them, they'd be dead code. Schema version will bump to v0.2 when a real agent in the catalog requires one.

7. **Praktor-style secret vault resolver** — `auth.env` and `auth.files` blocks accept literal values OR `"secret:<vault-key>"` refs. `session.SecretSource` resolves refs by reading from the Phase 3 vault location (`/run/secrets/<name>` tmpfs file, injected by the entrypoint shim). In Phase 02.5, the source is still the Phase 2 dev env var pattern (`AP_DEV_BYOK_KEY`); Phase 3 swaps the source to the encrypted pgcrypto vault without touching 02.5's resolver code.

8. **10 reference recipes** ported to the new YAML format:
   - Migrated from Phase 2: **picoclaw** (node/fifo, config_flavor_of openclaw), **hermes** (python/exec_per_message, sysbox-pending, backend=local forced)
   - New: **openclaw** (node/fifo, parent for picoclaw flavor), **aider** (python/exec_per_message, gold-standard BYOK), **plandex** (go/http_gateway), **hiclaw** (go+shell/http_gateway/compose), **auto-code-rover** (python/one_shot_task), **nullclaw** (zig/fifo), **ironclaw** (rust/exec_per_message), **claude-code** (node/fifo, `policy_flags: [non_oss, oauth_required_suppressed]`, oauth blocked, ANTHROPIC_API_KEY force-injected)
   - Each recipe lives at `agents/<id>/recipe.yaml` + `agents/<id>/templates/*.tmpl` + `agents/<id>/Dockerfile` (if custom overlay on the runtime base is needed).

9. **`GET /api/recipes` HTTP endpoint** — returns the loaded catalog as JSON (public metadata only: id, name, description, license, category, runtime.family, frontend block, policy_flags, tier_badge). Supports filter query params (`?tier=strict`, `?license=oss`, `?family=python`). **API only — no frontend work in Phase 02.5.** The endpoint exists so curl/scripts can list the catalog; Phase 5 consumes it from the browser.

10. **N × N matrix smoke test** — `make smoke-test-matrix` runs a combinatorial loop: for each recipe in the v1 catalog × for each configured model × for each configured provider (Anthropic direct + OpenRouter at minimum), spin a session, `POST /api/sessions/:id/message -d '{"text":"whoareyou"}'`, assert a non-empty response, tear down. **THIS is the Phase 02.5 hypothesis proof** — not a single curl, but the full matrix. Passes = substrate generalizes. Fails = investigate per-cell.

**Explicitly NOT in Phase 02.5:**

- **No Temporal session lifecycle wiring.** Phase 02.5 HTTP handlers still call `runner.RunWithLifecycle` directly (same pattern as Phase 2, just against YAML-loaded recipes instead of hardcoded structs). Phase 5 wraps with `SessionSpawn` / `SessionDestroy` workflows without breaking the HTTP contract. User-delegated call: scope discipline over pull-forward.
- **No browser UI. No WebSocket. No xterm.js. No `web/` or `frontend/` changes.** User: *"no UI for now"*. `GET /api/recipes` ships as a JSON endpoint; frontend consumption goes to Phase 5.
- **No `web/` deletion.** That stays in Phase 3 scope per `PHASE-02.5-PREP.md`.
- **No LLM-writability test gating.** PREP flagged it as an assumption; user refocused: *"why are we deciding this now?"*. Deferred to Phase 8 bootstrap research — Phase 02.5 is not blocked by it.
- **No Sysbox install, no custom seccomp, no egress allowlist.** All deferred to Phase 7.5. Recipes that need Sysbox (hermes) declare `isolation.tier: sysbox` but run on plain docker until 7.5 lands. Phase 02.5 accepts this.
- **No content-addressable recipe hashing.** Phase 8 territory. 02.5 uses simple file-path addressing.
- **No L3 testing of gateway-only recipes.** Cody/Amp/gh-copilot/Cursor-CLI are not in the 10.
- **No bootstrap path.** Phase 8.

**Mission context:** Phase 02.5 is the substrate every future phase depends on for multi-agent support. Phase 3 injects real encrypted secrets into the Praktor-style `secret:` resolver 02.5 builds. Phase 5 wraps 02.5's session-start code path in Temporal workflows. Phase 7.5 fills the `isolation.*` knobs 02.5 plumbs through. Phase 8's bootstrap path emits recipes into the very directory layout 02.5 defines. If 02.5 gets the shape wrong, everything downstream pays the cost.

## Implementation Decisions

### Scope + acceptance

- **D-01:** Phase 02.5's hypothesis proof is **the N × N matrix smoke test**, not the schema aesthetics, not the LLM-writability test, not the planning documents. The phase is complete when `make smoke-test-matrix` runs the full catalog × configured models × ≥2 providers (Anthropic direct + OpenRouter) and every cell returns a non-empty reply to `whoareyou`. Failures are per-cell investigations; the matrix is the contract.
- **D-02:** The canonical probe message is literally `whoareyou`. Every recipe must answer it. This is deliberately model-agnostic and agent-agnostic — no tool-call assertions, no JSON schema validation on the reply, just "did the agent reply at all". Passes = the substrate forwards messages correctly end-to-end. User verbatim: *"send whoareyou and it must answer"*.
- **D-03:** **OpenRouter is a first-class v0.1 provider**, equal footing with Anthropic direct. Every recipe's `providers[]` list includes OpenRouter where the agent supports the `OPENROUTER_API_KEY` + `OPENROUTER_BASE_URL` env pattern (aider, ironclaw, nullclaw certainly; plandex, hiclaw likely; picoclaw/openclaw/claude-code maybe — Phase 02.5 planning verifies per-recipe). OpenRouter auth is BYOK via `secret:openrouter-api-key`.
- **D-04:** **Phase 02.5 is API-only.** No UI, no browser, no frontend code. `GET /api/recipes` is an HTTP/JSON endpoint; consumers are curl and the eventual Phase 5 frontend. User verbatim: *"no UI for now"*, *"BROWSER UI WE CAN DEFER"*.
- **D-05:** **Temporal stays in Phase 5.** Phase 02.5 HTTP handlers call `runner.RunWithLifecycle` directly against YAML-loaded recipes. Phase 5 flips internals to `client.ExecuteWorkflow` without touching the HTTP contract. User-delegated: *"if will be surely covered, you decide"*. Scope discipline wins over pull-forward — 02.5 is already large.

### Recipe schema (YAML v0.1.0)

- **D-06:** Schema is **forked, not invented.** Every field traces to one of the 5 battle-tested prior arts. Provenance table lives in `RECIPE-SCHEMA-DRAFT.md` §Provenance and is the reviewer's reference. Downstream agents do NOT get to add novel fields without explicit user sign-off.
- **D-07:** **Required fields** (≤6): `id`, `name`, `runtime.family`, `install`, `launch.cmd`, `chat_io.mode`, `isolation.tier`. Every other field defaults. Design constraint from PREP.
- **D-08:** **Closed-vocabulary enums everywhere** — `runtime.family`, `chat_io.mode`, `install.type`, `auth.mechanism`, `isolation.tier`, `policy_flags`, `category` are all closed enums. LLM picks from a list; the Go validator rejects unknowns. Design constraint from PREP.
- **D-09:** **JSON Schema ships alongside the YAML** at `agents/schemas/recipe.schema.json` (JSON Schema Draft 2019-09, hand-written, ~400 lines). Both client-side (Go YAML loader) and server-side (HTTP POST /api/recipes body validator, when we add it later) validate against the same file. Single source of truth, not generated from Go struct.
- **D-10:** **v0.1 `chat_io.mode` enum = {fifo, exec_per_message, one_shot_task, http_gateway}** — 4 modes only. The draft's json_rpc_stdio / terminal_only / grpc_uds are **cut from v0.1 entirely** (not stubbed, not documented-but-unsupported — removed). When a future catalog agent forces one, the schema bumps to v0.2 and the mode is implemented robustly. No dead code, no "unsupported enum values", no 501 responses. Matches the user's "no stubs, no mocks, best robust only" rule.
- **D-11:** **v0.1 `auth.mechanism` enum** strips the 5 explicitly-unsupported mechanisms from PREP (`oauth_flow_with_keychain`, `oauth_device_code`, `interactive_wizard`, `exec_time_only` if no v0.1 recipe uses it, and any other "UNSUPPORTED v1" from the draft). Enum keeps only mechanisms that a v0.1 catalog recipe actually uses. Same "no stubs" rule as D-10. Planning verifies per-recipe which mechanisms make the cut.
- **D-12:** **`config_flavor_of`** inheritance: a flavor recipe inherits parent's `install`, `lifecycle`, `runtime` (base image), and `providers` by default, and overrides `launch.cmd`, `auth.files`, `auth.env`, `isolation`, `persistent_state`, `frontend.metadata` if it declares them. **Flavors cannot redeclare `install`** — the YAML loader rejects that combination with a clear error. **Flavors cannot chain** (no flavor-of-flavor in v0.1) — the loader rejects that too. picoclaw-of-openclaw is the canonical test case.
- **D-13:** **Schema lives at `agents/schemas/recipe.schema.json`**, recipes live at `agents/<id>/recipe.yaml`, templates at `agents/<id>/templates/*.tmpl`, per-recipe Dockerfiles at `agents/<id>/Dockerfile` when needed. Single `agents/` directory is the source of truth for the catalog. Phase 8 bootstrap writes into `agents/community/<id>/...`. PREP-consistent.

### Template registry (filesystem, not Go code)

- **D-14:** **Templates live on disk at `agents/<id>/templates/*.tmpl`**, NOT in a Go-keyed registry, NOT inline in recipe.yaml. The YAML loader reads template files at session-start time using Go's `text/template` engine. **Adding a new agent = pure directory drop + `recipe.yaml` write. Zero Go recompile.** This is the hard contract that D-25 from Phase 2 locks in: *"recipe abstraction must accept new agents as recipe YAML + Dockerfile, no code changes to ap-base, runner.go, or session handlers"*. User verbatim during discuss: *"we also agreed in a folder, on the repo, with the template for the agents"*.
- **D-15:** **Template body mirrors the upstream agent's canonical auth/config file shape**, not a shape we invent. For picoclaw we copy the `.security.yml` structure from upstream; for openclaw we copy `auth-profiles.json`; for hermes we copy `cli-config.yaml`. Placeholders are `{{ .secrets.<vault-key-name> }}`, `{{ .env.<var-name> }}`, `{{ .recipe.<path> }}`. User verbatim: *"each agent must probably have a pattern on its docs, right?"* — yes, and we mirror it rather than re-deriving.
- **D-16:** **Template security:**
  - Path allowlist: loader only reads files under `agents/<id>/templates/` for the current recipe being materialized. No `../` traversal. Symlinks rejected. Template path patterns validated with a strict regex (`^[a-z0-9][a-z0-9_-]*\.tmpl$`).
  - Template sandbox: custom `template.FuncMap` that EXCLUDES all exec-style helpers. Allowed funcs: `default`, `quote`, `lower`, `upper`, `trim`. No `exec`, no `include` (prevents recursive template inclusion DoS), no `printf` with user-controlled format strings.
  - Renderer runs in a `context.WithTimeout(ctx, 5s)` to prevent runaway templates. Output size capped at 64 KiB.
  - Rendered template content is written to container tmpfs via the Phase 2 entrypoint-shim pattern; never to host disk, never to logs.
- **D-17:** **Template caching:** parsed templates are cached in-process keyed by `(recipe_id, template_name, file_mtime)`. Cache invalidates on mtime bump. Avoids re-parsing on every session spawn. Not worth Redis-backing in v0.1 — in-proc is fine for single-host deployment.

### Runtime base images (5 families)

- **D-18:** **5 runtime base images**, not 1-per-agent: `ap-runtime-node`, `ap-runtime-python`, `ap-runtime-go`, `ap-runtime-rust`, `ap-runtime-zig`. Each FROM `ap-base:v0.1.0` and adds only the language toolchain at pinned versions. Image tag format: `ap-runtime-<family>:v0.1.0-<runtime-version>`. Example: `ap-runtime-python:v0.1.0-3.13`. Built via `make build-runtimes` before `make build-recipes`. Locked from PREP decision 7.
- **D-19:** **`ap-runtime-python` base is Python 3.13 + `uv`** (not 3.11 — Phase 2 corrected this, Hermes upstream uses 3.13 via `ghcr.io/astral-sh/uv:0.11.6-python3.13-trixie`). `uv` is the package manager — pip is disallowed in python recipes to keep install times under 30s. Recipes that declare `install.type: pip` are translated by the loader to `uv pip install` in the `postCreateCommand` hook.
- **D-20:** **Node base is Node 22 LTS** via `node:22-slim` (Debian-based, NOT Alpine — Phase 2's D-04 lock: Alpine's musl breaks too many Python wheels and Node native modules).
- **D-21:** **Go, Rust, Zig base images** are thin — just the toolchain + tini + gosu + tmux + ttyd on top of `ap-base`. Versions: Go 1.25 (matches api/), Rust stable (1.84 at time of plan), Zig 0.14 (stable). Phase 02.5 planning pins exact versions at plan-write time.
- **D-22:** **Recipe overlays inherit from runtime bases, not from `ap-base` directly.** A recipe's Dockerfile (if it has one) starts `FROM ap-runtime-python:v0.1.0-3.13`, not `FROM ap-base:v0.1.0`. This enforces the runtime family / base image invariant the YAML loader uses. Recipes without a custom Dockerfile (install is pure `pip`/`npm`/`cargo` via lifecycle hooks) run directly on the runtime base image.
- **D-23:** **Size target:** `ap-runtime-python` should land under 1.5 GB (down from Phase 2's `ap-hermes:5.54 GB`). `ap-runtime-node` under 800 MB. These are planning targets, not hard gates — if we can't hit them without breaking recipes, we document the delta and move on. User from PREP: *"Hermes drops from 5.54 GB → ~1.2 GB as a side effect"*.

### Lifecycle hooks (all 6, robust)

- **D-24:** **Implement all 6 Dev Containers lifecycle hooks in `runner.RunWithLifecycle`**: `initializeCommand`, `onCreateCommand`, `updateContentCommand`, `postCreateCommand`, `postStartCommand`, `postAttachCommand`. No stubs. No "implement the 2 we use and defer the rest". Full spec compliance. User rule: *"no stubs. no mocks. best robust only"*.
- **D-25:** **Hook ordering strictly per Dev Containers spec:**
  1. `initializeCommand` — runs on HOST before container start (rare; GPU probes, host-side prep)
  2. `onCreateCommand` — runs once on container create (fresh install only)
  3. `updateContentCommand` — runs on every session start (pull updates)
  4. `postCreateCommand` — runs after create (installs go here)
  5. `postStartCommand` — runs on every start (daemonize agents here)
  6. `postAttachCommand` — runs when chat bridge attaches (bridge setup)
- **D-26:** **`waitFor` readiness gating:** every recipe declares which hook blocks session readiness. Default = `postCreateCommand`. `runner.RunWithLifecycle` returns the session as ready only after the declared `waitFor` hook completes successfully. If the hook exits non-zero, the whole session errors out and the container is destroyed.
- **D-27:** **Parallel array semantics:** a hook can be a string (single command) OR an array of strings (sequential commands) OR an array-of-arrays (parallel command groups). Example: `postCreateCommand: [["apt-get", "install", "git"], ["uv", "pip", "install", "aider"]]` runs the two groups in parallel. Go implementation uses `errgroup.WithContext`.
- **D-28:** **Hooks are idempotent by contract.** The recipe author declares idempotency — the loader doesn't verify. On session restart, `onCreateCommand` is skipped, `postCreateCommand` is re-run. Recipes that can't be idempotent in `postCreateCommand` should move the non-idempotent work to `onCreateCommand`.
- **D-29:** **Hook execution environment:** hooks run inside the container as the agent user (non-root, post-gosu drop), with the Phase 2 sandbox posture (cap_drop ALL, no_new_privs, read_only_rootfs except tmpfs, default seccomp). Secrets from the vault are NOT available during `onCreateCommand` / `updateContentCommand` (the vault is only populated after `postCreateCommand`) — recipes that need secrets at install time are rejected by the schema validator. This prevents API keys from leaking into install logs.
- **D-30:** **Timeout per hook:** each hook has a max runtime (default 10 min). Exceeded timeout fails the session. Configurable per-recipe via `lifecycle.<hook>_timeout_sec`. Rationale: large Python installs (`uv pip install -e .[all]`) can legitimately take 5+ min; bounded upper limit prevents zombie builds.

### Session lifecycle (non-Temporal, per D-05)

- **D-31:** **HTTP handlers call `runner.RunWithLifecycle` directly.** Same shape as Phase 2's direct-runner pattern, just against YAML-loaded `Recipe` instances instead of hardcoded `recipes.Picoclaw` / `recipes.Hermes` structs. `POST /api/sessions` → load recipe by id → materialize → `runner.RunWithLifecycle` → return session id.
- **D-32:** **`POST /api/sessions/:id/message`** routing: session row carries `chat_io.mode`. Handler dispatches to one of 4 bridge implementations based on the mode:
  - `fifo` → Phase 2's existing `docker exec` FIFO bridge, unchanged
  - `exec_per_message` → Phase 2's existing exec-per-message bridge, unchanged
  - `one_shot_task` → NEW: agent writes to `/tmp/result.json` at workspace root; handler polls every 500ms for up to `response_timeout_sec`; returns parsed JSON body
  - `http_gateway` → NEW: Go `http.Client` targets the agent's internal port via `dockerapi.ContainerInspect` → bridge network IP; bridge is stateless per-message
- **D-33:** **Response timeout defaults** per chat_io.mode: fifo = 60s, exec_per_message = 90s, one_shot_task = 900s (15 min), http_gateway = 120s. Recipe can override via `chat_io.<mode>.response_timeout_sec`.
- **D-34:** **`DELETE /api/sessions/:id`** = `runner.Stop` + `runner.Remove` + DB state update to `status='stopped'`. Same as Phase 2, no change.
- **D-35:** **Recipe reload:** the loader reads recipes at process-start (Go embed is NOT used — recipes stay on disk, hot-reloadable). In-memory cache. `SIGHUP` triggers a reload (cheap, idempotent). No HTTP endpoint to trigger reload in v0.1 — ship the signal handler only.

### Secret vault (Praktor-shape, Phase 3-compatible)

- **D-36:** **`auth.env` values** are either literal (`"vim"`) or vault refs (`"secret:anthropic-api-key"`). The session-start code recognizes the `secret:` prefix, calls `session.SecretSource.Resolve("anthropic-api-key")`, and injects the resolved value as the container env var.
- **D-37:** **`auth.files` block** (Praktor-shape) is an array of `{secret, target, mode, template}` entries. At container start, the entrypoint shim reads the resolved secret value from tmpfs `/run/secrets/<secret-name>`, renders `agents/<id>/templates/<template>.tmpl` with the secret as `.secrets.<name>` context, writes the rendered body to `<target>` with `chmod <mode>`, then execs the agent. This is the Phase 2 entrypoint-shim pattern extended to use files from the filesystem template registry (D-14) instead of baked-in Dockerfile steps.
- **D-38:** **`session.SecretSource` interface** in `api/internal/session/secrets.go` has one method: `Resolve(key string) (string, error)`. Phase 02.5 ships a `DevEnvSecretSource` implementation that reads from `AP_DEV_BYOK_KEY` and `AP_DEV_OPENROUTER_KEY` (new) and any `AP_DEV_<UPPER>_KEY` env vars. Phase 3 ships `VaultSecretSource` reading from the pgcrypto table. Dependency injection via `WithSecretSource(SecretSource)` functional option on `server.New`, following the Phase 1 pattern.
- **D-39:** **Keys never enter logs.** Log redaction middleware in `api/internal/logging/` scrubs any value that matches the secret-key regex patterns (e.g., `sk-ant-[a-zA-Z0-9-_]{50,}`, `sk-or-v1-[a-f0-9]{64}`). This is defense-in-depth against accidental logging in recipe materialization paths.

### 10 reference recipes (L3 gate per recipe)

- **D-40:** **The 10 v0.1 recipes** match PREP's locked list: openclaw, picoclaw (flavor of openclaw), aider, hermes, plandex, hiclaw, auto-code-rover, nullclaw, ironclaw, claude-code. Coverage table lives in `RECIPE-SCHEMA-DRAFT.md` §"Coverage table".
- **D-41:** **Each recipe has a dedicated plan file** in Phase 02.5 (one plan per recipe). Each plan's acceptance criterion is a per-recipe L3 round-trip: spin container → send `whoareyou` → assert non-empty reply → teardown. The plan is complete when that passes for at least one (recipe, model, provider) triple. The planner spawns these as parallel waves.
- **D-42:** **Commit SHAs for each recipe are pinned at plan-write time**, not at phase-plan time. This keeps the pins fresh. Planning writes SHAs into both `agents/<id>/recipe.yaml` `metadata.source_sha` and the Dockerfile `git checkout <sha>` line. Both must match — the YAML loader can validate.
- **D-43:** **Non-OSS recipes (claude-code)** ship with `policy_flags: [non_oss, oauth_required_suppressed]`. Frontend (Phase 5+) filters non-OSS from the default view; users can opt in. OAuth flow is blocked at session-start (`runner.RunWithLifecycle` errors out if `policy_flags` contains `oauth_required_suppressed` and no alternative auth is declared). `ANTHROPIC_API_KEY` is force-injected via `auth.env` — if the user has no BYOK key, the session refuses to start with a clear error.
- **D-44:** **Recipe validation is the L3 test, not JSON Schema alone.** A recipe that passes JSON Schema + Go loader validation but fails its per-recipe L3 round-trip does NOT count as shipped. The matrix test (D-01) catches the aggregate view.
- **D-45:** **Lazy L2 verification (per PREP decision 6):** we do NOT run `pip install` across all 10 agents upfront. Each recipe's "verify install path" happens inside that recipe's plan execution. One recipe at a time. L3 is the hard gate.

### Matrix smoke test (the acceptance gate)

- **D-46:** **`make smoke-test-matrix` implementation:**
  - Inputs: `RECIPES=$(ls agents/*/recipe.yaml)`, `MODELS=$(cat test/matrix-models.txt)`, `PROVIDERS=(anthropic openrouter)`.
  - Outer loop: for each recipe. Inner loop: for each (provider, model) pair where the recipe's `providers[]` list contains the provider.
  - Per cell: `curl POST /api/sessions` with recipe + provider + model → `curl POST /api/sessions/:id/message -d '{"text":"whoareyou"}'` → assert reply non-empty → `curl DELETE /api/sessions/:id` → assert no dangling container.
  - Output: matrix table to stdout (ASCII art), with ✓/✗/SKIP per cell. Exit non-zero if any non-SKIP cell fails.
  - Total cells: 10 recipes × ~6 models × 2 providers = up to 120 cells. Not all are valid (aider doesn't support zig, ironclaw doesn't support python models, etc. — SKIP where `models[]` intersection is empty).
- **D-47:** **Minimum matrix to pass Phase 02.5:**
  - ≥6 recipes ship working (not all 10 — some L3 tests may legitimately fail for reasons we accept, e.g., claude-code's OAuth path when we only have an ANTHROPIC_API_KEY)
  - ≥2 providers working (Anthropic direct + OpenRouter — hard floor)
  - ≥3 distinct models working per shipped recipe
  - Total green cells ≥ 36 (6 × 3 × 2)
  - This is the hard gate. Fewer than that = Phase 02.5 is not done.
- **D-48:** **`test/matrix-models.txt`** is the source of truth for which models the matrix tests against. v0.1 content: `claude-sonnet-4-6`, `claude-opus-4-6`, `claude-haiku-4-5-20251001` (Anthropic + OpenRouter routes), `gpt-5.4`, `openrouter/auto`, `deepseek/deepseek-v3-coder`. Planning expands/prunes based on what's actually BYOK-testable at the host at plan-write time.
- **D-49:** **Probe message:** literal string `"whoareyou"`. No variation. User verbatim: *"send whoareyou and it must answer"*. Assert = HTTP 200 + non-empty response body + body is not a raw error JSON (`"error"` field absent or null).
- **D-50:** **BYOK keys for the matrix test** come from env vars: `AP_DEV_BYOK_KEY` (Anthropic), `AP_DEV_OPENROUTER_KEY` (OpenRouter, NEW — Phase 02.5 adds this env var to the server config). Test skips cleanly if neither is set (for CI without secrets).

### API surface (Phase 02.5 additions)

- **D-51:** **`GET /api/recipes`** — returns JSON array of loaded recipes, public-metadata view only: `{id, name, description, license, category, runtime.family, frontend.{display_name, slug, icon, category_badge, tooltip}, policy_flags, tier_badge, models[].id, providers[].id}`.
