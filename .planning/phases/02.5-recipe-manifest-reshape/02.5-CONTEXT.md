**Gathered:** 2026-04-14
**Status:** Ready for planning

## Phase Boundary

**This is an ARCHITECTURE phase, not a catalog phase.** Its job is to find the right pattern for *"any agent × any model × any user, in one click"* and lock it in code — schema, loader, template registry, lifecycle runner, chat-bridge abstraction, session-start pipeline. The specifics of which recipes ship with the platform come later (Phase 4 or a separate 02.6 backfill). Phase 02.5 ships the **smallest possible proof of the architecture**: two architecturally-different reference recipes (aider + picoclaw) that exercise every component of the new recipe substrate end-to-end, plus an architectural soundness acceptance test where a third recipe is added to the running system as a pure directory drop — no Go code changes, no recompile, just files.

The hypothesis proof for Phase 02.5 is NOT "we shipped N recipes". It is **"the pattern we designed is robust enough that recipe N+1 is configuration, not code"**. If that test fails, the schema is wrong and we iterate. If it passes, Phase 4 can mechanically backfill the remaining 8 recipes (openclaw, hermes, plandex, hiclaw, auto-code-rover, nullclaw, ironclaw, claude-code) as pure data work, no more architectural decisions. User verbatim: *"FIND A GOOD PATTERN ARCHTECTURE TO THE VISION n agents with any model. right? the specifics of recipe, could come later"*.

**Concretely, this phase ships:**

1. **Recipe YAML schema v0.1.0** (`agents/schemas/recipe.schema.json`, JSON Schema Draft 2019-09) forked from the 5 battle-tested prior arts named in `.planning/research/RECIPE-SCHEMA-DRAFT.md` — Praktor (auth), Daytona (runtime/resources), Dev Containers (6-hook lifecycle + security props), MCP (launch stdio), Plandex (providers/models), Coder (frontend metadata), OpenHands (runtime_backend enum). Nothing invented; every field traces to a prior art.

2. **YAML recipe loader** in `api/internal/recipes/` that replaces the hardcoded `recipes.AllRecipes` map. Reads `agents/<id>/recipe.yaml`, validates against the JSON Schema both client-side and server-side, materializes into an internal `Recipe` struct consumed by the session-start code path.

3. **Filesystem-based template registry.** Templates live at `agents/<id>/templates/*.tmpl` — NOT in Go code. Each template mirrors the shape of the upstream agent's canonical auth/config file (picoclaw's `.security.yml`, openclaw's `auth-profiles.json`, hermes's `cli-config.yaml`, etc.) with `{{ .secrets.<name> }}` placeholders. Adding a new agent is a pure directory drop + `recipe.yaml` write — **zero Go recompile**. This is the hard contract the Phase 8 bootstrap path depends on.

4. **2 runtime base images** published to the local image cache: **`ap-runtime-python`** and **`ap-runtime-node`** — one per reference recipe. Each inherits `ap-base`'s tini/tmux/ttyd/gosu pattern from Phase 2 and adds only the language toolchain. The old `ap-picoclaw:v0.1.0-c7461f9` and `ap-hermes:v0.1.0-5621fc4` Makefile targets are deleted — picoclaw's image is rebuilt on `ap-runtime-node`, aider's image is rebuilt on `ap-runtime-python`. The `go / rust / zig` runtime bases are **deferred to Phase 4** along with their consuming recipes — shipping runtime bases nobody uses violates the "no stubs" rule. The **build pattern** for runtime bases is locked in 02.5 (so adding `ap-runtime-go` in Phase 4 is one more Dockerfile following the template, zero code changes).

5. **`RunWithLifecycle` in `api/pkg/docker/runner.go`** — implements all 6 Dev Containers lifecycle hooks robustly (`initializeCommand`, `onCreateCommand`, `updateContentCommand`, `postCreateCommand`, `postStartCommand`, `postAttachCommand`) with `waitFor` readiness gating and parallel array semantics per spec. No stubs. Idempotent by contract.

6. **Chat bridge abstraction + 2 working implementations** — Phase 2's `fifo` (picoclaw) and `exec_per_message` (aider) bridges are preserved and promoted from ad-hoc per-recipe code to a real `ChatBridge` interface with two concrete implementations in `api/internal/session/bridge/`. The interface is the architectural contract — adding `one_shot_task`, `http_gateway`, or any future mode is implementing one interface. v0.1 schema enum = **{fifo, exec_per_message}** only — the 2 modes the reference recipes use. `one_shot_task` and `http_gateway` are **deferred to Phase 4** (when auto-code-rover / plandex / hiclaw ship); `json_rpc_stdio` / `terminal_only` / `grpc_uds` stay cut entirely per the earlier "no stubs" rule. The abstraction is what 02.5 locks; the mode catalog grows with the recipe catalog.

7. **Praktor-style secret vault resolver** — `auth.env` and `auth.files` blocks accept literal values OR `"secret:<vault-key>"` refs. `session.SecretSource` resolves refs by reading from the Phase 3 vault location (`/run/secrets/<name>` tmpfs file, injected by the entrypoint shim). In Phase 02.5, the source is still the Phase 2 dev env var pattern (`AP_DEV_BYOK_KEY`); Phase 3 swaps the source to the encrypted pgcrypto vault without touching 02.5's resolver code.

8. **2 reference recipes** ported to the new YAML format, deliberately chosen to be architecturally maximally different:
   - **`aider`** — python / exec_per_message / env_var BYOK / pip install. The **simplest possible recipe**: no template files, no secret-file-mount magic, no long-running process, no config_flavor_of, no nested container. Agent reads `ANTHROPIC_API_KEY` or `OPENROUTER_API_KEY` from the environment, runs `aider --message "<text>"` per message. Proves the schema + loader + runtime base + exec_per_message bridge + env-var auth + OpenRouter path end-to-end.
   - **`picoclaw`** (standalone, NOT config_flavor_of openclaw in 02.5 — inheritance pattern deferred to Phase 4) — node / fifo / secret_file_mount via template registry / npm install. The **hardest path the architecture has to support**: long-running agent process, FIFO-based chat bridge, auth-file-mount rendered from a template that mirrors picoclaw's upstream `.security.yml` shape. Proves template registry + fifo bridge + file-mount auth + lifecycle hooks (agent daemonizes in `postStartCommand`, chat attaches in `postAttachCommand`).
   - These 2 recipes cover **2 runtimes, 2 chat_io modes, 2 auth mechanisms, 2 install types** — enough architectural surface to prove every component of the new substrate. They live at `agents/aider/` and `agents/picoclaw/` with their `recipe.yaml`, optional `templates/*.tmpl`, and optional per-recipe `Dockerfile` (only if the runtime base needs customization).
   - **`hermes` is NOT in Phase 02.5.** It's deferred to Phase 4 with the 7 other recipes. Rationale: Hermes is Phase 2's hardest case and was the stress test for the Phase 2 substrate, but once the new YAML pattern is proven with aider + picoclaw, Hermes becomes a data exercise (write YAML + template + Dockerfile) that Phase 4 handles along with the rest of the catalog. Phase 02.5 NOT shipping Hermes does not mean Hermes breaks — Phase 2's hardcoded `recipes.Hermes` struct can stay in parallel with the new YAML loader until Phase 4 ports it, OR we delete it cleanly if the user prefers a clean cut.

9. **`GET /api/recipes` HTTP endpoint** — returns the loaded catalog as JSON (public metadata only: id, name, description, license, category, runtime.family, frontend block, policy_flags, tier_badge). Supports filter query params (`?tier=strict`, `?license=oss`, `?family=python`). **API only — no frontend work in Phase 02.5.** The endpoint exists so curl/scripts can list the catalog; Phase 5 consumes it from the browser.

10. **Minimal matrix smoke test + architectural drop-in test** — Phase 02.5 acceptance has **two gates** that run end-to-end:

    **Gate A: `make smoke-test-matrix`** — plumbing proof. The matrix is `{aider, picoclaw} × {anthropic, openrouter}` = **up to 4 cells** (SKIP if a recipe doesn't declare a provider). Each cell uses the cheapest model from the recipe's `models[]` list for that provider (convention: first entry is cheapest — no new schema field). Per cell: spin container → send `whoareyou` → assert non-empty reply → teardown → assert no dangling container. All non-SKIP cells must pass. This proves the plumbing works for the 2 reference recipes against 2 providers on the cheapest models available.

    **Gate B: `make test-architectural-drop-in`** — the real acceptance test, and the reason Phase 02.5 exists. The test picks a third agent the project has NOT pre-written a recipe for (suggested target: **openclaw**, because we already know its shape from Phase 2; alternative: **plandex** or any fresh pick), writes its `recipe.yaml` + optional `templates/*.tmpl` + optional `Dockerfile` from scratch using only the locked schema + documentation, drops it into `agents/openclaw/`, and runs it through the same smoke test loop. **Zero Go code changes. Zero recompile. Zero new Dockerfile in `deploy/`.** If it works on first try (or with clearly-scoped fixes to the new recipe's own files, not the substrate), the architecture is good. If it requires substrate changes, we iterate on the schema until it doesn't. **Gate B is the hypothesis proof for Phase 02.5.**

    The test is implemented as a one-time shell script that can be run against a freshly-cloned repo with only the 2 reference recipes present. It's exercised manually at the end of the phase (not in CI — it's an acceptance test, not a regression test).

**Explicitly NOT in Phase 02.5:**

- **No Temporal session lifecycle wiring.** Phase 02.5 HTTP handlers still call `runner.RunWithLifecycle` directly (same pattern as Phase 2, just against YAML-loaded recipes instead of hardcoded structs). Phase 5 wraps with `SessionSpawn` / `SessionDestroy` workflows without breaking the HTTP contract. User-delegated call: scope discipline over pull-forward.
- **No browser UI. No WebSocket. No xterm.js. No `web/` or `frontend/` changes.** User: *"no UI for now"*. `GET /api/recipes` ships as a JSON endpoint; frontend consumption goes to Phase 5.
- **No `web/` deletion.** That stays in Phase 3 scope per `PHASE-02.5-PREP.md`.
- **No LLM-writability test gating.** PREP flagged it as an assumption; user refocused: *"why are we deciding this now?"*. Deferred to Phase 8 bootstrap research — Phase 02.5 is not blocked by it.
- **No Sysbox install, no custom seccomp, no egress allowlist.** All deferred to Phase 7.5. Recipes that need Sysbox (hermes) declare `isolation.tier: sysbox` but run on plain docker until 7.5 lands. Phase 02.5 accepts this.
- **No content-addressable recipe hashing.** Phase 8 territory. 02.5 uses simple file-path addressing.
- **No L3 testing of gateway-only recipes.** Cody/Amp/gh-copilot/Cursor-CLI are not in the 10.
- **No bootstrap path.** Phase 8.

**Mission context:** Phase 02.5 is the substrate every future phase depends on for multi-agent support. Phase 3 injects real encrypted secrets into the Praktor-style `secret:` resolver 02.5 builds. Phase 5 wraps 02.5's session-start code path in Temporal workflows. Phase 7.5 fills the `isolation.*` knobs 02.5 plumbs through. Phase 8's bootstrap path emits recipes into the very directory layout 02.5 defines. If 02.5 gets the shape wrong, everything downstream pays the cost.

## Implementation Decisions

### Scope + acceptance

- **D-01:** **Phase 02.5 is an architecture phase, not a catalog phase.** Its deliverable is *the pattern*, not *the recipes*. The schema, loader, template registry, lifecycle runner, chat bridge abstraction, runtime base image build system — all locked. The 2 reference recipes (aider + picoclaw) exist to exercise every component of the pattern end-to-end, not to populate the catalog. Phase 4 populates the catalog. User verbatim: *"FIND A GOOD PATTERN ARCHTECTURE TO THE VISION n agents with any model. right? the specifics of recipe, could come later"*.
- **D-01b:** **The architectural drop-in test (Gate B) is the hypothesis proof**, not the matrix (Gate A). Gate B = pick a third agent the codebase has never seen as a recipe (suggested: openclaw), write its `agents/openclaw/recipe.yaml` + optional templates + optional Dockerfile using only the locked schema + public documentation, drop it into the running system, and have it pass the same `whoareyou` smoke test as the 2 reference recipes. **Zero Go code changes, zero recompile, zero substrate edits.** If the drop-in requires the schema to grow, the lifecycle runner to handle a new case, or the bridge to add a mode — the pattern failed and we iterate until the drop-in works without substrate changes. Passing Gate B is what locks the architecture and closes Phase 02.5. Failing Gate B means extend the phase until the iteration converges.
- **D-01c:** Gate A (matrix smoke test) is **4 cells max** — `{aider, picoclaw} × {anthropic, openrouter}` — not a 36-cell floor. Its job is to prove the 2 reference recipes work against 2 providers on the cheapest valid models. Gate A is a prerequisite for Gate B but is not itself the acceptance gate. Any non-SKIP cell failing blocks phase completion.
- **D-02:** The canonical probe message is literally `whoareyou`. Every recipe must answer it. This is deliberately model-agnostic and agent-agnostic — no tool-call assertions, no JSON schema validation on the reply, just "did the agent reply at all". Passes = the substrate forwards messages correctly end-to-end. User verbatim: *"send whoareyou and it must answer"*.
- **D-02b:** **The matrix uses the cheapest valid model per cell, not a model grid.** The test's job is to prove the 10 plumbing steps below work, not to evaluate model quality. Burning expensive models on a plumbing test is wasteful. One cell = one (recipe, provider) pair = one cheapest model that the recipe supports on that provider. If the recipe's `models[]` list declares multiple models for a provider, planning picks the cheapest as the matrix cell's model; the other models are still declared in the recipe for users to pick at runtime. **The 10 plumbing steps the matrix actually proves:**
  1. YAML loader reads `agents/<id>/recipe.yaml` correctly
  2. Template registry renders `agents/<id>/templates/*.tmpl` with the right secrets
  3. Runtime base image starts, lifecycle hooks run, agent installs cleanly
  4. Entrypoint shim injects BYOK secret into the correct env var / file path
  5. Chat bridge for the recipe's `chat_io.mode` delivers the user message
  6. Agent makes an HTTP call to the provider's base URL using the agent's built-in LLM client
  7. Provider accepts the auth, routes to a model, returns a reply
  8. Agent writes the reply back through the chat bridge
  9. Go handler returns it as the HTTP response
  10. Teardown cleans up with no dangling container
  If a recipe-provider pair works with a $0.0001/request model, it'll work with Opus. That's the entire logic behind this decision.
- **D-03:** **OpenRouter is a first-class v0.1 provider**, equal footing with Anthropic direct. Every recipe's `providers[]` list includes OpenRouter where the agent supports the `OPENROUTER_API_KEY` + `OPENROUTER_BASE_URL` env pattern (aider, ironclaw, nullclaw certainly; plandex, hiclaw likely; picoclaw/openclaw/claude-code maybe — Phase 02.5 planning verifies per-recipe). OpenRouter auth is BYOK via `secret:openrouter-api-key`. OpenRouter is load-bearing because it's the cheapest path to prove the plumbing works for any agent that speaks OpenAI-compatible protocol.
- **D-04:** **Phase 02.5 is API-only.** No UI, no browser, no frontend code. `GET /api/recipes` is an HTTP/JSON endpoint; consumers are curl and the eventual Phase 5 frontend. User verbatim: *"no UI for now"*, *"BROWSER UI WE CAN DEFER"*.
- **D-05:** **Temporal stays in Phase 5.** Phase 02.5 HTTP handlers call `runner.RunWithLifecycle` directly against YAML-loaded recipes. Phase 5 flips internals to `client.ExecuteWorkflow` without touching the HTTP contract. User-delegated: *"if will be surely covered, you decide"*. Scope discipline wins over pull-forward — 02.5 is already large.

### Recipe schema (YAML v0.1.0)

- **D-06:** Schema is **forked, not invented.** Every field traces to one of the 5 battle-tested prior arts. Provenance table lives in `RECIPE-SCHEMA-DRAFT.md` §Provenance and is the reviewer's reference. Downstream agents do NOT get to add novel fields without explicit user sign-off.
- **D-07:** **Required fields** (≤6): `id`, `name`, `runtime.family`, `install`, `launch.cmd`, `chat_io.mode`, `isolation.tier`. Every other field defaults. Design constraint from PREP.
- **D-08:** **Closed-vocabulary enums everywhere** — `runtime.family`, `chat_io.mode`, `install.type`, `auth.mechanism`, `isolation.tier`, `policy_flags`, `category` are all closed enums. LLM picks from a list; the Go validator rejects unknowns. Design constraint from PREP.
- **D-09:** **JSON Schema ships alongside the YAML** at `agents/schemas/recipe.schema.json` (JSON Schema Draft 2019-09, hand-written, ~400 lines). Both client-side (Go YAML loader) and server-side (HTTP POST /api/recipes body validator, when we add it later) validate against the same file. Single source of truth, not generated from Go struct.
- **D-10:** **v0.1 `chat_io.mode` enum = {fifo, exec_per_message, one_shot_task, http_gateway}** — 4 modes only. The draft's json_rpc_stdio / terminal_only / grpc_uds are **cut from v0.1 entirely** (not stubbed, not documented-but-unsupported — removed). When a future catalog agent forces one, the schema bumps to v0.2 and the mode is implemented robustly. No dead code, no "unsupported enum values", no 501 responses. Matches the user's "no stubs, no mocks, best robust only" rule.
- **D-11:** **v0.1 `auth.mechanism` enum** strips the 5 explicitly-unsupported mechanisms from PREP (`oauth_flow_with_keychain`, `oauth_device_code`, `interactive_wizard`, `exec_time_only` if no v0.1 recipe uses it, and any other "UNSUPPORTED v1" from the draft). Enum keeps only mechanisms that a v0.1 catalog recipe actually uses. Same "no stubs" rule as D-10. Planning verifies per-recipe which mechanisms make the cut.
- **D-12:** **`config_flavor_of`** inheritance: a flavor recipe inherits parent's `install`, `lifecycle`, `runtime` (base image), and `providers` by default, and overrides `launch.cmd`, `auth.files`, `auth.env`, `isolation`, `persistent_state`, `frontend.metadata` if it declares them. **Flavors cannot redeclare `install`** — the YAML loader rejects that combination with a clear error. **Flavors cannot chain** (no flavor-of-flavor in v0.1) — the loader rejects that too. picoclaw-of-openclaw is the canonical test case.
- **D-13:** **Schema lives at `agents/schemas/recipe.schema.json`**, recipes live at `agents/<id>/recipe.yaml`, templates at `agents/<id>/templates/*.tmpl`, per-recipe Dockerfiles at `agents/<id>/Dockerfile` when needed. Single `agents/` directory is the source of truth for the catalog. Phase 8 bootstrap writes into `agents/community/<id>/...`. PREP-consistent.

### Template registry (filesystem, not Go code)

- **D-14:** **Templates live on disk at `agents/<id>/templates/*.tmpl`**, NOT in a Go-keyed registry, NOT inline in recipe.yaml. The YAML loader reads template files at session-start time using Go's `text/template` engine. **Adding a new agent = pure directory drop + `recipe.yaml` write. Zero Go recompile.** This is the hard contract that D-25 from Phase 2 locks in: *"recipe abstraction must accept new agents as recipe YAML + Dockerfile, no code changes to ap-base, runner.go, or session handlers"*. User verbatim during discuss: *"we also agreed in a folder, on the repo, with the template for the agents"*.
- **D-15:** **Template body mirrors the upstream agent's canonical auth/config file shape**, not a shape we invent. For picoclaw we copy the `.security.yml` structure from upstream; for openclaw we copy `auth-profiles.json`; for hermes we copy `cli-config.yaml`. Placeholders are `{{ .secrets.<vault-key-name> }}`, `{{ .env.<var-name> }}`, `{{ .recipe.<path> }}`. User verbatim: *"each agent must probably have a pattern on its docs, right?"* — yes, and we mirror it rather than re-deriving.
- **D-16:** **Template security:**
  - Path allowlist: loader only reads files under `agents/<id>/templates/` for the current recipe being materialized. No `../` traversal. Symlinks rejected. Template path patterns validated with a strict regex (`^[a-z0-9][a-z0-9_-]*\.tmpl$`).
  - Template sandbox: custom `template.FuncMap` that EXCLUDES all exec-style helpers. Allowed funcs: `default`, `quote`, `lower`, `upper`, `trim`. No `exec`, no `include` (prevents recursive template inclusion DoS), no `printf` with user-controlled format strings.
  - Renderer runs in a `context.WithTimeout(ctx, 5s)` to prevent runaway templates. Output size capped at 64 KiB.
  - Rendered template content is written to container tmpfs via the Phase 2 entrypoint-shim pattern; never to host disk, never to logs.
- **D-17:** **Template caching:** parsed templates are cached in-process keyed by `(recipe_id, template_name, file_mtime)`. Cache invalidates on mtime bump. Avoids re-parsing on every session spawn. Not worth Redis-backing in v0.1 — in-proc is fine for single-host deployment.

### Runtime base images (5 families)

- **D-18:** **5 runtime base images**, not 1-per-agent: `ap-runtime-node`, `ap-runtime-python`, `ap-runtime-go`, `ap-runtime-rust`, `ap-runtime-zig`. Each FROM `ap-base:v0.1.0` and adds only the language toolchain at pinned versions. Image tag format: `ap-runtime-<family>:v0.1.0-<runtime-version>`. Example: `ap-runtime-python:v0.1.0-3.13`. Built via `make build-runtimes` before `make build-recipes`. Locked from PREP decision 7.
- **D-19:** **`ap-runtime-python` base is Python 3.13 + `uv`** (not 3.11 — Phase 2 corrected this, Hermes upstream uses 3.13 via `ghcr.io/astral-sh/uv:0.11.6-python3.13-trixie`). `uv` is the package manager — pip is disallowed in python recipes to keep install times under 30s. Recipes that declare `install.type: pip` are translated by the loader to `uv pip install` in the `postCreateCommand` hook.
- **D-20:** **Node base is Node 22 LTS** via `node:22-slim` (Debian-based, NOT Alpine — Phase 2's D-04 lock: Alpine's musl breaks too many Python wheels and Node native modules).
- **D-21:** **Go, Rust, Zig base images** are thin — just the toolchain + tini + gosu + tmux + ttyd on top of `ap-base`. Versions: Go 1.25 (matches api/), Rust stable (1.84 at time of plan), Zig 0.14 (stable). Phase 02.5 planning pins exact versions at plan-write time.
- **D-22:** **Recipe overlays inherit from runtime bases, not from `ap-base` directly.** A recipe's Dockerfile (if it has one) starts `FROM ap-runtime-python:v0.1.0-3.13`, not `FROM ap-base:v0.1.0`. This enforces the runtime family / base image invariant the YAML loader uses. Recipes without a custom Dockerfile (install is pure `pip`/`npm`/`cargo` via lifecycle hooks) run directly on the runtime base image.
- **D-23:** **Size target:** `ap-runtime-python` should land under 1.5 GB (down from Phase 2's `ap-hermes:5.54 GB`). `ap-runtime-node` under 800 MB. These are planning targets, not hard gates — if we can't hit them without breaking recipes, we document the delta and move on. User from PREP: *"Hermes drops from 5.54 GB → ~1.2 GB as a side effect"*.

### Lifecycle hooks (all 6, robust)

- **D-24:** **Implement all 6 Dev Containers lifecycle hooks in `runner.RunWithLifecycle`**: `initializeCommand`, `onCreateCommand`, `updateContentCommand`, `postCreateCommand`, `postStartCommand`, `postAttachCommand`. No stubs. No "implement the 2 we use and defer the rest". Full spec compliance. User rule: *"no stubs. no mocks. best robust only"*.
- **D-25:** **Hook ordering strictly per Dev Containers spec:**
  1. `initializeCommand` — runs on HOST before container start (rare; GPU probes, host-side prep)
  2. `onCreateCommand` — runs once on container create (fresh install only)
  3. `updateContentCommand` — runs on every session start (pull updates)
  4. `postCreateCommand` — runs after create (installs go here)
  5. `postStartCommand` — runs on every start (daemonize agents here)
  6. `postAttachCommand` — runs when chat bridge attaches (bridge setup)
- **D-26:** **`waitFor` readiness gating:** every recipe declares which hook blocks session readiness. Default = `postCreateCommand`. `runner.RunWithLifecycle` returns the session as ready only after the declared `waitFor` hook completes successfully. If the hook exits non-zero, the whole session errors out and the container is destroyed.
- **D-27:** **Parallel array semantics:** a hook can be a string (single command) OR an array of strings (sequential commands) OR an array-of-arrays (parallel command groups). Example: `postCreateCommand: [["apt-get", "install", "git"], ["uv", "pip", "install", "aider"]]` runs the two groups in parallel. Go implementation uses `errgroup.WithContext`.
- **D-28:** **Hooks are idempotent by contract.** The recipe author declares idempotency — the loader doesn't verify. On session restart, `onCreateCommand` is skipped, `postCreateCommand` is re-run. Recipes that can't be idempotent in `postCreateCommand` should move the non-idempotent work to `onCreateCommand`.
- **D-29:** **Hook execution environment:** hooks run inside the container as the agent user (non-root, post-gosu drop), with the Phase 2 sandbox posture (cap_drop ALL, no_new_privs, read_only_rootfs except tmpfs, default seccomp). Secrets from the vault are NOT available during `onCreateCommand` / `updateContentCommand` (the vault is only populated after `postCreateCommand`) — recipes that need secrets at install time are rejected by the schema validator. This prevents API keys from leaking into install logs.
- **D-30:** **Timeout per hook:** each hook has a max runtime (default 10 min). Exceeded timeout fails the session. Configurable per-recipe via `lifecycle.<hook>_timeout_sec`. Rationale: large Python installs (`uv pip install -e .[all]`) can legitimately take 5+ min; bounded upper limit prevents zombie builds.

### Session lifecycle (non-Temporal, per D-05)

- **D-31:** **HTTP handlers call `runner.RunWithLifecycle` directly.** Same shape as Phase 2's direct-runner pattern, just against YAML-loaded `Recipe` instances instead of hardcoded `recipes.Picoclaw` / `recipes.Hermes` structs. `POST /api/sessions` → load recipe by id → materialize → `runner.RunWithLifecycle` → return session id.
- **D-32:** **`POST /api/sessions/:id/message`** routing: session row carries `chat_io.mode`. Handler dispatches to one of 4 bridge implementations based on the mode:
  - `fifo` → Phase 2's existing `docker exec` FIFO bridge, unchanged
  - `exec_per_message` → Phase 2's existing exec-per-message bridge, unchanged
  - `one_shot_task` → NEW: agent writes to `/tmp/result.json` at workspace root; handler polls every 500ms for up to `response_timeout_sec`; returns parsed JSON body
  - `http_gateway` → NEW: Go `http.Client` targets the agent's internal port via `dockerapi.ContainerInspect` → bridge network IP; bridge is stateless per-message
- **D-33:** **Response timeout defaults** per chat_io.mode: fifo = 60s, exec_per_message = 90s, one_shot_task = 900s (15 min), http_gateway = 120s. Recipe can override via `chat_io.<mode>.response_timeout_sec`.
- **D-34:** **`DELETE /api/sessions/:id`** = `runner.Stop` + `runner.Remove` + DB state update to `status='stopped'`. Same as Phase 2, no change.
- **D-35:** **Recipe reload:** the loader reads recipes at process-start (Go embed is NOT used — recipes stay on disk, hot-reloadable). In-memory cache. `SIGHUP` triggers a reload (cheap, idempotent). No HTTP endpoint to trigger reload in v0.1 — ship the signal handler only.

### Secret vault (Praktor-shape, Phase 3-compatible)

- **D-36:** **`auth.env` values** are either literal (`"vim"`) or vault refs (`"secret:anthropic-api-key"`). The session-start code recognizes the `secret:` prefix, calls `session.SecretSource.Resolve("anthropic-api-key")`, and injects the resolved value as the container env var.
- **D-37:** **`auth.files` block** (Praktor-shape) is an array of `{secret, target, mode, template}` entries. At container start, the entrypoint shim reads the resolved secret value from tmpfs `/run/secrets/<secret-name>`, renders `agents/<id>/templates/<template>.tmpl` with the secret as `.secrets.<name>` context, writes the rendered body to `<target>` with `chmod <mode>`, then execs the agent. This is the Phase 2 entrypoint-shim pattern extended to use files from the filesystem template registry (D-14) instead of baked-in Dockerfile steps.
- **D-38:** **`session.SecretSource` interface** in `api/internal/session/secrets.go` has one method: `Resolve(key string) (string, error)`. ### Gate A — matrix smoke test (plumbing prerequisite)

- **D-46:** **`make smoke-test-matrix` implementation (2 recipes × 2 providers):**
  - Inputs: `RECIPES=(aider picoclaw)`, `PROVIDERS=(anthropic openrouter)`.
  - Outer loop: for each recipe. Inner loop: for each provider in `recipe.providers[]`. Model is the **first entry in `recipe.models[]` that declares the provider** — convention: recipes author cheapest-first. No new schema field, no `cost_tier` tag, no separate model list. The recipe itself is the source of truth for what to run.
  - Per cell: `curl POST /api/sessions` with `{recipe, provider, model=first-match}` → `curl POST /api/sessions/:id/message -d '{"text":"whoareyou"}'` → assert HTTP 200 + non-empty body + body is not a raw error envelope → `curl DELETE /api/sessions/:id` → assert no dangling `playground-*` container remains on the host.
  - Output: ASCII matrix table to stdout, ✓/✗/SKIP per cell. Exit non-zero if any non-SKIP cell fails.
  - Total cells: **up to 4** (aider × {anthropic, openrouter} + picoclaw × {anthropic, openrouter}). SKIP when the recipe doesn't declare the provider. Per-cell runtime = container cold-start (5-15s) + one model reply (1-5s) = ~10-20s. Full matrix runtime ≤ 90s. Safe to run in CI on every PR.
- **D-47:** **Minimum matrix to pass Gate A:**
  - Both `aider` and `picoclaw` must pass on at least one provider each (not SKIP, not FAIL)
  - At least one of them must pass on both providers (proves the OpenRouter path works against a real recipe)
  - Zero non-SKIP failures
  - If picoclaw's OpenRouter support cannot be verified at plan-write time (picoclaw may hardcode the Anthropic base URL and ignore `OPENROUTER_BASE_URL`), that cell SKIPs and aider takes the both-providers slot. Planning flags this during Plan 02.5-08 L1 verification.
- **D-48:** **Model selection = first entry convention.** The recipe author lists models in cost-ascending order. The matrix picks `models[0]` matching the provider. No new schema field. No separate `cost_tier` enum. No `test/matrix-models.txt` file. This is deliberately simple and data-driven from the recipe itself. **Recommended cheapest models at time of plan-writing (planning locks exact IDs at plan-write time based on current pricing):**
  - **Anthropic direct:** `claude-haiku-4-5-20251001` — cheapest Claude, ~$0.25/$1.25 per 1M tokens, reliable on `whoareyou`
  - **OpenRouter:** `openrouter/auto` (router auto-picks cheapest valid model for the query — zero maintenance, adapts to catalog changes) OR an explicit cheap model like `meta-llama/llama-3.2-3b-instruct` if deterministic behavior is preferred in CI. Recipe author picks; 02.5 planning confirms.
  - The principle: **the cheapest model that returns a non-empty reply to `whoareyou`.** Nothing else matters for Gate A.
- **D-49:** **Probe message:** literal string `"whoareyou"`. No variation, no templating, no per-recipe override. User verbatim: *"send whoareyou and it must answer"*. Assert = HTTP 200 + non-empty response body + body is not a raw error JSON (top-level `"error"` key absent or null).
- **D-50:** **BYOK keys for the matrix test** come from env vars: `AP_DEV_BYOK_KEY` (Anthropic), `AP_DEV_OPENROUTER_KEY` (OpenRouter, NEW — Phase 02.5 adds this env var to the server config and the `SecretSource` resolver). Test skips ALL cells cleanly if neither is set; test skips only the affected provider's cells if one is set and the other is not. Makes it safe to run in CI without secrets AND locally with partial key coverage.

### Gate B — architectural drop-in test (the acceptance gate)

- **D-50a:** **Gate B is the phase's acceptance gate. Gate A is a prerequisite, not the gate.** Passing Gate A with 0 failures unlocks Gate B; passing Gate B closes Phase 02.5.
- **D-50b:** **Gate B protocol:**
  1. Pick a target agent the repo has never had a recipe for. Suggested: **openclaw** — we know its shape from Phase 2 (`gateway-websocket` chat_io documented in `SPIKE-REPORT.md §Spike 2`, MSV has the dockerized pattern at `infra/picoclaw/Dockerfile`). If openclaw's gateway-websocket chat_io mode is not yet in the v0.1 schema (it isn't — cut per D-10), pick a different agent whose chat_io mode IS in the v0.1 enum. Alternative targets: `ironclaw` (rust/exec_per_message — new runtime family, so reject — we cut rust base image), `nullclaw` (zig — reject, cut), OR **write a deliberately minimal test agent** like a hand-crafted Python script that speaks the Anthropic API directly and has an `exec_per_message` CLI (a "null-recipe" that exists solely to prove the drop-in mechanics).
  2. Time-box the drop-in exercise to **one calendar day** (target: ≤2 hours of active work). The operator (human or Claude) has only: the locked schema + JSON Schema + the 2 reference recipes as few-shot examples + the upstream target agent's public documentation. The operator writes `agents/<target>/recipe.yaml` + `agents/<target>/templates/*.tmpl` (if needed) + `agents/<target>/Dockerfile` (if needed). No Go code changes allowed. No recompile of the API server. No `go build` during the exercise.
  3. Drop the new directory into the running repo, restart the API server (which re-reads recipes on startup — no dynamic reload), run `make smoke-test-matrix` with the new recipe added to `RECIPES`, assert the new cells pass.
  4. If the drop-in succeeds: **architecture locked, Gate B passed, phase complete.**
  5. If the drop-in fails because the new recipe needs a schema field the v0.1 schema doesn't have, or a chat_io mode we cut, or a lifecycle hook that doesn't work as spec'd, or a template feature we don't support: **the pattern failed.** Iterate on the substrate (schema, loader, bridge — whichever failed) until the drop-in works on re-run. The iteration itself is the planned work of Phase 02.5; the gate is passing, not "passing on first try".
  6. If the drop-in fails because the target agent's upstream is simply broken (repo moved, install script errors, unreachable provider): pick a different target. This is not an architecture failure, it's a target failure.
- **D-50c:** **What "no Go code changes" means precisely:**
  - ✅ Allowed: editing `agents/<new-id>/recipe.yaml`, `agents/<new-id>/templates/*.tmpl`, `agents/<new-id>/Dockerfile`, and nothing else
  - ❌ Forbidden: editing anything under `api/`, `deploy/`, `Makefile` (runtime base targets), `agents/schemas/recipe.schema.json`
  - ✅ Allowed: adding a new runtime base image Dockerfile at `deploy/ap-runtime-<family>/Dockerfile` IF the target agent's runtime family is one we declared support for but haven't built yet — this is an escape hatch for honest scope (e.g., a go agent). But if the target requires a runtime family we cut (rust/zig), we don't count that as a drop-in success.
  - The spirit: "a new recipe's author needs exactly three directories and their contents. They should not have to understand the Go code, run `go build`, or touch the substrate."
- **D-50d:** **Gate B fallback:** if no suitable target agent can be picked (all candidates fail for target-specific reasons), fall back to a **synthetic drop-in test** — the operator hand-crafts a minimal "echo recipe" that wraps a Python one-liner speaking the Anthropic API directly, packaged as a recipe. This is a less convincing proof of the architecture (it tests less real-world variation) but it is a valid floor: if the architecture can't even absorb a synthetic recipe without code changes, something is very wrong and we iterate until it can.
