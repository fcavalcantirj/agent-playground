<objective>
Ship **Gate B** — the architectural drop-in test, Phase 02.5's actual hypothesis proof per D-01b. This is the test that says: "pick a third agent the codebase has never seen as a recipe, write its `agents/<target>/recipe.yaml` + optional templates/*.tmpl + optional Dockerfile FROM a runtime base, drop it into the running system as a pure directory add, restart the API, run the Gate A matrix with the new recipe added, assert the new cells pass — with zero Go code changes, zero recompile, zero substrate edits."

**If the drop-in works on first try (or with clearly-scoped fixes to the new recipe's own files, not the substrate), the architecture is good and Phase 02.5 closes.** If it requires substrate changes, the executor iterates on the schema/loader/bridge until it doesn't.

**Target selection** per D-50b:
1. First choice: **openclaw** — its shape is documented in Phase 2's spike report (`gateway-websocket` mode, but we cut that from v0.1). Because the v0.1 chat_io enum is `{fifo, exec_per_message}` only, openclaw's native mode does NOT fit the schema. Openclaw is therefore a BAD Gate B target for 02.5 — it would force either (a) schema iteration to add `gateway-websocket` back, or (b) expressing openclaw as `exec_per_message` with a custom CLI wrapper.
2. Better choice: **A real agent that fits the v0.1 enums.** Candidates: plandex (go, exec_per_message), but plandex is in the deferred-to-Phase-4 list. hiclaw / nullclaw / ironclaw — each requires a new runtime base (go/zig/rust), which is allowed by D-50c as a "deploy/ap-runtime-<family>/Dockerfile addition escape hatch".
3. **Recommended fallback: D-50d synthetic drop-in.** Hand-craft a minimal "null-recipe" — a tiny Python script that speaks the Anthropic API directly via the official SDK, packaged as an `agents/null-echo/recipe.yaml` + a one-file Python wrapper. The "agent" does nothing but echo "whoareyou" → "I am null-echo" via a single Anthropic call. This is a less convincing proof than a real agent but IS a valid floor: if the architecture can't absorb a synthetic recipe without code changes, something is very wrong.

**Decision for Plan 11:** the operator (human running this plan) picks the target at execution time per D-50b. The Plan 11 tasks explicitly allow either a real agent OR the D-50d synthetic fallback. The plan's success condition is "Gate B passes on SOME target that fits the v0.1 enums without requiring substrate changes".

Output: `test/drop-in.sh` + Makefile target + one human-driven checkpoint where the operator actually does the drop-in exercise. Plan 11 is heavy on process and light on code.

**File 1 — test/drop-in.sh:**

```bash
#!/usr/bin/env bash
# Gate B: architectural drop-in test (D-50a..d).
#
# Protocol:
#   1. Operator picks a target agent (default: synthetic null-echo, or $1)
#   2. Operator writes agents/<target>/recipe.yaml + optional templates + optional Dockerfile
#      by hand, using only the locked v0.1 schema + the 2 reference recipes as examples
#   3. Run this script:
#      a. Snapshot git state — no edits under api/, deploy/ (except ap-runtime-<new>/),
#         Makefile, agents/schemas/ allowed during the exercise
#      b. Restart the API server so the recipe loader picks up the new recipe
#      c. Invoke smoke-matrix.sh with RECIPES extended to include $TARGET
#      d. Assert the new cells pass
#
# Pass = Gate B passes = Phase 02.5 architectural hypothesis proven

set -uo pipefail

TARGET="${1:-null-echo}"
AGENTS_DIR="agents/$TARGET"
API_URL="${API_URL:-http://127.0.0.1:8080}"

C_GREEN=$'\e[32m'; C_RED=$'\e[31m'; C_YELLOW=$'\e[33m'; C_RESET=$'\e[0m'

echo "=== Gate B: architectural drop-in test ==="
echo "Target: $TARGET"
echo "Agents dir: $AGENTS_DIR"
echo

# --- Step 1: Sanity — the target directory must exist (operator wrote it) ---
if [ ! -d "$AGENTS_DIR" ]; then
    echo "${C_RED}FAIL${C_RESET}: $AGENTS_DIR does not exist."
    echo
    echo "Gate B protocol:"
    echo "  1. Create $AGENTS_DIR/ with the following files (as needed):"
    echo "       recipe.yaml              — required"
    echo "       templates/*.tmpl         — optional"
    echo "       Dockerfile               — optional (FROM ap-runtime-<family>:v0.1.0-*)"
    echo "  2. Use only the locked v0.1 schema: agents/schemas/recipe.schema.json"
    echo "  3. Use the 2 reference recipes as examples: agents/aider/, agents/picoclaw/"
    echo "  4. Re-run: bash test/drop-in.sh $TARGET"
    echo
    echo "Fallback: use the synthetic null-echo recipe per D-50d (hand-craft a minimal"
    echo "Python-via-Anthropic-SDK recipe that answers 'whoareyou')."
    exit 1
fi

if [ ! -f "$AGENTS_DIR/recipe.yaml" ]; then
    echo "${C_RED}FAIL${C_RESET}: $AGENTS_DIR/recipe.yaml missing"
    exit 1
fi

# --- Step 2: D-50c enforcement — NO substrate edits during the exercise ---
# Allowed edits: agents/<target>/** and agents/<target>/templates/**
# Forbidden:     api/**, agents/schemas/**, Makefile, most of deploy/**
# Escape hatch:  a new deploy/ap-runtime-<family>/Dockerfile for a family that
#                wasn't built yet (honest scope expansion per D-50c)

echo "[1/5] Verifying D-50c: no substrate edits..."

# Untracked files check — anything new under api/ or agents/schemas/ or Makefile is a violation
if git status --porcelain api/ agents/schemas/ Makefile 2>/dev/null | grep -qE '^[ADMR?]{1,2} '; then
    echo "${C_RED}D-50c VIOLATION${C_RESET}: uncommitted changes under api/, agents/schemas/, or Makefile"
    git status --short api/ agents/schemas/ Makefile
    exit 1
fi

# deploy/ — allow new ap-runtime-* subdirs but nothing else
if [ -d deploy ]; then
    deploy_violations=$(git status --porcelain deploy/ 2>/dev/null | grep -vE '^\?\? deploy/ap-runtime-[a-z]+/' | grep -E '^[ADMR?]{1,2} ')
    if [ -n "$deploy_violations" ]; then
        echo "${C_RED}D-50c VIOLATION${C_RESET}: edits under deploy/ outside the ap-runtime-<family>/ escape hatch:"
        echo "$deploy_violations"
        exit 1
    fi
fi

echo "  ${C_GREEN}OK${C_RESET}: no substrate edits detected"

# --- Step 3: Optionally build a per-recipe Dockerfile ---
if [ -f "$AGENTS_DIR/Dockerfile" ]; then
    echo "[2/5] Building $AGENTS_DIR/Dockerfile..."
    IMAGE_TAG="ap-$TARGET:v0.2.0-dropin"
    if ! docker build -t "$IMAGE_TAG" "$AGENTS_DIR/" 2>&1 | tail -20; then
        echo "${C_RED}FAIL${C_RESET}: docker build failed"
        exit 1
    fi
    echo "  ${C_GREEN}OK${C_RESET}: built $IMAGE_TAG"
else
    echo "[2/5] No custom Dockerfile — recipe runs directly on its declared runtime base"
fi

# --- Step 4: Restart the API server (SIGHUP reload) or instruct operator ---
echo "[3/5] Signaling API server to reload recipes..."
if pgrep -f 'go run ./cmd/server' >/dev/null 2>&1 || pgrep -f 'ap-server' >/dev/null 2>&1; then
    pkill -HUP -f 'go run ./cmd/server' 2>/dev/null || true
    pkill -HUP -f 'ap-server' 2>/dev/null || true
    sleep 2
    echo "  ${C_GREEN}OK${C_RESET}: SIGHUP sent; recipes reloaded via Plan 01 StartSIGHUPWatcher"
else
    echo "${C_YELLOW}WARN${C_RESET}: no running API server detected."
    echo "       Start the API server (Plan 10 prereq) and re-run Gate B."
    exit 1
fi

# --- Step 5: Verify the new recipe is visible via GET /api/recipes ---
echo "[4/5] Verifying recipe loaded..."
COOKIE_JAR=$(mktemp)
trap 'rm -f "$COOKIE_JAR"' EXIT
curl -fsS -c "$COOKIE_JAR" -X POST "$API_URL/api/dev/login" >/dev/null 2>&1

if ! curl -fsS -b "$COOKIE_JAR" "$API_URL/api/recipes/$TARGET" >/dev/null 2>&1; then
    echo "${C_RED}FAIL${C_RESET}: GET /api/recipes/$TARGET returned non-200"
    echo "       Likely: recipe.yaml failed schema validation OR loader semantic check."
    echo "       Inspect: tail -50 the API server log and look for 'recipes:' errors."
    exit 1
fi
echo "  ${C_GREEN}OK${C_RESET}: $TARGET loaded"

# --- Step 6: Run the extended matrix ---
echo "[5/5] Running smoke-test-matrix with extended RECIPES..."
export DROP_IN_TARGET="$TARGET"
# Invoke smoke-matrix.sh with RECIPES extended. The cleanest approach is to
# call smoke-matrix.sh directly with a patched array — but bash-script coupling
# is fragile. Instead, invoke it in a subshell with an override env var that
# smoke-matrix.sh reads if set.
#
# SIMPLER: copy smoke-matrix.sh's logic into a one-off inline loop here.
# See the matrix block below — we run the same 2 assertions (aider/picoclaw must
# still pass unchanged) + a 3rd cell set for the new target.

if ! RECIPES_OVERRIDE="aider picoclaw $TARGET" bash test/smoke-matrix.sh; then
    echo
    echo "${C_RED}=== Gate B FAILED ===${C_RESET}"
    echo "The drop-in failed. Diagnose:"
    echo "  (a) Did the NEW target's cells fail? → iterate the recipe (not the substrate)"
    echo "  (b) Did aider/picoclaw cells regress? → the new recipe somehow broke something; revert and investigate"
    echo "  (c) Schema rejected recipe? → architecture failed; iterate Plan 01 schema + re-run"
    echo
    echo "Per D-50b iteration protocol, up to 3 iterations are allowed before"
    echo "returning PHASE ITERATION EXHAUSTED."
    exit 1
fi

echo
echo "${C_GREEN}=== Gate B PASSED ===${C_RESET}"
echo "Architectural drop-in succeeded: $TARGET added with zero substrate edits."
echo "Phase 02.5 hypothesis proven."
exit 0
```

**Important change to test/smoke-matrix.sh:** the script above invokes `bash test/smoke-matrix.sh` with a `RECIPES_OVERRIDE` env var. The matrix script in Plan 10 reads `RECIPES=(aider picoclaw)` as a hardcoded array. We need to make it accept an override:
