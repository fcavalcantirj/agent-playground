<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP message → ChatBridge.SendMessage | User-supplied text reaches the container. Phase 2's T-02-04 / T-02-04b threat notes documented the shell-safety invariants; this plan MUST preserve them. |
| chat_io.mode string → BridgeRegistry.Dispatch | Mode value comes from the recipe YAML which is schema-validated in Plan 01 (closed enum). Defense-in-depth: registry also rejects unknown modes. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02.5-05 | Tampering / Elevation of Privilege | FIFO bridge shell injection via user text | mitigate | Port Phase 2's `ExecWithStdin` pattern verbatim: user text flows over stdin into `cat >> /run/ap/chat.in`, never as an argv element, never into sh -c. The Docker exec argv is constant `["sh","-c","cat >> $FIFO_IN"]` and the user text is piped through stdin. Phase 2's T-02-04 threat note is the canonical reference; this plan does not modify the logic. |
| T-02.5-05a | Tampering / Elevation of Privilege | Exec bridge shell injection via user text | mitigate | Port Phase 2's `Runner.Exec` pattern verbatim: user text becomes one argv element inside a literal exec-form cmd_template (e.g. `["aider", "--message", "<user text>"]`). No sh -c wrapping. The template's `{text}` placeholder (if any) is replaced by Go string substitution BEFORE the argv slice is built, but because the value lands in its own slice element, shell metacharacters are inert. Phase 2's T-02-04b note. |
| T-02.5-05b | Elevation of Privilege | BridgeRegistry dispatches an unknown mode to a nil-checked zero value | mitigate | Registry uses map lookup + explicit `ok` check. Unknown mode returns `ErrUnsupportedMode` (error envelope in Plan 09 maps to `chat_bridge_unsupported_mode`). Plan 01's schema closed-enum makes this unreachable through normal flow; this is defense-in-depth for future flavors or community recipes. |
| T-02.5-05c | Information Disclosure | Cross-tenant bridge reuse (session A's FIFO reaches session B) | mitigate | BridgeRegistry is stateless per-call; every SendMessage invocation passes its own `containerID` from the authenticated session. FIFO paths are inside each container's private `/run/ap/` tmpfs — there is no shared host-side FIFO surface. The session row's `container_id` is validated by Plan 09's handler via ownership check before Dispatch is called. |

</threat_model>
