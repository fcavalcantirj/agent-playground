# Phase 02.5 Plan 01: Recipe Schema + Loader + SIGHUP Reload Summary

**One-liner:** ap.recipe/v1 Draft 2019-09 JSON Schema + embedded validator + strict YAML loader with semantic checks, flavor resolution, RWMutex-guarded cache, and SIGHUP reload — all running green under `go test -race`.

## Objective Recap

Lock the foundation for the Phase 02.5 recipe manifest rework: the
canonical schema file, the typed `Recipe` Go struct (JSON tags only),
the embedded `SchemaValidator`, the strict loader (validate → unmarshal
→ semanticCheck → flavor-resolve → atomic swap), and the SIGHUP reload
watcher. No downstream consumer is wired yet — Plan 02.5-02..11 build
on top of this package.

## Actual vs Planned File Layout

The plan called for files in `api/internal/recipes/` plus fixtures
under `testdata/`. Final layout matches the plan exactly with one
addition: a `flavor_chain/base/recipe.yaml` fixture (not in the plan's
<files> list but required so the flavor-chain test has a real parent
to hang the child off). This is a minor fidelity adjustment to the
fixture tree, not a deviation from the loader behavior spec.

The `flavor_redeclare/` fixture pair is also new vs the plan text —
the plan listed `testdata/invalid/flavor_redeclares_install.yaml` but
the Loader test actually needs a full fixture directory (`base/` +
`rogue/`) to trigger the second-pass check, because `rogue` must
reference `base` by name. The invalid standalone `.yaml` still exists
for potential direct schema-level use by later plans.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocking] Schema top-level required list kept `install` but flavor fixtures need to skip it**

- **Found during:** Task 2 fixture authoring.
- **Issue:** The original schema listed `install` in the top-level
  `required` array. But `config_flavor_of` recipes are supposed to
  inherit `install` from their parent and must NOT declare their own
  (Task 2's `ErrFlavorRedeclaresInstall` check) — so a valid flavor
  recipe has no `install` block and was being rejected at schema-level
  before the loader ever got a chance to run its flavor-aware second
  pass.
- **Fix:** Removed `install` from the top-level schema `required`
  array. Added `install.type` requirement for non-flavor recipes in
  `Loader.semanticCheck` (same error surface; now correct for both
  flavor and non-flavor shapes).
- **Files modified:** `agents/schemas/recipe.schema.json`,
  `api/internal/recipes/schema/recipe.schema.json` (via `make
  copy-schema`), `api/internal/recipes/loader.go` (added
  `install.type required for non-flavor` branch in semanticCheck).
- **Commit:** `391c5a2` (rolled into Task 2 commit)

**2. [Rule 3 - Blocking] Phase 2 `Recipe`/`AuthFile` collide with new YAML-backed types**

- **Found during:** Task 1, when adding the new exported `Recipe` type
  alongside the Phase 2 hardcoded catalog.
- **Issue:** The existing `api/internal/recipes/recipes.go` already
  exports `type Recipe struct` (a totally different shape) plus
  `type AuthFile struct`, `var AllRecipes`, and `func Get`. The new
  YAML-backed `Recipe` needs the name `Recipe` to be the canonical
  public type. Deleting Phase 2 code would break the Phase 2 session
  handler + bridge + their tests.
- **Fix:** Renamed Phase 2 exports to `LegacyRecipe`, `LegacyAuthFile`,
  `LegacyAllRecipes`, `GetLegacy`. Updated `session/handler.go`,
  `session/bridge.go`, `session/bridge_test.go`, and
  `recipes/recipes_test.go` to use the `Legacy*` names. This is
  exactly the "temporary alias then mechanical rename" strategy the
  plan <action> block described. Plan 02.5-09 will delete the entire
  legacy path once the handler is swapped to the new Loader.
- **Files modified:** `api/internal/recipes/recipes.go`,
  `api/internal/recipes/recipes_test.go`,
  `api/internal/session/handler.go`, `api/internal/session/bridge.go`,
  `api/internal/session/bridge_test.go`.
- **Commit:** `380257e` (rolled into Task 1 commit)

## Risks & Follow-ups for Downstream Plans

1. **`Recipe.Lifecycle.*Command` is `any`** — Plan 02.5-03 must add
   `NormalizeHook(v any) [][]string` to reduce the string | []string |
   [][]string union into a single dispatch form. Until then, downstream
   callers can't run lifecycle hooks. Not this plan's scope.
2. **No in-process consumers yet** — the loader exists but nothing
   imports it except the tests. Plan 02.5-09 wires the session handler.
   Plan 02.5-02 consumes `Recipe.Auth.Files` + the template registry.
3. **`agents/` recipe dirs are still empty of YAML** — Plan 02.5-09 (or
   whichever plan migrates picoclaw + hermes) writes the first real
   `agents/<id>/recipe.yaml` files on disk. The test fixtures under
   `testdata/fixtures/` are NOT the production recipes.
4. **SIGHUP watcher not wired into `main.go`** — intentional: the
   watcher starts only when `StartSIGHUPWatcher(ctx, loader, logger)`
   is called, and nobody calls it until a Loader instance exists in the
   server struct. Plan 02.5-09 is the natural home for that wiring.
5. **Schema JSON file is the source of truth.** If any downstream plan
   wants to edit `agents/schemas/recipe.schema.json`, they MUST also
   run `make copy-schema` before testing, or the embedded copy will
   drift from the canonical one and tests will fail.
