### Gaps Summary

Phase 02.5 overwhelmingly achieved its goal. The architectural hypothesis is proven:
- Gate B passed on first try with zero substrate edits (`agents/null-echo/` directory drop + SIGHUP reload + matrix extension).
- Gate A passed with 3/3 non-SKIP cells green, OpenRouter wiring proven end-to-end via aider.
- All 10 substrate components (schema, loader, template registry, lifecycle runner, chat bridge interface, secrets pipeline, runtime base images, reference recipes, HTTP handlers, gate infrastructure) exist, are WIRED, and pass their unit + integration tests.

**One real deviation** from the locked decisions: D-40b + SC7 lock-in required picoclaw to be `chat_io.mode=fifo`. Execute-time debugging (commit 29214c2) flipped it to `exec_per_message` because picoclaw supports a one-shot `--message` mode that aligns with the simpler exec bridge, and the long-running tmux + FIFO chain was failing under strict read-only rootfs. The FIFOBridge code IS lifted from Phase 2, wired to BridgeRegistry, and unit-tested — but no shipping recipe drives a real message through it at matrix time, which means the phase goal wording "exercise every component of the new substrate end-to-end" is not literally met for the FIFO path. The goal's central claim — "recipe N+1 is configuration, not code" — is not affected by this deviation; Gate B proved it directly with a pure directory drop.

Three resolution options (pick one):

1. **Accept the deviation via VERIFICATION.md override.** Add an `overrides:` entry to this file's frontmatter with `accepted_by`, a rationale citing the strict read-only rootfs constraint + the fact that FIFOBridge is unit-tested + the architectural hypothesis is proven by Gate B, and re-run verification. This is the cheapest close given that Phase 4 is explicitly scheduled to re-ship picoclaw alongside openclaw/hermes/hiclaw and can pick up a fifo-native recipe then.

2. **Restore picoclaw to fifo.** Move picoclaw back to `chat_io.mode=fifo`, re-introduce the tmux chat window + postAttachCommand FIFO verify, rebuild `ap-picoclaw:v0.2.0-773a94c` with whatever tmpfs/exec adjustments are needed under read-only rootfs, and re-run Gate A to confirm picoclaw×anthropic still passes through the FIFOBridge path. Most faithful to the locked plan but highest iteration cost.

3. **Ship a third reference recipe that exercises fifo.** Add a minimal fifo-only recipe (e.g. a hand-crafted shell-script "agent" that reads stdin from `/run/ap/chat.in` and writes to `/run/ap/chat.out`) and extend Gate A to include it. Preserves picoclaw's current working state and exercises the FIFO path end-to-end, at the cost of one more recipe to maintain.

Recommendation: **Option 1 (override)** is appropriate here because the architectural hypothesis — the only thing Phase 02.5 is actually gated on per D-50a — was proven cleanly by Gate B with zero code changes. Fifo coverage can be a Phase 4 line item.
