## Deviations from Plan

### Auto-fixed issues

**1. [Rule 3 — Blocking issue] Template filename cannot contain dots**

- **Found during:** Task 2 loader validation
- **Issue:** The plan's acceptance criteria and files_modified list `templates/security.yml.tmpl` and `template: security.yml`. Plan 01's shipped JSON Schema enforces `auth.files[].template` must match `^[a-z0-9][a-z0-9_-]*$` — no dots allowed. The loader rejected the recipe with: `'security.yml' does not match pattern '^[a-z0-9][a-z0-9_-]*\$'`. Plan 02's TemplateRegistry internally allows dots in template filenames (regex `^[a-z0-9][a-z0-9._-]*\.tmpl$`), but the schema-level field pattern in the YAML is stricter.
- **Fix:** Renamed the on-disk file from `security.yml.tmpl` → `security-yml.tmpl`, and the YAML field from `template: security.yml` → `template: security-yml`. The `.tmpl` suffix is still added by TemplateRegistry.Render.
- **Files modified:** `agents/picoclaw/recipe.yaml`, `agents/picoclaw/templates/security-yml.tmpl`
- **Commit:** `bcb952c`
- **Impact on acceptance criteria:** The plan's `grep -cE '^      template: security.yml$' ... returns 1` and `test -f agents/picoclaw/templates/security.yml.tmpl` criteria are both now FALSE as written; the updated forms are `grep -cE '^      template: security-yml$' ... returns 1` and `test -f agents/picoclaw/templates/security-yml.tmpl` → both PASS. The plan author should update the plan text to reflect the shipping schema.

**2. [Rule 1 — Bug] Template comments leaked the BYOK key on render**

- **Found during:** Task 2 TemplateRegistry render probe
- **Issue:** The first draft of `security-yml.tmpl` contained multi-paragraph design notes as `# ...` comments explaining the porting decisions. Several of those comments included the literal string `{{ quote .secrets.anthropic_key }}` (as documentation). text/template evaluates template expressions *everywhere in the body* — including comment lines — so the rendered `.security.yml` body had the BYOK key repeated inside several comment lines in addition to the actual `api_keys:` position.
- **Fix:** Moved ALL design documentation to `agents/picoclaw/README.md`. The template body is now a single purpose-explaining comment block that contains zero template expressions + the raw YAML body with one `{{ quote .secrets.anthropic_key }}` inside `api_keys:`. Render output was re-verified.
- **Files modified:** `agents/picoclaw/templates/security-yml.tmpl`, `agents/picoclaw/README.md`
- **Commit:** `bcb952c`
- **Blast radius:** would not have leaked to operators directly (the rendered file still lives on per-session tmpfs mode 0600 bind-mounted read-only into the container), but would have widened the file's surface for log accidents and made the rendered artifact harder to review.

**3. [Rule 1 — Bug] Dockerfile comment contained literal `FROM ap-base`**

- **Found during:** Task 2 acceptance criteria re-run
- **Issue:** The plan requires `grep -c 'FROM ap-base' agents/picoclaw/Dockerfile` returns 0 (D-22 enforcement). A draft comment said "Recipes must NOT FROM ap-base — they must FROM an ap-runtime-<family> base..." which triggered the grep.
- **Fix:** Reworded the comment: "Recipes must never inherit ap-base directly — they must target an ap-runtime-<family> base". Same meaning, no literal `FROM ap-base` string.
- **Commit:** `bcb952c`

**4. [Rule 1 — Bug] recipe.yaml comment contained literal `config_flavor_of`**

- **Found during:** Task 2 acceptance criteria first run
- **Issue:** Plan criterion `grep -c 'config_flavor_of' agents/picoclaw/recipe.yaml` must return 0. A draft comment said "STANDALONE in 02.5 — NOT config_flavor_of openclaw".
- **Fix:** Reworded to "STANDALONE in 02.5 — NOT a flavor of openclaw". Same meaning, no literal field name.
- **Commit:** `bcb952c`
