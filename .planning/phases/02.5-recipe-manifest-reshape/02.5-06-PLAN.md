**File 1 — deploy/ap-runtime-python/Dockerfile:**

```dockerfile
# ap-runtime-python — Phase 02.5 runtime base for Python recipes.
#
# Lock chain:
#   - FROM ap-base:v0.1.0 (Phase 2, Debian trixie + tini + tmux + ttyd + gosu)
#   - COPY --from ghcr.io/astral-sh/uv:0.11.6-python3.13-trixie
#   - D-19: Python 3.13 + uv; pip is forbidden in python recipes (loader translates pip → uv pip)
#   - D-21: thin layer — language toolchain only, no other tooling
#   - D-22: recipe overlays FROM this image, never from ap-base directly
#
# Size target: <1.5 GB (D-23, planning guardrail not hard gate)

FROM ap-base:v0.1.0

USER root

# Astral's uv image bundles uv + python3.13 + stdlib on Debian trixie. COPY --from
# pulls them deterministically — no apt pin dance, no cache pollution.
COPY --from=ghcr.io/astral-sh/uv:0.11.6-python3.13-trixie /usr/local/bin/uv /usr/local/bin/uv
COPY --from=ghcr.io/astral-sh/uv:0.11.6-python3.13-trixie /usr/local/bin/uvx /usr/local/bin/uvx
COPY --from=ghcr.io/astral-sh/uv:0.11.6-python3.13-trixie /usr/local/bin/python3.13 /usr/local/bin/python3.13
COPY --from=ghcr.io/astral-sh/uv:0.11.6-python3.13-trixie /usr/local/lib/python3.13 /usr/local/lib/python3.13

# Make python3 / python resolve to 3.13, ensure agent user owns its cache dir.
RUN ln -sf /usr/local/bin/python3.13 /usr/local/bin/python3 && \
    ln -sf /usr/local/bin/python3.13 /usr/local/bin/python && \
    mkdir -p /home/agent/.cache /home/agent/.local/bin && \
    chown -R agent:agent /home/agent/.cache /home/agent/.local

USER agent
WORKDIR /home/agent

ENV PYTHON_VERSION=3.13 \
    UV_LINK_MODE=copy \
    UV_SYSTEM_PYTHON=1 \
    PATH=/home/agent/.local/bin:/usr/local/bin:$PATH

# No ENTRYPOINT / CMD — ap-base provides tini + entrypoint shim. Inherit unchanged.
```

**File 2 — deploy/ap-runtime-node/Dockerfile:**

```dockerfile
# ap-runtime-node — Phase 02.5 runtime base for Node recipes.
#
# Lock chain:
#   - FROM ap-base:v0.1.0 (Phase 2, Debian trixie + tini + tmux + ttyd + gosu)
#   - COPY --from node:22-slim (Debian-based, NOT Alpine — D-20 + Phase 2 D-04)
#   - Node 22 LTS is the current LTS line (active support through 2027)
#   - D-21: thin layer — language toolchain only
#
# Size target: <800 MB (D-23, planning guardrail)

FROM ap-base:v0.1.0

USER root

# Copy the node binary + npm/npx + node_modules from the official node:22-slim
# image. Using COPY --from avoids pulling Debian's node packages (they lag upstream
# by months) and avoids a separate apt download step that would inflate the layer.
COPY --from=node:22-slim /usr/local/bin/node /usr/local/bin/node
COPY --from=node:22-slim /usr/local/include/node /usr/local/include/node
COPY --from=node:22-slim /usr/local/lib/node_modules /usr/local/lib/node_modules

RUN ln -sf /usr/local/lib/node_modules/npm/bin/npm-cli.js /usr/local/bin/npm && \
    ln -sf /usr/local/lib/node_modules/npm/bin/npx-cli.js /usr/local/bin/npx && \
    ln -sf /usr/local/lib/node_modules/corepack/dist/corepack.js /usr/local/bin/corepack && \
    mkdir -p /home/agent/.npm && \
    chown -R agent:agent /home/agent/.npm

USER agent
WORKDIR /home/agent

ENV NODE_VERSION=22 \
    NPM_CONFIG_CACHE=/home/agent/.npm \
    PATH=/home/agent/.local/bin:/usr/local/bin:$PATH

# No ENTRYPOINT / CMD — inherit from ap-base.
```

```makefile
# Phase 02.5 runtime base images — 2 of the 5 families (D-18).

AP_RUNTIME_PYTHON_TAG := ap-runtime-python:v0.1.0-3.13
AP_RUNTIME_NODE_TAG   := ap-runtime-node:v0.1.0-22

.PHONY: build-runtime-python build-runtime-node build-runtimes clean-runtimes

build-runtime-python: build-ap-base
	docker build -t $(AP_RUNTIME_PYTHON_TAG) deploy/ap-runtime-python/

build-runtime-node: build-ap-base
	docker build -t $(AP_RUNTIME_NODE_TAG) deploy/ap-runtime-node/

build-runtimes: build-runtime-python build-runtime-node
	@echo "Built runtimes: $(AP_RUNTIME_PYTHON_TAG) $(AP_RUNTIME_NODE_TAG)"
	@docker images --format 'table {{"{{.Repository}}"}}:{{"{{.Tag}}"}}\t{{"{{.Size}}"}}' | grep -E 'ap-base|ap-runtime-'

clean-runtimes:
	-docker rmi $(AP_RUNTIME_PYTHON_TAG) $(AP_RUNTIME_NODE_TAG)
```

<threat_model>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Upstream Docker images (`ghcr.io/astral-sh/uv`, `node:22-slim`) → local build | Build-time download. Content-pinned via explicit image tags. Supply chain trust = whoever published those tags. |
| `ap-base:v0.1.0` → runtime bases | Internal image produced by Phase 2. All Phase 2 sandbox posture carries over — cap_drop, no_new_privs, read_only_rootfs, seccomp. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02.5-04 | Elevation of Privilege | Runtime base image regresses sandbox posture (e.g. resets USER to root) | mitigate | Dockerfiles explicitly `USER agent` at the end after any root-required COPY. Acceptance criteria includes a `USER agent` grep gate in Task 1 (implied by file content). Task 2 Step 4 re-verifies tini/gosu/tmux/ttyd presence — if any is missing the inheritance broke. Phase 2 sandbox knobs (cap_drop ALL, no_new_privs, read_only_rootfs) are applied at RUN time by `pkg/docker/runner.go`, not by the Dockerfile — runtime bases cannot override them. |
| T-02.5-07 | Supply Chain | Malicious upstream image replaces tag under us | accept | `ghcr.io/astral-sh/uv:0.11.6-python3.13-trixie` and `node:22-slim` are pinned by tag, not by digest. Tag mutation is a real risk but mitigated by (a) v0.1 local-only builds (not a public registry), (b) the Phase 2 local Docker daemon already trusts these images, (c) Phase 7.5/8 will add image-digest pinning when supply-chain attestation ships. Documented as accepted for v0.1. |
| T-02.5-08 | Tampering | Runtime base image size exceeds host disk capacity | accept | Planning guardrails (1.5 GB / 800 MB) are informational. Task 2 records actual sizes. Hetzner dedicated box has ample disk (tracked by PER-06 at Phase 7). Out of scope for 02.5. |

</threat_model>
