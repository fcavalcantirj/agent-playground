**2. [Rule 1 — Bug] Swapped `uv pip install --system` → `uv tool install` in `lifecycle.postCreateCommand`**

- **Found during:** Task 1 (L2 verification — Step 4, after A3 fallback to 3.12)
- **Issue:** The plan's draft recipe uses `[uv, pip, install, --system, --no-cache, aider-chat]`. Running this as user `agent` (the user the Plan 03 lifecycle runner exec-s as) fails with `Permission denied (os error 13)` on `/usr/local/lib/python3.12/site-packages/*` because that directory is root-owned in the ap-runtime-python base image. This is NOT a 3.12-vs-3.13 issue — the same failure occurs on the 3.13 base image under `--user agent`. The plan's draft command was theoretically correct only if the lifecycle runner exec-d as root.
- **Root cause:** `api/pkg/docker/runner.go::Runner.Exec` hardcodes `User: "agent"` in its `ExecCreateOptions`. That's the Phase 2 security posture (`agent` is uid 10000, non-root) and is the right call — the fix belongs in the recipe, not the runner.
- **Fix:** Replaced the postCreateCommand argv with `[uv, tool, install, --no-cache, aider-chat]`. `uv tool install` installs aider into `/home/agent/.local/share/uv/tools/aider-chat/` (a subtree already owned by `agent`) and symlinks the entrypoint to `/home/agent/.local/bin/aider`. The image's `ENV PATH=/home/agent/.local/bin:/usr/local/bin:$PATH` means `launch.cmd: [aider, --help]` and the exec_per_message `cmd_template` resolve `aider` correctly.
- **Files modified:** `agents/aider/recipe.yaml` (postCreateCommand argv + inline comment)
- **Verification:** L2 install succeeds in 6s; `/home/agent/.local/bin/aider` is on PATH; `aider --version` → `aider 0.86.2`; `aider --help` runs clean.
- **Committed in:** `366c775`
- **`install.type` impact:** Kept `install.type: pip` — it reflects the package source (PyPI), and the loader's semantic check only verifies `install.type ↔ runtime.family` coherence (pip ↔ python), which is unchanged. `uv tool install` is uv's isolated-tool flavor of a pip install; there is no `install.type: uv_tool` in the v0.1 schema enum and adding one would be a Rule 4 architectural change.

## Issues Encountered

- **Entrypoint override needed for L2 verification.** `ap-runtime-python:v0.1.0-3.13` inherits ap-base's `ENTRYPOINT [/usr/bin/tini, -g, --, /entrypoint.sh]` which starts tmux/ttyd by default. The first L2 verification attempt (without `--entrypoint /bin/bash`) hung because the entrypoint did its own thing. Resolved by adding `--entrypoint /bin/bash` to the `docker run` test invocation. This is a test-harness concern, NOT a recipe concern — when the lifecycle runner exec-s postCreateCommand, it does so via `docker exec` into an already-running container, so the entrypoint is not a factor.
- **numpy 1.26.4 vs Python 3.13.** Initially the uv install command on 3.13 appeared to exit with `rc=0` in a permissive shell, but the underlying pypa resolver had already printed a fatal error; the shell was swallowing it. Re-running with `set -e` surfaced the real failure and confirmed the numpy sdist compile path. This led directly to the A3 falsification.
