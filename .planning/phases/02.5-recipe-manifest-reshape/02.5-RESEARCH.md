# Phase 02.5: Recipe Manifest Reshape — Research

**Researched:** 2026-04-14
**Domain:** Go YAML/JSON-Schema recipe loader + Dev-Containers lifecycle runner + chat-bridge interface + Praktor-shape secret vault + Dockerfile runtime bases, on top of the Phase 2 substrate (`api/pkg/docker/runner.go`, `api/internal/session/bridge.go`, `api/internal/recipes/recipes.go`).
**Confidence:** HIGH on library picks (Context7-verifiable), MEDIUM-HIGH on lifecycle/bridge implementation shapes (ported from Dev Containers spec + Phase 2's existing bridge).

## Summary

Phase 02.5's implementation unknowns collapse to **seven concrete choices**: (1) YAML parser, (2) JSON Schema validator, (3) template-engine sandbox strategy, (4) Dev Containers 6-hook sequencer, (5) `moby/moby/client` exec-streaming pattern for hooks, (6) Praktor `secret:`/file-mount resolver shape, and (7) runtime-base Dockerfile layouts for Python/Node. None of these are research-hard — they are "pick the one library that is currently maintained, verified for our draft level, and minimally invasive." The heavy architectural work (schema fields, decisions, acceptance gates) is already locked in `02.5-CONTEXT.md` and `RECIPE-SCHEMA-DRAFT.md`.

The single riskiest decision is the **JSON Schema library** — Go has ~5 candidates and only two support Draft 2019-09 cleanly. The single most load-bearing carry-over is the **existing `bridge.go` FIFO + exec patterns** from Phase 2, which already work against picoclaw and Hermes in production — 02.5 must preserve their exact behavior while hiding them behind a `ChatBridge` interface, not reinvent them.

**Primary recommendation:** `sigs.k8s.io/yaml` (YAML→JSON round-trip, strict mode) + `github.com/santhosh-tekuri/jsonschema/v6` (Draft 2019-09 native) + hand-written template FuncMap with `default/quote/lower/upper/trim` only + `errgroup.WithContext` for parallel-array lifecycle semantics + port Phase 2's `Bridge.fifoMode`/`execMode` verbatim behind a new `ChatBridge` interface. Runtime base images are thin `FROM debian:trixie-slim` layers (NOT Alpine, per Phase 2 D-04) that install tini+tmux+ttyd+gosu from `ap-base` and add only the language toolchain.

Every implementation of Dev Containers lifecycle hooks in the ecosystem (Dev Containers CLI itself, Daytona) uses the same goroutine-per-command + shared-context pattern. `[ASSUMED]` that `x/sync` is already an indirect dep via pgx or echo; if not, direct add. | BSD-3-Clause |

### Template engine

| Library | Version | Purpose | Why standard |
|---------|---------|---------|--------------|
| `text/template` (stdlib) | Go 1.25 | Render `agents/<id>/templates/*.tmpl` with secret values | **Use stdlib, do not add a 3rd-party templater.** `text/template` is the only sandbox-safe choice for secret injection: (a) by default it has no `exec`, no file I/O, no network, no reflection; (b) all helper functions are explicit opt-in via `template.FuncMap`; (c) it parses once and executes against injected data; (d) it has no plugin mechanism so D-16's path allowlist alone is sufficient. **Do NOT use `html/template`** — it auto-escapes for HTML contexts which will corrupt YAML/JSON/ini file output. `[CITED: oligo.security/blog/safe-by-default-or-vulnerable-by-design-golang-server-side-template-injection]` — the SSTI research confirms text/template is safe-by-default; vulnerabilities only arise when callers register dangerous FuncMap entries. D-16's allowlist (`default, quote, lower, upper, trim`) is exactly the set of "zero-side-effect string helpers" that the security research recommends. |

### Pattern 1: Embedded JSON Schema + parse-once validator

**What:** Compile the schema once at process start, reuse the compiled `*jsonschema.Schema` for every recipe load. Use `//go:embed` so the schema ships inside the binary for the `go run` path AND sits on disk at `agents/schemas/recipe.schema.json` for external validators.

**When:** All recipe loading.

**Example** (source: `[CITED: pkg.go.dev/github.com/santhosh-tekuri/jsonschema/v6]`):

```go
package recipes

import (
    _ "embed"
    "fmt"
    "strings"

    "github.com/santhosh-tekuri/jsonschema/v6"
    "sigs.k8s.io/yaml"
)

//go:embed ../../../agents/schemas/recipe.schema.json
var recipeSchemaJSON []byte

type SchemaValidator struct {
    schema *jsonschema.Schema
}

func NewSchemaValidator() (*SchemaValidator, error) {
    doc, err := jsonschema.UnmarshalJSON(strings.NewReader(string(recipeSchemaJSON)))
    if err != nil {
        return nil, fmt.Errorf("recipes: parse schema: %w", err)
    }
    c := jsonschema.NewCompiler()
    c.DefaultDraft(jsonschema.Draft2019)
    if err := c.AddResource("recipe.schema.json", doc); err != nil {
        return nil, fmt.Errorf("recipes: add schema resource: %w", err)
    }
    sch, err := c.Compile("recipe.schema.json")
    if err != nil {
        return nil, fmt.Errorf("recipes: compile schema: %w", err)
    }
    return &SchemaValidator{schema: sch}, nil
}

// ValidateYAML takes raw recipe.yaml bytes and validates against the schema.
// Returns a validation error with path-qualified field references on failure.
func (v *SchemaValidator) ValidateYAML(raw []byte) error {
    // Round-trip YAML → JSON → generic value so the validator sees exactly
    // what the JSON Schema is written against.
    jsonBytes, err := yaml.YAMLToJSONStrict(raw)
    if err != nil {
        return fmt.Errorf("recipes: yaml to json: %w", err)
    }
    inst, err := jsonschema.UnmarshalJSON(strings.NewReader(string(jsonBytes)))
    if err != nil {
        return fmt.Errorf("recipes: unmarshal json: %w", err)
    }
    if err := v.schema.Validate(inst); err != nil {
        return fmt.Errorf("recipes: schema validation failed: %w", err)
    }
    return nil
}
```

**Why this shape:**
- `YAMLToJSONStrict` rejects duplicate keys per D-08's "closed enums, no ambiguity" contract.
- `jsonschema.Draft2019` lock is explicit per D-09.
- `//go:embed` guarantees schema-in-binary so nothing at runtime can tamper with it.
- Compilation happens once; `Validate` is the only hot-path call.

### Pattern 2: Typed recipe struct with strict unmarshal

**What:** After schema validation passes, unmarshal the same JSON bytes into a typed Go struct with JSON tags (not YAML tags — because sigs.k8s.io/yaml uses the JSON path). Use strict unmarshal so typos in field names (especially in closed enums) fail at load time, not at runtime.

**Example:**

```go
// Recipe mirrors agents/schemas/recipe.schema.json for typed consumption.
// JSON tags (NOT yaml tags) because sigs.k8s.io/yaml round-trips through JSON.
type Recipe struct {
    ID          string             `json:"id"`
    Name        string             `json:"name"`
    Description string             `json:"description,omitempty"`
    Version     string             `json:"version,omitempty"`
    License     string             `json:"license,omitempty"`
    Category    string             `json:"category,omitempty"`

    Runtime        RecipeRuntime      `json:"runtime"`
    ConfigFlavorOf string             `json:"config_flavor_of,omitempty"`
    Install        RecipeInstall      `json:"install"`
    Lifecycle      RecipeLifecycle    `json:"lifecycle,omitempty"`
    Launch         RecipeLaunch       `json:"launch"`
    ChatIO         RecipeChatIO       `json:"chat_io"`
    Auth           RecipeAuth         `json:"auth,omitempty"`
    Providers      []RecipeProvider   `json:"providers,omitempty"`
    Models         []RecipeModel      `json:"models,omitempty"`
    Isolation      RecipeIsolation    `json:"isolation"`
    Frontend       RecipeFrontend     `json:"frontend,omitempty"`
    Metadata       RecipeMetadata     `json:"metadata,omitempty"`
    // ... (full shape from RECIPE-SCHEMA-DRAFT.md §Full schema spec)
}

// RecipeLifecycle uses any-typed fields because each hook can be
// string | []string | [][]string per D-27.
type RecipeLifecycle struct {
    InitializeCommand    any    `json:"initializeCommand,omitempty"`
    OnCreateCommand      any    `json:"onCreateCommand,omitempty"`
    UpdateContentCommand any    `json:"updateContentCommand,omitempty"`
    PostCreateCommand    any    `json:"postCreateCommand,omitempty"`
    PostStartCommand     any    `json:"postStartCommand,omitempty"`
    PostAttachCommand    any    `json:"postAttachCommand,omitempty"`
    WaitFor              string `json:"waitFor,omitempty"`
}

func UnmarshalRecipe(raw []byte) (*Recipe, error) {
    // Use sigs.k8s.io/yaml UnmarshalStrict for field-name strictness.
    var r Recipe
    if err := yaml.UnmarshalStrict(raw, &r); err != nil {
        return nil, fmt.Errorf("recipes: unmarshal: %w", err)
    }
    return &r, nil
}
```

**Why `any` on lifecycle hooks:** Dev Containers spec allows three shapes per hook — string (single command), array (sequential commands), or object/array-of-arrays (parallel groups). A strongly-typed wrapper forces a custom `UnmarshalJSON` per hook that normalizes to a canonical `[][]string` form. See Pattern 4 below for the normalizer.

### Pattern 3: Loader pipeline (REC-02)

**What:** At API startup, walk `agents/*/recipe.yaml`, validate each, unmarshal into Recipe, run semantic post-checks (enum values, `config_flavor_of` validity, etc.), register in an in-memory map. On any failure, refuse to start.

**Example:**

```go
type Loader struct {
    root        string // "agents/"
    validator   *SchemaValidator
    cache       map[string]*Recipe
    templates   *TemplateRegistry
    logger      zerolog.Logger
}

func (l *Loader) LoadAll(ctx context.Context) error {
    entries, err := os.ReadDir(l.root)
    if err != nil {
        return fmt.Errorf("recipes: read root: %w", err)
    }
    loaded := make(map[string]*Recipe)
    for _, e := range entries {
        if !e.IsDir() || e.Name() == "schemas" || e.Name() == "community" {
            continue
        }
        id := e.Name()
        raw, err := os.ReadFile(filepath.Join(l.root, id, "recipe.yaml"))
        if errors.Is(err, os.ErrNotExist) {
            continue // directory without a recipe.yaml → ignore
        }
        if err != nil {
            return fmt.Errorf("recipes: read %s: %w", id, err)
        }
        if err := l.validator.ValidateYAML(raw); err != nil {
            return fmt.Errorf("recipes: validate %s: %w", id, err)
        }
        r, err := UnmarshalRecipe(raw)
        if err != nil {
            return fmt.Errorf("recipes: unmarshal %s: %w", id, err)
        }
        if r.ID != id {
            return fmt.Errorf("recipes: id mismatch in %s: recipe.yaml has id=%q", id, r.ID)
        }
        if err := l.semanticCheck(r); err != nil {
            return fmt.Errorf("recipes: semantic check %s: %w", id, err)
        }
        loaded[id] = r
    }
    // Second pass: resolve config_flavor_of references (D-12).
    for _, r := range loaded {
        if r.ConfigFlavorOf != "" {
            parent, ok := loaded[r.ConfigFlavorOf]
            if !ok {
                return fmt.Errorf("recipes: %s references unknown parent %q", r.ID, r.ConfigFlavorOf)
            }
            if parent.ConfigFlavorOf != "" {
                return fmt.Errorf("recipes: %s is a flavor of flavor %s — not allowed", r.ID, parent.ID)
            }
        }
    }
    l.cache = loaded
    l.logger.Info().Int("count", len(loaded)).Msg("recipes loaded")
    return nil
}
```

**Why:** Single startup pass with loud failure satisfies REC-02. The two-pass design handles `config_flavor_of` without ordering dependencies. SIGHUP (D-35) calls `LoadAll` again with a fresh map and atomically swaps under a `sync.RWMutex`.

### Pattern 4: Lifecycle hook normalization + errgroup parallel execution

**What:** Normalize every hook to `[][]string` (array of parallel groups, each group is argv). `string` becomes `[["sh", "-c", "<string>"]]`. `[]string` becomes `[[s1, s2, ...]]` (single sequential group). `[][]string` passes through. Then execute: each outer element runs in sequence; each element's argv is one `docker exec` call; if the outer element is itself an array-of-arrays, use errgroup.

**Example** (source: Dev Containers spec + Go errgroup patterns):

```go
// Hook is the normalized form of any lifecycle hook value.
// Outer dimension = sequence of phases. Inner dimension = argv in one phase.
// Multiple argvs in one phase are run sequentially within that phase.
// Parallelism: a SEPARATE hook form (array of Hooks) handles parallel groups.
type Hook [][]string

// NormalizeHook turns any | string | []string | [][]string into a canonical Hook.
func NormalizeHook(v any) (Hook, error) {
    switch x := v.(type) {
    case nil:
        return nil, nil
    case string:
        if x == "" {
            return nil, nil
        }
        return Hook{{"sh", "-c", x}}, nil
    case []any:
        if len(x) == 0 {
            return nil, nil
        }
        // Distinguish [string, string, ...] from [[string, ...], [string, ...]].
        if _, isStr := x[0].(string); isStr {
            argv := make([]string, 0, len(x))
            for _, item := range x {
                s, ok := item.(string)
                if !ok {
                    return nil, fmt.Errorf("hook: mixed-type array")
                }
                argv = append(argv, s)
            }
            return Hook{argv}, nil
        }
        // Array of arrays — each inner array is a parallel group.
        var out Hook
        for _, item := range x {
            inner, ok := item.([]any)
            if !ok {
                return nil, fmt.Errorf("hook: expected array-of-arrays, got %T", item)
            }
            argv := make([]string, 0, len(inner))
            for _, s := range inner {
                str, ok := s.(string)
                if !ok {
                    return nil, fmt.Errorf("hook: non-string in argv")
                }
                argv = append(argv, str)
            }
            out = append(out, argv)
        }
        return out, nil
    default:
        return nil, fmt.Errorf("hook: unsupported type %T", v)
    }
}

// RunHook executes a normalized Hook against a running container.
// Each outer element is one parallel group (run via errgroup).
// Honors timeout from recipe.lifecycle.<name>_timeout_sec (default 10 min per D-30).
func (r *Runner) RunHook(ctx context.Context, containerID string, h Hook, timeout time.Duration) error {
    if len(h) == 0 {
        return nil
    }
    if timeout == 0 {
        timeout = 10 * time.Minute
    }
    ctx, cancel := context.WithTimeout(ctx, timeout)
    defer cancel()

    g, gctx := errgroup.WithContext(ctx)
    for _, argv := range h {
        argv := argv // capture
        g.Go(func() error {
            out, err := r.Exec(gctx, containerID, argv)
            if err != nil {
                return fmt.Errorf("hook exec %v: %w (output=%q)", argv, err, string(out))
            }
            return nil
        })
    }
    return g.Wait()
}
```

**Why errgroup:** `errgroup.WithContext` cancels sibling goroutines on first error, which matches D-26's "any hook exits non-zero → whole session errors out" semantic exactly. Hand-rolling with `sync.WaitGroup` + a result channel reimplements errgroup and loses the cancellation propagation.

### Pattern 5: RunWithLifecycle — the 6-hook sequencer

**What:** The new `Runner.RunWithLifecycle(ctx, recipe, opts)` method that D-24 requires. Creates the container, then executes hooks in strict D-25 order, respecting `waitFor` from D-26.

```go
type LifecycleSession struct {
    ContainerID string
    ReadyCh     chan struct{} // closed when waitFor hook completes
}

func (r *Runner) RunWithLifecycle(
    ctx context.Context,
    recipe *recipes.Recipe,
    opts RunOptions,
) (*LifecycleSession, error) {
    // 1. initializeCommand — runs on HOST (not in container). 02.5 no-op
    //    unless recipe declares host-side probe. Execute via os/exec with
    //    the same hardening as Runner.Exec (arg-level validation, timeout).
    if hook, err := recipes.NormalizeHook(recipe.Lifecycle.InitializeCommand); err != nil {
        return nil, err
    } else if len(hook) > 0 {
        if err := r.runHostHook(ctx, hook, hookTimeout(recipe, "initialize")); err != nil {
            return nil, fmt.Errorf("initializeCommand: %w", err)
        }
    }

    // 2. Create + start container.
    containerID, err := r.Run(ctx, opts)
    if err != nil {
        return nil, fmt.Errorf("create container: %w", err)
    }

    // 3. onCreateCommand (skipped on restart — use a container label to detect).
    if !opts.Labels["ap.already-created"] == "true" { // pseudo; use real flag from DB
        if err := r.runHookByName(ctx, containerID, recipe, "onCreateCommand"); err != nil {
            r.teardown(ctx, containerID)
            return nil, err
        }
    }

    // 4. updateContentCommand (every start).
    if err := r.runHookByName(ctx, containerID, recipe, "updateContentCommand"); err != nil {
        r.teardown(ctx, containerID)
        return nil, err
    }

    // 5. postCreateCommand (installs go here — D-25).
    //    CRITICAL: secrets are NOT available here per D-29.
    if err := r.runHookByName(ctx, containerID, recipe, "postCreateCommand"); err != nil {
        r.teardown(ctx, containerID)
        return nil, err
    }

    // At this point the entrypoint shim populates /run/secrets/<name> (D-37).

    // 6. postStartCommand (daemonize agents here).
    if err := r.runHookByName(ctx, containerID, recipe, "postStartCommand"); err != nil {
        r.teardown(ctx, containerID)
        return nil, err
    }

    // 7. postAttachCommand (bridge setup — opening FIFOs, etc.).
    if err := r.runHookByName(ctx, containerID, recipe, "postAttachCommand"); err != nil {
        r.teardown(ctx, containerID)
        return nil, err
    }

    return &LifecycleSession{ContainerID: containerID}, nil
}
```

**Key nuance — D-29 secret availability:** Phase 2's existing entrypoint shim reads `/run/secrets/*_key` at PHASE 2 of its execution (after `gosu agent`). 02.5 extends this so the shim *also* renders any `auth.files` templates at the same time. That means secrets become available to the agent process, but NOT to `onCreateCommand` or `updateContentCommand` — those run via `docker exec` as the agent user, before the tmpfs `/run/secrets` bind-mount is populated with the BYOK values the session handler generated for THIS session. This ordering matches D-29's "prevents API keys from leaking into install logs" rationale.

**Schema check for D-29:** the JSON Schema `allOf`/`if-then-else` should reject recipes whose `onCreateCommand` references `secret:` env vars or `auth.files` — those can only land in `postCreateCommand` or later.

### Pattern 6: Template renderer with hardened FuncMap (D-16)

**What:** A restrained `text/template` setup that can only stringify secret values into file content. No reflection, no `exec`, no file I/O helpers.

```go
package recipes

import (
    "bytes"
    "context"
    "fmt"
    "os"
    "path/filepath"
    "regexp"
    "strings"
    "text/template"
    "time"
)

var templateNameRE = regexp.MustCompile(`^[a-z0-9][a-z0-9_-]*\.tmpl$`)

const (
    templateMaxBytes    = 64 * 1024
    templateMaxDuration = 5 * time.Second
)

type TemplateRegistry struct {
    root string // "agents/"
    // Simple mtime-keyed cache per D-17.
    mu    sync.RWMutex
    cache map[string]cachedTmpl
}

type cachedTmpl struct {
    mtime time.Time
    tmpl  *template.Template
}

var safeFuncs = template.FuncMap{
    "default": func(def, v any) any {
        if v == nil || v == "" {
            return def
        }
        return v
    },
    "quote": func(s string) string {
        return strconv.Quote(s)
    },
    "lower": strings.ToLower,
    "upper": strings.ToUpper,
    "trim":  strings.TrimSpace,
}

// Render loads agents/<recipeID>/templates/<name>.tmpl, validates the name,
// rejects symlinks and path traversal, caches by mtime, and renders with
// the provided context inside a 5s timeout with a 64 KiB output cap.
func (r *TemplateRegistry) Render(
    ctx context.Context,
    recipeID, name string,
    data any,
) (string, error) {
    if !templateNameRE.MatchString(name) {
        return "", fmt.Errorf("template: invalid name %q", name)
    }
    dir := filepath.Join(r.root, recipeID, "templates")
    path := filepath.Join(dir, name)

    // Reject symlinks.
    info, err := os.Lstat(path)
    if err != nil {
        return "", fmt.Errorf("template: stat: %w", err)
    }
    if info.Mode()&os.ModeSymlink != 0 {
        return "", fmt.Errorf("template: symlink forbidden: %s", path)
    }

    // Reject path traversal: resolved path must still be inside dir.
    absDir, _ := filepath.Abs(dir)
    absPath, _ := filepath.Abs(path)
    if !strings.HasPrefix(absPath, absDir+string(os.PathSeparator)) {
        return "", fmt.Errorf("template: path escape: %s", path)
    }

    // Cache check.
    r.mu.RLock()
    cached, ok := r.cache[path]
    r.mu.RUnlock()
    var tmpl *template.Template
    if ok && cached.mtime.Equal(info.ModTime()) {
        tmpl = cached.tmpl
    } else {
        raw, err := os.ReadFile(path)
        if err != nil {
            return "", fmt.Errorf("template: read: %w", err)
        }
        tmpl, err = template.New(name).Funcs(safeFuncs).Option("missingkey=error").Parse(string(raw))
        if err != nil {
            return "", fmt.Errorf("template: parse: %w", err)
        }
        r.mu.Lock()
        if r.cache == nil {
            r.cache = make(map[string]cachedTmpl)
        }
        r.cache[path] = cachedTmpl{mtime: info.ModTime(), tmpl: tmpl}
        r.mu.Unlock()
    }

    // Execute with timeout + size cap.
    ctx, cancel := context.WithTimeout(ctx, templateMaxDuration)
    defer cancel()
    type result struct {
        s   string
        err error
    }
    ch := make(chan result, 1)
    go func() {
        var buf bytes.Buffer
        lw := &limitedWriter{w: &buf, max: templateMaxBytes}
        err := tmpl.Execute(lw, data)
        ch <- result{s: buf.String(), err: err}
    }()
    select {
    case <-ctx.Done():
        return "", fmt.Errorf("template: timeout")
    case res := <-ch:
        if res.err != nil {
            return "", fmt.Errorf("template: exec: %w", res.err)
        }
        return res.s, nil
    }
}

type limitedWriter struct {
    w   io.Writer
    max int
    n   int
}

func (l *limitedWriter) Write(p []byte) (int, error) {
    if l.n+len(p) > l.max {
        return 0, fmt.Errorf("template: output exceeds %d bytes", l.max)
    }
    n, err := l.w.Write(p)
    l.n += n
    return n, err
}
```

**Security notes:**
- `Option("missingkey=error")` — referencing an undefined `.secrets.foo` becomes a loud error, not silent empty string. This defends against typos that would produce an auth-file with an empty API key (and thus bake an authless config).
- `safeFuncs` excludes `printf` (format string injection surface), `index`/`slice` (arbitrary attribute access), `js`/`html` (would HTML-encode YAML quotes), and anything from `sprig`.
- Symlink rejection + path-prefix check defeats `{"template": "../../../../etc/passwd.tmpl"}` attacks on recipes authored by future community contributors (in-scope because Phase 8 bootstrap writes recipes into `agents/community/`).
- The goroutine+channel pattern is needed because `text/template.Execute` cannot be cancelled mid-render. The goroutine will leak if the template is malicious (infinite loop via `range` on an infinite sequence — which our `data` shape cannot produce since secrets are flat map[string]string, but defense in depth).

**Citation:** `[CITED: oligo.security/blog/safe-by-default-or-vulnerable-by-design-golang-server-side-template-injection]`, `[CITED: snyk.io/articles/understanding-server-side-template-injection-in-golang/]` — SSTI analysis confirms text/template without user-controlled FuncMap is safe; these patterns close the remaining attack surface (symlink, output flood, infinite loop, undefined key).

func (s *DevEnvSource) Resolve(ref string) (string, error) {
    // Callers pass values that MAY start with "secret:".
    key, ok := strings.CutPrefix(ref, "secret:")
    if !ok {
        return ref, nil // literal value
    }
    switch key {
    case "anthropic-api-key", "anthropic-key":

            return "", ErrSecretMissing
        }
        return s.AnthropicKey, nil
    case "openrouter-api-key":
        if s.OpenRouterKey == "" {
            return "", ErrSecretMissing
        }
        return s.OpenRouterKey, nil
    default:
        if v, ok := s.extras[key]; ok {
            return v, nil
        }
        return "", ErrSecretMissing
    }
}

**Session-start flow (session/handler.go):**

```go
// Materialize a recipe for a specific session: resolve every auth.env and
// auth.files ref, render templates, and produce a MaterializedRecipe the
// runner can consume directly.
func (h *Handler) materialize(ctx context.Context, recipe *recipes.Recipe, provider, modelID string) (*MaterializedRecipe, error) {
    envOut := make(map[string]string, len(recipe.Auth.Env))
    for k, v := range recipe.Auth.Env {
        resolved, err := h.secrets.Resolve(v)
        if err != nil {
            return nil, fmt.Errorf("resolve env %q: %w", k, err)
        }
        envOut[k] = resolved
    }

    // Render auth.files templates with the resolved secrets as context.
    var files []MaterializedFile
    ctxData := map[string]any{
        "secrets": envOut,   // already-resolved values available to templates
        "env":     envOut,
        "recipe":  recipe,
    }
    for _, f := range recipe.Auth.Files {
        body, err := h.templates.Render(ctx, recipe.ID, f.Template, ctxData)
        if err != nil {
            return nil, fmt.Errorf("render %s/%s: %w", recipe.ID, f.Template, err)
        }
        files = append(files, MaterializedFile{
            Target: f.Target,
            Mode:   f.Mode,
            Body:   body,
        })
    }

    return &MaterializedRecipe{Recipe: recipe, Env: envOut, Files: files}, nil
}
```

**What the entrypoint shim does with the Materialized files:** The session handler writes each `MaterializedFile` to the per-session secrets directory at `/tmp/ap/secrets/<session-id>/` on the HOST (same as Phase 2 does today for `picoclaw-security.yml`), then bind-mounts that directory read-only into the container at `/run/secrets/`. The entrypoint shim (`deploy/ap-base/entrypoint.sh`) reads `/run/secrets/<f.Target.basename>` and writes it to `f.Target` with `chmod f.Mode`, then execs the agent. **02.5 does NOT need to change the entrypoint shim** — it already reads `/run/secrets/*` and renders into place. 02.5's change is on the Go side: the rendering is now template-driven from `agents/<id>/templates/` instead of Go-hardcoded `renderPicoclawSecurityYAML`.

### Anti-patterns to avoid

- **Don't `embed` recipes too** — recipes stay on disk for D-35 hot-reload (`SIGHUP`). Only the schema is embedded.
- **Don't use `yaml.Unmarshal` from `gopkg.in/yaml.v3` directly** — it uses `yaml:"..."` tags, not JSON tags, which would require dual-tagging every struct field. sigs.k8s.io/yaml gives JSON-tag-only cleanliness.

- **Don't let `onCreateCommand` see BYOK secrets** — D-29 rationale. The tmpfs bind-mount for `/run/secrets` is attached *before* the container starts, but the session-specific key file is written by the handler only after `postCreate` succeeds. Onboarding failures then cannot leak keys into image layers or bash history.
- **Don't recompile the schema per-request** — `SchemaValidator` compiles once at startup; `Validate` is the only hot-path call. Recompiling per-session would be a ~10 ms CPU pig.
- **Don't reinvent Phase 2's bridge logic** — `fifoMode` and `execMode` are production-tested and have documented threat models. Lift verbatim, interface them, move on. "Improvement" is the enemy of "no regression".
- **Don't trust `config_flavor_of` to work in v0.1** — no 02.5 recipe exercises it (D-12, D-40b). The schema supports it, the loader resolves it, but the code path is untested until Phase 4 lands openclaw. The loader must still reject bad flavor configs (flavor-of-flavor, flavor redeclaring install).

## Don't Hand-Roll

| Problem | Don't build | Use instead | Why |
|---------|-------------|-------------|-----|
| YAML parsing | Custom YAML tokenizer | `sigs.k8s.io/yaml` | YAML 1.2 is a 100-page spec; `go-yaml` has 8 years of corner-case fixes. |
| JSON Schema validation (Draft 2019-09) | Hand-rolled `if`-tree against the schema | `github.com/santhosh-tekuri/jsonschema/v6` | Draft 2019-09 has `$ref`, `$recursiveRef`, `unevaluatedProperties`, `if/then/else`, format validators. Hand-rolling is ~2 weeks of work and will be wrong. |
| Template engine | Custom `strings.ReplaceAll` per-field | `text/template` | Missing-key detection, struct traversal, no reflection needed. Stdlib. Zero deps. |
| Parallel goroutine error propagation | `sync.WaitGroup` + result channel | `golang.org/x/sync/errgroup` | errgroup is the idiomatic "fan out, cancel on first error" primitive. Hand-rolling reinvents it with more bugs. |
| Docker exec streaming | `os/exec` calling `docker exec` CLI | `moby/moby/client` ExecCreate/ExecAttach + stdcopy | CLAUDE.md §What NOT to Use explicitly forbids shelling out. Phase 2 already wraps the SDK in `docker.Runner`. |
| TTY/pty for lifecycle hooks | `creack/pty` + custom framing | `docker exec` with `AttachStdout/AttachStderr` non-TTY | Lifecycle hooks are non-interactive. TTY adds failure modes (resize, signal propagation) for zero benefit. ttyd inside the container handles the *interactive* web terminal, separately. |
| FIFO-based chat bridge | Named pipes written from Go via `unix.Mkfifo` on host | `docker exec "sh -c 'cat >> /run/ap/chat.in'"` with stdin piped | Phase 2's existing pattern. Works today. Hardened against shell-metacharacter injection (user text flows over stdin, never argv). See Phase 2 `bridge.go` lines 108–150 and THREAT NOTE T-02-04. |
| `exec_per_message` bridge | `os/exec` calling `docker exec` CLI | `Runner.Exec` (already in Phase 2) with argv directly | Same reasons as above. Phase 2's `execMode` (lines 92–106) is already safe and tested. |
| Secret vault encryption | Home-grown XOR or plaintext env vars | Phase 3's `pgcrypto` / age vault (arrives later) | 02.5 uses the dev env-var source per D-38. Phase 3 ships the real vault. Don't touch encryption in 02.5. |
| Dockerfile-authoring for runtime bases | `docker build` shelling from Go | Plain Dockerfiles + `make build-runtimes` | Build-time is a human concern, not a runtime concern. Keep images in the Makefile; keep Go code out of the build path. |
| Log redaction for secrets | String search-and-replace in every log call | zerolog hook that runs a regex over the marshaled event | D-39 specifies regex patterns for `sk-ant-…` and `sk-or-v1-…`. Implement as a zerolog `Hook` in `api/internal/logging/` — one place, every log line. |

**Key insight:** Every "don't hand-roll" above is a case where Phase 2 already picked the right tool. 02.5 adds two new libraries (sigs.k8s.io/yaml, santhosh-tekuri/jsonschema/v6) and reuses everything else. Resist scope creep.

## Common Pitfalls

### Pitfall 1: Mixing YAML tags and JSON tags on the Recipe struct

**What goes wrong:** Author adds `yaml:"runtime_family"` tag alongside `json:"runtime.family"`, expects sigs.k8s.io/yaml to honor the YAML tag. It doesn't — sigs.k8s.io/yaml goes YAML → JSON → struct, so only `json:"..."` tags are read. Field silently becomes zero-valued. Closed-enum fields (family, chat_io.mode) then fail semantic check, but the error points at the wrong field.

**Why:** Developer muscle memory from `gopkg.in/yaml.v3`. The two libraries look similar on import but have opposite tag behavior.

**How to avoid:** Use **only** `json:"..."` tags in the Recipe struct. Document loudly in the struct comment. Consider a `go vet`/custom linter check that rejects `yaml:"..."` tags in `internal/recipes/`.

**Warning signs:** `Recipe` loads successfully but fields are empty. Validation passes (JSON Schema saw the JSON-tagged names) but the in-memory struct has zero values.

### Pitfall 2: `text/template` swallowing undefined keys (produces empty strings, not errors)

**What goes wrong:** Template references `{{ .secrets.anthropik_key }}` (typo). Default template behavior: missing key → empty string → the rendered `.security.yml` contains `api_keys: ["",]`. Container boots, picoclaw starts, agent calls Anthropic with empty key, Anthropic returns 401, smoke-test fails with a confusing "unauthorized" error 30 seconds after a successful container start.

**Why:** `text/template`'s default `missingkey=default` behavior substitutes a zero value for undefined keys.

**How to avoid:** Always use `template.New(name).Option("missingkey=error")` in the renderer (Pattern 6 above). An undefined key becomes a loud `template: exec: template: security.yml:5:7: executing "security.yml" at <.secrets.anthropik_key>: map has no entry for key "anthropik_key"` — which points at the exact line and key name.

**Warning signs:** Container starts fine, lifecycle hooks pass, but the agent's first LLM call returns 401/403. Check rendered file contents before suspecting the provider.

### Pitfall 3: Dev Containers "parallel" = object syntax, not array-of-arrays

**What goes wrong:** Recipe author writes `postCreateCommand: [["cmd1"], ["cmd2"]]` expecting parallel execution per the Dev Containers spec. But the Dev Containers spec actually uses an **object syntax** (`{"label1": "cmd1", "label2": "cmd2"}`) for parallel groups, not array-of-arrays. 02.5's D-27 picks array-of-arrays as a **deliberate simplification** because it's more YAML-friendly, but this means our schema is NOT a strict superset of Dev Containers.

**Why:** Dev Containers' choice of object syntax is confusing (the key is just a label, not a name that matters); YAML authors find array-of-arrays more natural. 02.5 chose the latter for authoring clarity but must document the divergence.

**How to avoid:** Document in `agents/schemas/recipe.schema.json` `$comment` that 02.5's parallel semantics are "array-of-arrays for parallel groups, string or array-of-strings for sequential". Add a clear example comment in every reference recipe's `lifecycle` block. Reject the Dev Containers object syntax at schema level with a `not: {type: object}` guard on each hook — clean error instead of silent misinterpretation.

**Warning signs:** `postCreateCommand` executes commands sequentially when the recipe author expected parallel. Install time is the sum of the slowest two, not the max.

`[CITED: github.com/devcontainers/spec/blob/main/docs/specs/devcontainer-reference.md]` — the official spec for reference.

### Pitfall 4: FIFO chat bridge deadlock on container restart

**What goes wrong:** Phase 2's entrypoint shim holds `/run/ap/chat.in` and `/run/ap/chat.out` open via `exec 3<>"$FIFO_IN"; exec 4<>"$FIFO_OUT"` — a critical Pitfall 2 fix already captured in the Phase 2 entrypoint. If 02.5's `postStartCommand` assumes the FIFOs are already open and restarts the agent process without holding the fds from the new process's parent, the first `cat >> /run/ap/chat.in` from the Go bridge blocks forever waiting for a reader.

**Why:** POSIX FIFO semantics — `open()` blocks until both a reader and writer are present, unless `O_NONBLOCK` or `O_RDWR` is used.

**How to avoid:** 02.5's `postStartCommand` for picoclaw must NOT try to recreate or reopen the FIFOs. The Phase 2 entrypoint shim already handles this (`[ -p "$FIFO_IN" ] || mkfifo`). 02.5's `postAttachCommand` is allowed to `test -p /run/ap/chat.in` to verify the FIFO exists, but MUST NOT `cat < /run/ap/chat.in`  or `echo > /run/ap/chat.out` — those would consume bytes or introduce spurious content.

**Warning signs:** Smoke test's first `POST /api/sessions/:id/message` hangs for the full `response_timeout_sec` (60s) and returns `ErrTimeout`.

**Reference:** `deploy/ap-base/entrypoint.sh` lines 56–60 (exec 3<>, exec 4<>) is the correct pattern. See Phase 2's Pitfall 2 in `02-DISCUSSION-LOG.md`.

### Pitfall 5: `config_flavor_of` inheritance exists in schema but is untested in 02.5

**What goes wrong:** Schema declares `config_flavor_of` as a valid field per D-12. Loader resolves it per the Pattern 3 second-pass. But no 02.5 recipe exercises it (D-40b — picoclaw is standalone). Phase 4 adds openclaw as the parent and re-does picoclaw as the flavor. If the loader has a bug in the inheritance path, it will surface in Phase 4, not 02.5.

**Why:** Scope discipline — testing a feature nothing uses is busywork, but untested code paths rot.

**How to avoid:** Write **unit tests** for the inheritance resolver in Plan 02.5-01, even though no runtime recipe exercises it. Use synthetic test fixtures: a parent recipe + a flavor recipe + a flavor-of-flavor recipe + a flavor-redeclaring-install recipe. Loader rejects the last two with specific error messages; accepts the first two. Coverage target ≥90% for the resolver function.

**Warning signs:** None in 02.5. Failure mode is latent until Phase 4 plans openclaw+picoclaw-as-flavor.

### Pitfall 6: Semantic check ordering — catching bad recipes AFTER schema validation

**What goes wrong:** JSON Schema validates `runtime.family == "python"`. Loader semantic check then tests `Recipe.Install.Type == "pip"` and runs through the D-19 `uv pip install` translation. But recipe author has `install.type: git_build` with `runtime.family: python`, which passes the schema (both are valid enums) but semantically means "build from source with Go toolchain against a Python base" — nonsense. Silent misconfiguration produces a broken container at session-start time, not at load time.

**Why:** JSON Schema can enforce field-level constraints but struggles with cross-field invariants beyond simple `if/then/else`.

**How to avoid:** Every semantic invariant gets a Go-side check in `loader.semanticCheck(recipe)`. Examples: (a) `install.type=git_build` → recipe must declare its own `Dockerfile`; (b) `install.type=pip` → `runtime.family=python`; (c) `install.type=npm` → `runtime.family=node`; (d) `chat_io.mode=fifo` → at least one `postAttachCommand` that verifies the FIFO exists; (e) any `secret:` ref in `auth.env` → `secrets_schema` declares the corresponding name. Fail loudly at load time with the recipe path + the invariant name.

**Warning signs:** Session-start errors that point at Docker or the agent binary, but where the real bug is a recipe-authoring mistake.

### Schema validation against recipe.yaml

```go
// Source: adapted from pkg.go.dev/github.com/santhosh-tekuri/jsonschema/v6
// + sigs.k8s.io/yaml docs
func (l *Loader) validateAndUnmarshal(path string) (*Recipe, error) {
    raw, err := os.ReadFile(path)
    if err != nil {
        return nil, err
    }

    // Step 1: YAML → JSON (strict — rejects duplicate keys).
    jsonBytes, err := yaml.YAMLToJSONStrict(raw)
    if err != nil {
        return nil, fmt.Errorf("yaml→json: %w", err)
    }

    // Step 2: Validate JSON against the compiled schema.
    inst, err := jsonschema.UnmarshalJSON(bytes.NewReader(jsonBytes))
    if err != nil {
        return nil, fmt.Errorf("unmarshal for validate: %w", err)
    }
    if err := l.validator.schema.Validate(inst); err != nil {
        // santhosh-tekuri provides Detailed() / Basic() / Flag() output modes.
        // Use Detailed() so the error message includes the JSON Pointer to
        // the offending field — exactly what the planner needs to debug.
        var ve *jsonschema.ValidationError
        if errors.As(err, &ve) {
            return nil, fmt.Errorf("schema: %s", ve.DetailedOutput())
        }
        return nil, fmt.Errorf("schema: %w", err)
    }

    // Step 3: Strict unmarshal into Recipe.
    var r Recipe
    if err := yaml.UnmarshalStrict(raw, &r); err != nil {
        return nil, fmt.Errorf("unmarshal recipe: %w", err)
    }
    return &r, nil
}
```

### Gate A smoke-test matrix runner (shell sketch)

```bash
#!/bin/bash
# scripts/smoke-test-matrix.sh — Gate A per D-46
# Source: extends Phase 2's scripts/smoke-e2e.sh pattern.
set -euo pipefail

RECIPES=("aider" "picoclaw")
PROVIDERS=("anthropic" "openrouter")
API_URL="${API_URL:-http://127.0.0.1:8080}"
PROBE="whoareyou"

results=()
pass=0
fail=0
skip=0

for recipe in "${RECIPES[@]}"; do
  for provider in "${PROVIDERS[@]}"; do
    case "$provider" in
      anthropic)  key="$AP_DEV_BYOK_KEY" ;;
      openrouter) key="$AP_DEV_OPENROUTER_KEY" ;;
    esac
    if [ -z "${key:-}" ]; then
      echo "[SKIP] $recipe × $provider — no key"
      results+=("$recipe|$provider|SKIP")
      skip=$((skip+1))
      continue
    fi

    # Does this recipe declare this provider? Ask the API.
    if ! curl -sf "$API_URL/api/recipes/$recipe" | jq -e ".providers[] | select(.id == \"$provider\")" >/dev/null; then
      echo "[SKIP] $recipe × $provider — recipe does not declare provider"
      results+=("$recipe|$provider|SKIP")
      skip=$((skip+1))
      continue
    fi

    # First model matching this provider (D-48).
    model=$(curl -sf "$API_URL/api/recipes/$recipe" | jq -r "[.models[] | select(.provider == \"$provider\")][0].id")
    [ -z "$model" ] && { echo "[SKIP] $recipe × $provider — no model"; skip=$((skip+1)); continue; }

    # Spawn session.
    session_id=$(curl -sf -X POST "$API_URL/api/sessions" \
      -H "content-type: application/json" \
      -d "{\"recipe\":\"$recipe\",\"provider\":\"$provider\",\"model\":\"$model\"}" \
      | jq -r .id)
    if [ -z "$session_id" ] || [ "$session_id" = "null" ]; then
      echo "[FAIL] $recipe × $provider — session create failed"
      results+=("$recipe|$provider|FAIL")
      fail=$((fail+1))
      continue
    fi

    # Send whoareyou.
    reply=$(curl -sf -X POST "$API_URL/api/sessions/$session_id/message" \
      -H "content-type: application/json" \
      -d "{\"text\":\"$PROBE\"}" \
      | jq -r .text 2>/dev/null || echo "")

    # Teardown.
    curl -sf -X DELETE "$API_URL/api/sessions/$session_id" >/dev/null || true

    # Assert no dangling container.
    if docker ps -a --filter "label=session_id=$session_id" -q | grep -q .; then
      echo "[FAIL] $recipe × $provider — dangling container after DELETE"
      results+=("$recipe|$provider|FAIL")
      fail=$((fail+1))
      continue
    fi

    if [ -n "$reply" ] && [ "$reply" != "null" ]; then
      echo "[PASS] $recipe × $provider × $model → ${reply:0:60}…"
      results+=("$recipe|$provider|PASS")
      pass=$((pass+1))
    else
      echo "[FAIL] $recipe × $provider — empty reply"
      results+=("$recipe|$provider|FAIL")
      fail=$((fail+1))
    fi
  done
done

echo ""
echo "=== Gate A Matrix Result ==="
printf '%s\n' "${results[@]}" | column -t -s '|'
echo "=== $pass pass / $fail fail / $skip skip ==="
[ "$fail" -gt 0 ] && exit 1 || exit 0
```

**Integration with Phase 2's existing `scripts/smoke-e2e.sh`:** Phase 2's script runs one recipe end-to-end; 02.5's `smoke-test-matrix.sh` is a superset that iterates the product. Reuse the curl patterns; don't duplicate the teardown logic.

   - **RESOLVED: see Plan 02.5-07** — L2 eager verify measures real install time; the aider recipe overrides `postCreate_timeout_sec: 900` if install exceeds 5 min, documented in the recipe comment.

5. **Does the entrypoint shim need to be updated to render templates, or does the Go handler render before container start?**
   - What we know: D-37 says the entrypoint shim renders templates. Phase 2's current entrypoint reads files from `/run/secrets/` that were pre-rendered by the Go handler.
   - What's unclear: Moving rendering into the shim means Go-side template code runs inside the container (requires shipping Go template engine in ap-base, which it does not currently have). Keeping rendering Go-side means the shim just `cp`s pre-rendered files from `/run/secrets/<name>` to the target path.
   - **RESOLVED: see Plan 02.5-05** — Go handler renders templates before `Runner.RunWithLifecycle`; the entrypoint shim (unchanged from Phase 2) copies pre-rendered files from `/run/secrets/` to their target paths. D-37's wording "renders at container start" is satisfied in spirit: the rendered file lands in the container at entry.

### Known threat patterns for this stack

| Pattern | STRIDE | Standard mitigation |
|---------|--------|---------------------|
| YAML duplicate-key injection (two `auth.env` blocks, second overrides) | Tampering | `yaml.YAMLToJSONStrict` rejects duplicates |
| YAML billion-laughs / zip bomb | DoS | sigs.k8s.io/yaml has bounded recursion; 02.5 also caps file size at load time (plan-level check: reject `recipe.yaml` > 64 KiB) |
| JSON Schema `$ref` recursion DoS | DoS | santhosh-tekuri v6 detects and rejects recursive `$ref` without a `$id` anchor |
| Template path traversal (`../../../etc/passwd.tmpl`) | Tampering | Pattern 6 path allowlist regex + symlink rejection + prefix check |
| Template SSTI via malicious FuncMap helper | Elevation of Privilege | Closed FuncMap allowlist: `default, quote, lower, upper, trim` only |
| Template output flood (DoS log ingestion) | DoS | 64 KiB output cap (`limitedWriter`), 5s render timeout |
| Secret leak via log line containing BYOK key | Information Disclosure | D-39 zerolog hook regex redaction for `sk-ant-[a-zA-Z0-9-_]{50,}`, `sk-or-v1-[a-f0-9]{64}`, plus `ulimit -c 0` from SEC-05 (Phase 3) |
| Secret leak via rendered template in logs | Information Disclosure | Rendered templates go to container tmpfs, never logged; D-16 rule |
| Container restart inherits stale FIFOs | DoS (chat bridge hang) | Phase 2 entrypoint holds FDs 3 and 4 open from PID 1 (`exec 3<>`, `exec 4<>`) |
| `onCreateCommand` leaks BYOK keys into image layer | Information Disclosure | D-29: secrets not available during onCreate/updateContent; schema rejects recipes that reference `secret:` refs in those hooks |
| Recipe drop-in from Phase 8 bootstrap writes hostile `lifecycle` command | Elevation of Privilege | Closed-enum `install.type` + Dockerfile inheritance from `ap-runtime-*` (no `FROM scratch`) + sandbox posture (cap_drop ALL, no_new_privs) + template allowlist |
| Flavor-of-flavor recipe inheritance cycle | DoS | Loader rejects chain depth > 1 (Pattern 3 second pass) |
| `config_flavor_of` flavor redeclares `install` | Confused Deputy | Loader rejects per D-12; explicit error |
| Model ID not in recipe's declared `models[]` | Bad Request | D-53 validation rejects at handler level with `model_not_supported` error code |
| Provider not in recipe's `providers[]` | Bad Request | D-53 validation rejects with `provider_not_supported` |
| Stale container from previous session poisons matrix test | DoS | D-46 asserts zero dangling containers after DELETE; `docker ps --filter label=session_id=...` check in smoke-test-matrix.sh |

**Full CRIT-1 bootstrap mitigation is deferred to Phase 7.5 + Phase 8.** 02.5 is curated-only. The template-sandbox + schema-closed-enums are the "how we make this safe when bootstrap starts writing recipes into `agents/community/`" preparation.
