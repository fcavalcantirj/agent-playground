## Decisions Made

See `key-decisions` in frontmatter. The load-bearing ones:

1. **COPY --from over apt install** — deterministic version pin (0.11.6 for uv, 22-slim for node), no cache pollution, no apt wait time in the build.
2. **USER root for COPY then USER agent at end** — required so COPY --from can place files under /usr/local/{bin,lib} and the RUN block can chown agent-owned cache dirs, but the final layer is unprivileged.
3. **No ENTRYPOINT / CMD** — inherited from ap-base unchanged. This is the D-22 rule; breaking it would silently regress Phase 2's tini + entrypoint shim.
4. **Debian trixie for both, never Alpine** — D-20 + Phase 2 D-04. Alpine's musl libc routinely breaks native Python wheels and Node native addons; planning guardrail says trixie everywhere.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Plan Dockerfile sketch omitted libpython3.13.so.1.0**

- **Found during:** Task 1 (first build of ap-runtime-python, before proceeding to Task 2 checkpoint)
- **Issue:** The plan's Dockerfile sketch copied `/usr/local/bin/python3.13` and `/usr/local/lib/python3.13` (stdlib) from the Astral uv image, but NOT `/usr/local/lib/libpython3.13.so.1.0`. The python3.13 binary in that image is dynamically linked against libpython3.13.so.1.0 — without it, `python3 --version` fails with `error while loading shared libraries: libpython3.13.so.1.0: cannot open shared object file`.
- **Fix:** Added one more `COPY --from=ghcr.io/astral-sh/uv:0.11.6-python3.13-trixie /usr/local/lib/libpython3.13.so.1.0 /usr/local/lib/libpython3.13.so.1.0` to `deploy/ap-runtime-python/Dockerfile` immediately after the existing COPYs. Left an inline comment explaining why the file is load-bearing.
- **Files modified:** `deploy/ap-runtime-python/Dockerfile`
- **Verification:** `docker run --rm --user agent ap-runtime-python:v0.1.0-3.13 python3 --version` now prints `Python 3.13.13` and exits 0. Confirmed again at Smoke Check 1 during the Task 2 human-verify gate.
- **Committed in:** `8001875` (rolled into the Task 1 commit — the auto-fix happened before the Task 1 build ever succeeded, so there was no earlier working state to separate from)

This is a Rule 1 (bug fix) — the planned Dockerfile would not have produced a working image. Under Rule 1 the executor applies the fix inline and keeps going; the plan sketch was wrong about what the upstream image layout contains, and correcting it is in-scope for Task 1 as written (the task's acceptance criteria include a working `docker image inspect` + the human-verify python3 smoke check, which is only reachable with the .so file present).
