# Phase 02.5 Plan 03: RunWithLifecycle + Dev Containers hook sequencer Summary

**One-liner:** Ship the Dev Containers 6-hook sequencer on `*docker.Runner` with errgroup parallel groups, waitFor gating, per-hook timeouts, and teardown-on-failure — powered by a pure-Go hook normalizer that lives in the recipes package to preserve the docker→recipes import direction.

## Scope

- Task 1 — `recipes.Hook` + `NormalizeHook` + `HookTimeout` helper + `DefaultHookTimeout` constant
- Task 2 — `Runner.RunWithLifecycle` + `LifecycleSession` + host-side `initializeCommand` + in-container 5-hook sequencer + errgroup parallel dispatch + per-hook timeout + failure teardown

## What Landed

### `api/internal/recipes/hook.go` (pure Go, no external deps beyond stdlib)

- `type Hook [][]string` — outer = parallel groups, inner = argv of one `docker exec` call
- `DefaultHookTimeout = 10 * time.Minute`
- `NormalizeHook(v any) (Hook, error)` — collapses `nil | "" | string | []any-of-strings | [][]any | []string` to canonical `Hook`
  - `string s` → `Hook{{"sh","-c",s}}` (sh-c wrap, single group, single argv)
  - `[]any` all-strings → `Hook{argv}` (literal exec form, single group, single argv)
  - `[]any` all-`[]any` → parallel groups, each inner is a single argv
  - Mixed / non-string / empty-inner / unsupported types → errors
- `HookTimeout(r *Recipe, name string) time.Duration` — reads `r.Lifecycle.<hook>TimeoutSec`, falls back to default

### `api/internal/recipes/hook_test.go` (16 tests, all green)

Covers: nil, empty string, plain string (sh-c wrap), array-of-strings (aider case), array-of-arrays (parallel groups), empty array, mixed type error, non-string inner error, empty inner argv error, go-native `[]string` convenience path, unsupported root type error, unsupported first-element type error, `HookTimeout` default for all 6 named hooks, nil recipe safety, per-hook overrides for all 6 hooks, unknown hook name fall-through.

### `api/pkg/docker/runner_lifecycle.go` (new file, *Runner method)

- `type LifecycleSession { ContainerID, Recipe, ReadyCh }`
- `(r *Runner) RunWithLifecycle(ctx, recipe, opts) (*LifecycleSession, error)` — production entry, wires real deps
- `(r *Runner) runWithLifecycleWithDeps(ctx, recipe, opts, deps)` — testable core
- `runHookByName(ctx, cid, recipe, name, execFn) error` — reads named hook, normalizes, dispatches under errgroup with per-hook timeout
- `runHostInitialize(ctx, recipe) error` — same pattern for host-side `initializeCommand` via `os/exec.CommandContext`
- `(r *Runner) lifecycleTeardown(_, cid) error` — bounded Stop+Remove with 30s context.Background

Execution sequence (strict D-25 order):

1. `initializeCommand` — host (before any container exists)
2. `Runner.Run(ctx, opts)` — create + start container
3. `onCreateCommand` — skipped iff `opts.Labels["ap.already-created"]=="true"`
4. `updateContentCommand`
5. `postCreateCommand`  ← default `waitFor` — closes `ReadyCh` after this returns
6. `postStartCommand`
7. `postAttachCommand`

Any non-nil error at steps 3-7 → bounded teardown (Stop+Remove) → wrapped error; `errors.Join` preserves both error chains when teardown itself also fails.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 - Blocker] macOS `exec.LookPath` for host hook test binaries**
- **Found during:** Task 2 initial test run
- **Issue:** Plan sketch had `/bin/true` and `/bin/false` hardcoded. macOS ships those at `/usr/bin/true` and `/usr/bin/false` — test failed with `fork/exec /bin/true: no such file or directory`.
- **Fix:** Replaced hardcoded paths with `exec.LookPath("true")` / `exec.LookPath("false")` so the test works on both Linux and macOS dev boxes.
- **Files modified:** `api/pkg/docker/runner_lifecycle_test.go`
- **Commit:** `3e1a223` (rolled into Task 2's commit — single-commit fix during RED→GREEN loop)

**2. [Rule 2 - Robustness] Teardown context must not derive from cancelled parent ctx**
- **Found during:** Task 2 implementation
- **Issue:** The plan sketch had `r.Stop(ctx, cid, nil)` using the caller's ctx. If the failure path is triggered by the per-hook `context.WithTimeout` firing, that derived ctx is already `Done`, so `Stop`/`Remove` on the daemon would return immediately with `context.DeadlineExceeded` and LEAK the container.
- **Fix:** `lifecycleTeardown` creates a fresh 30s `context.WithTimeout(context.Background(), ...)` that is NOT derived from the caller's ctx. Also simplified to `r.Stop(tctx, cid)` because the real Runner.Stop only takes (ctx, id).
- **Files modified:** `api/pkg/docker/runner_lifecycle.go` (production code, not just tests)
- **Commit:** `3e1a223`
- **Also noted in decisions (key-decisions section above).**

**3. [Rule 1 - Bug] Plan sketch used `r.Stop(ctx, cid, nil)` 3-arg form that doesn't exist**
- **Found during:** Task 2 implementation
- **Issue:** Phase 2's `Runner.Stop` is `(ctx, containerID) error` — no timeout pointer. Plan sketch copied an older signature.
- **Fix:** Use `r.Stop(tctx, cid)` in `lifecycleTeardown`.
- **Files modified:** `api/pkg/docker/runner_lifecycle.go`
- **Commit:** `3e1a223`

## Key Decisions (preserved verbatim for forward reference)

1. **Test seam = unexported `lifecycleDeps`** with `createStartContainer`, `exec`, `teardown`, `now` — chosen over DockerClient mock because the interesting logic is the sequencer, not the SDK plumbing (already covered by Phase 2 runner_test.go).
2. **Teardown uses fresh `context.Background` + 30s bound**, NOT derived from caller ctx — so teardown still runs after per-hook timeout cancels the caller ctx.
3. **`errors.Join(hookErr, teardownErr)`** preserves both chains on double-failure; tests assert both substrings present.
4. **errgroup promotion** — `golang.org/x/sync` was already indirect; `go get golang.org/x/sync/errgroup@latest` promoted to direct v0.20.0. No new node in the dep graph.
5. **Hook normalizer lives in `recipes` package, not `docker`**, to preserve the `docker → recipes` import direction (verified by `go list -deps`).
