# Phase 02.5 Plan 09: Sessions + Recipes HTTP Wiring Summary

**One-liner:** Wired the entire 02.5 substrate (Loader, TemplateRegistry, Materialize, RunWithLifecycle, BridgeRegistry) into the session HTTP handler, added GET /api/recipes public catalog endpoints, and deleted the Phase 2 LegacyRecipe hardcoded catalog in a single atomic swap.

## What Shipped

### Task 1 — Recipes endpoints + error envelope (commit e99cbce)
- `api/internal/handler/errors.go` — D-54 error code constants (`recipe_not_found`, `provider_not_supported`, `model_not_supported`, `secret_missing`, `template_render_failed`, `lifecycle_hook_failed`, `chat_bridge_unsupported_mode`, plus the Phase 1 baseline codes) + `WriteError` helper emitting the stable `{error:{code,message}}` envelope.
- `api/internal/handler/recipes.go` — `RecipePublicView` allowlist projection; `GET /api/recipes` with AND-combined filters (`?family`, `?tier`, `?license`, `?provider`); `GET /api/recipes/:id` with 404 on miss. `RecipeLoaderIface` is a narrow interface so tests inject a fake without touching the filesystem.
- `api/internal/handler/recipes_test.go` — 10 unit tests covering happy-path list, every filter, no-match, detail-found, 404 envelope, no-lifecycle-leak (threat T-02.5-02 guard), and nil-loader short-circuit.
- `api/internal/server/server.go` — registers the new routes on the authed `/api` group next to `/api/me`.

### Task 2 — Session handler swap + legacy deletion (commit 52a818a)
- `api/internal/session/handler.go` — complete rewrite:
  - `POST /api/sessions` accepts canonical `provider`/`model` fields (plus Phase 2 `model_provider`/`model_id` aliases folded at the top of the handler).
  - Recipe is loaded via `h.loader.Get(req.Recipe)`, provider/model validated against `recipe.Providers[]`/`recipe.Models[]`, session row inserted via store, then `session.Materialize` resolves every `secret:<name>` ref and renders every `auth.files` template.
  - `buildRunOptions` layers the recipe's `Runtime`/`Isolation`/`PersistentState`/`Launch` onto `DefaultSandbox`, writes each `MaterializedFile` to `<baseSecretsDir>/<sessID>/<basename>` with mode 0600, and adds the `:ro` bind-mount at `/run/secrets`.
  - `runner.RunWithLifecycle(ctx, recipe, opts)` sequences the full Dev Containers hook set (Plan 03).
  - `ap.user_id` / `ap.session_id` / `ap.recipe` / `ap.provider` labels stamped on every container for Phase 5 reconciliation (SBX-09).
  - `POST /api/sessions/:id/message` dispatches through `bridge.BridgeRegistry.Dispatch(recipe.ChatIO.Mode)` directly — no more Legacy→Recipe synth step. `ErrUnsupportedMode` surfaces as `chat_bridge_unsupported_mode`.
  - `DELETE /api/sessions/:id` unchanged semantics; per-session secrets dir is os.RemoveAll'd alongside the container teardown.
- `api/internal/session/handler_test.go` — 19 pure-unit tests against fake `SessionStore`, `ContainerRunner`, `RecipeLoader`, `SecretSource`, `TemplateRenderer`, and `BridgeDispatcher`. Covers every new error code, the backwards-compat alias, provider defaulting on single-provider recipes, model-provider binding validation, lifecycle-hook failure, conflict, unauthorized, delete ownership, bridge dispatch, unsupported mode, and timeout.
- `api/internal/session/bridge.go` — reduced to a 30-line re-export shim (`session.ErrTimeout`, `session.RunnerExec`). `synthRecipeFromLegacy`, `mapLegacyChatIOMode`, `errNilRecipe`, and the legacy `SendMessage(*LegacyRecipe)` signature are gone.
- `api/internal/session/bridge_test.go` — rewritten as a compile-time assertion that `session.RunnerExec` is an alias of `bridge.RunnerExec` and that `session.ErrTimeout` points at the same sentinel as `bridge.ErrTimeout`.
- `api/internal/recipes/recipes.go` — gutted to a package-doc comment. All of `LegacyRecipe`, `LegacyAllRecipes`, `GetLegacy`, `LegacyAuthFile`, `ChatIOMode`, `ChatIOFIFO`, `ChatIOExec`, `ChatIO`, `ResourceOverrides`, and `renderPicoclawSecurityYAML` deleted.
- `api/internal/recipes/recipes_test.go` — deleted (only covered legacy types).
- `api/pkg/migrate/sql/0003_sessions_provider.sql` — idempotent `ADD COLUMN IF NOT EXISTS model_provider` + `CREATE INDEX IF NOT EXISTS idx_sessions_provider`. Because Phase 2's 002_sessions.sql already defines the column, this migration is effectively a **no-op ALTER plus index creation**, documented in the file comment.
- `api/cmd/server/main.go` — wires `recipes.NewTemplateRegistry` through `server.WithTemplateRegistry`, constructs `bridge.NewBridgeRegistry(runner, logger)` directly, and passes the new 7-arg `session.NewHandler` signature.

## Migration 0003 Disposition

**No-op ALTER IF NOT EXISTS.** Phase 2's 002_sessions.sql already declared `model_provider TEXT NOT NULL` on the sessions table, so there is no schema change; the migration exists to:
1. Be idempotent if run against a fresh database where 002 somehow didn't add the column.
2. Add the `idx_sessions_provider` index that Phase 5 billing queries ("all sessions on openrouter") will want.
3. Serve as an audit trail for the D-53 semantic shift: the column now carries whatever value the client sent in the `provider` field (with `model_provider` as a compat alias during one rollout wave).

## Backwards-Compat Alias Fields

`CreateSessionReq` keeps these optional fields for one rollout wave:

```go
type createRequest struct {
    Recipe   string `json:"recipe"`
    Provider string `json:"provider"`
    Model    string `json:"model"`

    // Phase 2 backwards-compat aliases.
    ModelProvider string `json:"model_provider,omitempty"`
    ModelID       string `json:"model_id,omitempty"`
}
```

The handler folds `ModelProvider`→`Provider` and `ModelID`→`Model` at the top of `create()`. `TestCreateSession_BackwardsCompat_ModelProviderAlias` pins this behavior. Phase 4 deletes the alias fields once Plan 10's smoke-matrix.sh and any external clients have migrated.

## Error Code Surface (D-54)

Every session/recipe endpoint now emits a stable `{error:{code,message}}` envelope. Code strings:

| Code | HTTP | When |
|------|------|------|
| `recipe_not_found` | 404 | `GET /api/recipes/:id` miss, or `POST /api/sessions` with unknown recipe |
| `provider_not_supported` | 400 | Provider not in `recipe.Providers[]` |
| `model_not_supported` | 400 | Model not in `recipe.Models[]` or bound to the wrong provider |
| `secret_missing` | 400 | `session.Materialize` returned `ErrSecretMissing` |
| `template_render_failed` | 500 | `recipes.ErrTemplatePath`/`ErrTemplateSize`/`ErrTemplateTimeout` or any "template"/"render" error from Materialize |
| `lifecycle_hook_failed` | 500 | `RunWithLifecycle` returned an error prefixed with a hook name |
| `chat_bridge_unsupported_mode` | 500 | `bridge.ErrUnsupportedMode` from `BridgeRegistry.Dispatch` |
| `invalid_request` | 400 | Malformed JSON, missing required fields, provider required on multi-provider recipe |
| `conflict` | 409 | Store returned `ErrConflictActive` (one-active-per-user invariant) |
| `timeout` | 504 | Bridge returned `ErrTimeout` |
| `unauthorized` | 401 | Middleware did not set user id |
| `forbidden` | 403 | Session owner mismatch |
| `internal` | 500 | Fallback |
