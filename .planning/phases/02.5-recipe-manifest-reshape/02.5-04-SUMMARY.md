# Phase 02.5 Plan 04: Extract ChatBridge Interface Summary

One-liner: Promoted Phase 2's inline fifoMode / execMode dispatch into a proper ChatBridge interface with FIFOBridge, ExecBridge, and a closed-enum BridgeRegistry, keeping Phase 2 callers byte-equivalent via a thin shim.

## What Shipped

### New subpackage `api/internal/session/bridge/`

- `bridge.go` — defines the single-method `ChatBridge` interface, the shared `RunnerExec` narrow interface (alias-compatible with Phase 2 session.RunnerExec), the `ErrTimeout` + `ErrUnsupportedMode` sentinels, and `StripANSI` lifted byte-for-byte from Phase 2.
- `fifo.go` — `FIFOBridge` struct + `SendMessage`. Body lifted from Phase 2 `session.(*Bridge).fifoMode`. Reads `recipe.ChatIO.FIFO.FIFOIn / FIFOOut` with Phase 2 defaults (`/run/ap/chat.in`, `/run/ap/chat.out`) when the YAML leaves them unset. THREAT NOTE T-02-04 preserved: user text travels over stdin, the argv is the constant `["sh","-c","cat >> <fifo_in>"]` and `<fifo_in>` is recipe-controlled, never user-controlled.
- `exec.go` — `ExecBridge` struct + `SendMessage`. Body lifted from Phase 2 `session.(*Bridge).execMode`. Uses `slices.Clone` on the recipe's `ChatIO.ExecPerMessage.CmdTemplate`, appends `recipe.ModelFlag + modelID` (when both set), then appends user text as the final argv element. THREAT NOTE T-02-04b preserved: user text is a single argv slice element, never concatenated into a pre-formed command string, never shell-expanded.
- `registry.go` — `BridgeRegistry` with a `map[string]ChatBridge` keyed on the D-10 closed enum `{fifo, exec_per_message}`. `Dispatch(mode)` returns the matching bridge or wraps `ErrUnsupportedMode` with the rejected mode string. `NewBridgeRegistry(runner, logger)` is the sole constructor.

### Rewritten `api/internal/session/bridge.go` (shim)

- `session.Bridge` now holds a `*bridge.BridgeRegistry` and nothing else.
- `session.NewBridge(runner)` preserves the Phase 2 single-argument constructor signature; it wires a `zerolog.Nop()` logger into the subpackage so existing callers (`cmd/server/main.go`, `handler_test.go`, `handler.go` wiring) stay unchanged.
- `session.Bridge.SendMessage(ctx, cid, *recipes.LegacyRecipe, modelID, text)` is the preserved Phase 2 public method. It:
  1. Applies the response timeout at the shim layer using the raw `time.Duration` (so sub-second Phase 2 values are honored — see Deviations).
  2. Converts the legacy `ChatIOMode` constant (`"stdin_fifo"` / `"exec_per_message"`) to the D-10 enum key (`"fifo"` / `"exec_per_message"`) via `mapLegacyChatIOMode`.
  3. Projects `*LegacyRecipe` onto a minimally populated `*recipes.Recipe` via `synthRecipeFromLegacy` — only the fields the bridges actually read are filled in.
  4. Calls `registry.Dispatch(modeKey).SendMessage(...)`.
- `session.ErrTimeout` is re-exported from `bridge.ErrTimeout` (same `error` value, same identity, so `errors.Is` continues to work).
- `session.RunnerExec` is a `type ... = bridge.RunnerExec` alias, so `*docker.Runner` satisfies both symbols without adapter code.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] Sub-second response timeouts lost in legacy-to-Recipe conversion**
- **Found during:** Task 1 GREEN-phase verification against Phase 2 `TestBridge_Timeout`
- **Issue:** The `synthRecipeFromLegacy` adapter converted `time.Duration` → `int` seconds via `int(d / time.Second)`. Phase 2's `TestBridge_Timeout` uses `ResponseTimeout: 50 * time.Millisecond`, which truncates to 0 — the `FIFOBridge` then applied no timeout, the mockRunner always returned empty, and the poll loop ran until the test harness killed the whole suite after 10 minutes.
- **Root cause:** `RecipeChatIO.ResponseTimeoutSec int` is the Plan 01 YAML type; Phase 2 used `time.Duration`. The shim lost sub-second resolution because of the unit change. This is a legacy-adapter bug, not a bridge bug.
- **Fix:** Apply the `recipe.ChatIO.ResponseTimeout` at the shim layer using `context.WithTimeout(ctx, recipe.ChatIO.ResponseTimeout)` BEFORE calling `registry.Dispatch`. The synthetic recipe leaves `ResponseTimeoutSec` at 0 so the bridge does not double-wrap. Result: Phase 2 sub-second tests pass, Phase 02.5 second-resolution tests pass.
- **Files modified:** `api/internal/session/bridge.go`
- **Commit:** part of `122f35e`

## Behavior Invariants Preserved

- **T-02-04 (FIFO shell safety):** user text travels over stdin via `ExecWithStdin`, shell only sees the constant argv `["sh","-c","cat >> <fifo_in>"]` where `<fifo_in>` is recipe-controlled.
- **T-02-04b (Exec shell safety):** user text is the final argv element in a cloned `CmdTemplate` slice, passed to `runner.Exec` which hands it to dockerd as `[]string` — no shell interposed.
- **T-02.5-05b (Registry defense-in-depth):** unknown chat_io.mode values return `ErrUnsupportedMode` wrapped with the rejected mode string. Plan 01's schema closed-enum makes this path unreachable through normal flow.
- **Recipe non-mutation:** `slices.Clone` on the exec cmd template prevents bridges from mutating recipe state; tests verify this.
- **ANSI stripping:** unconditional in both FIFO and Exec modes (Phase 2 behavior).
- **ErrTimeout identity:** `session.ErrTimeout == bridge.ErrTimeout` (same sentinel), so existing `errors.Is` checks in `handler.go:319` continue to match.
