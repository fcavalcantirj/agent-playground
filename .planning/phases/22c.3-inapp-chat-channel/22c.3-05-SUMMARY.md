## Accomplishments

- **3-way contract switch shipped:** `services/inapp_dispatcher.py::_dispatch_http_localhost` matches on `inapp.contract` against the canonical names `openai_compat` / `a2a_jsonrpc` / `zeroclaw_native`. Each arm builds the verbatim request envelope per RESEARCH §Pitfall 6 + §Per-Recipe Feasibility Matrix. Unknown contract raises `RuntimeError(f"unknown_contract:{value}")`; the outer `_handle_row` catches it and routes to `_terminal_failure`.
- **Plan 04 store consumed verbatim:** Every state transition (`mark_forwarded`, `mark_done`, `mark_failed`, `fetch_pending_for_dispatch`) is imported from `inapp_messages_store`. No inlined SQL on `inapp_messages` in this module — the `grep -cE "queued|dispatched|delivered|ledger_seq|inapp_messages_state_history|NOTIFY|LISTEN"` gate returns 0.
- **Outbox-only fan-out:** Every terminal outcome (success → `inapp_outbound`; failure → `inapp_outbound_failed`) INSERTs an `agent_events` row with `published=false`. Plan 07's outbox pump fans these out; the dispatcher does NOT touch Redis. Tests assert `published is False` on the INSERTed row.
- **D-40 no-auto-retry implemented:** All 6 failure modes (timeout, 5xx, empty, container_not_ready, unknown_contract, recipe_lacks_inapp_channel) transition DIRECTLY to `status='failed'`. The reaper (Plan 22c.3-06) handles stuck `'forwarded'` rows. Test `test_no_auto_retry_on_failure` proves attempts stays at 1 on failure.
- **Persist-before-action wired:** `mark_forwarded` is committed BEFORE the bot HTTP call (D-28). On any error path, the next write is `mark_failed` — never a re-queue back to `pending`. On success, `mark_done` + `insert_agent_event` share a single transaction so the outbox pump never sees a state mismatch.
- **InappRecipeIndex shipped:** Lazy LRU cache of `recipes/*.yaml` `channels.inapp` blocks keyed by `recipe_name`; mtime-based invalidation re-reads on file touch. 60s container-IP cache via `docker.from_env().containers.get(...).attrs['NetworkSettings']['Networks'][network_name]['IPAddress']` per RESEARCH §Don't Hand-Roll table. ## Adapter contracts (verbatim from RESEARCH §Pitfall 6)

```python
match inapp.contract:
    case "openai_compat":
        # Used by hermes / nanobot / openclaw. POST {url}{endpoint}
        # body={model: contract_model_name or "agent",
        #       messages:[{role:user,content:row.content}]}
        # parse: data["choices"][0]["message"]["content"]
        # auth_mode='bearer' uses agent_containers.inapp_auth_token

    case "a2a_jsonrpc":
        # Used by nullclaw native A2A. POST {url}/a2a (endpoint='/a2a')
        # body={jsonrpc:"2.0", id:str(row.id), method:"message/send",
        #       params:{message:{role:user,parts:[{kind:text,text:row.content}],
        #                        messageId:str(row.id)}}}
        # parse: data["result"]["artifacts"][0]["parts"][0]["text"]
        # error path: if "error" in data → RuntimeError(f"a2a_error:{msg}")

    case "zeroclaw_native":
        # Used by zeroclaw native /webhook. POST {url}/webhook
        # body={message: row.content}
        # headers: X-Idempotency-Key (when row.idempotency_key present),
        #          X-Session-Id = f"inapp:{user_id}:{agent_id}"
        # parse: data["response"]

    case other:
        raise RuntimeError(f"unknown_contract:{other}")
```

## Decisions Made

- **`_dispatch_http_localhost(timeout_seconds=...)` is a parameter, not a module constant.** The default is `BOT_TIMEOUT_SECONDS=600` per D-40, but `state.bot_timeout_seconds` overrides it. Tests pass `bot_timeout_seconds=10.0` (or 0.1 for the timeout test) without monkey-patching the module — the production lifespan can set it from a future Pydantic Setting.
- **`_truncate_error_type(value)` projects free-form `last_error` strings to the InappOutboundFailedPayload regex enum.** The dispatcher's `last_error` carries the full detail (`bot_5xx:500`, `unknown_contract:totally_made_up`); the agent_events payload's `error_type` field has a 9-value Pydantic regex pattern. The helper extracts the leading token and falls back to `internal` for unmapped strings.
- **`recipe_lacks_inapp_channel` (the dispatcher's enum) maps to `recipe_no_inapp_channel` (the Pydantic enum).** The dispatcher uses the more-natural English; the payload's regex pattern uses the shorter form. `_truncate_error_type` performs the projection — keeps both layers happy.
- **Per-row work runs in its own connection, NOT inside the loop's fetch transaction.** Holding the SELECT-FOR-UPDATE transaction across the 600s bot call would defeat SKIP LOCKED (other replicas would block, not skip). Trade-off: between fetch-tx commit and `mark_forwarded` commit there's a millisecond window where two replicas could in principle both fetch the same row. Mitigation: `mark_forwarded` is idempotent at the SQL level (no-op when row is already `forwarded`) and PG's READ COMMITTED isolation makes the second UPDATE wait for the first.
- **`_FakeDispatcherState` test harness substitutes for Plan 22c.3-09's lifespan-built state.** Duck-typed — only the 4 attributes the dispatcher reads (`db`, `recipe_index`, `bot_http_client`, `bot_timeout_seconds`). Plan 09 will replace this with the real `app.state` shape; the dispatcher itself doesn't need to change.
