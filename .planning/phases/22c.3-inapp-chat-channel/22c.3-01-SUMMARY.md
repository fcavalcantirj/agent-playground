## Deviations from Plan

None at the rule-deviation level — plan executed as written. All three tasks completed in their planned order with all verify gates green.

Honest contextual observations (not deviations):

- **Hermes bootstrap detail:** the plan's common probe template did not include hermes's required `/opt/data/config.yaml` (`model.default`/`model.provider`/`model.base_url`) bootstrap. The first naive boot (env-only API_SERVER_* settings) returned HTTP 200 with `choices[0].message.content` containing the upstream "No models provided" error. Adding the model config to the entrypoint script (3 lines) made the LLM call succeed. This is a normal recipe-bootstrap detail, not a deviation; Plan 22c.3-10 (hermes recipe modification) absorbs it as part of the `channels.inapp.persistent_argv_override`. Recorded in the hermes spike file's "Anomalies observed" section.
- **Verify regex precision tweak on nullclaw spike:** the plan's Task 2 verify command tests `! grep -q "apk add python3\|nohup python3\|INAPP_BOT_BIN"` against the file content. The first draft of the v3 spike supersession banner mentioned those tokens by name to document what was dropped. The verify regex doesn't differentiate "this content is now dropped" from "this content is the active path" — it just wants the file completely free of those tokens (a conservative belt-and-suspenders against accidental v2 reintroduction). Reworded the banner + checklist line to convey supersession without using those exact tokens. The banner still names the v2 pattern in the abstract ("runtime package install, background-process bot binary indirection, env-var override") which preserves the history without arming the regex. No semantic change; verify gate now PASSES.
