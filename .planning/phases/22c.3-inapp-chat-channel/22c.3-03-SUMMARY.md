## Decisions Made

1. **redis_url default = compose hostname (`redis://redis:6379/0`).** Symmetric with DATABASE_URL pointing at `postgres:5432`. The Pydantic Settings field absorbs the host-vs-container choice — host-venv tests opt in by setting `AP_REDIS_URL=redis://localhost:6379/0` (relying on the local override's 127.0.0.1:6379 binding).

2. **Edit BOTH `pyproject.toml` AND `tools/Dockerfile.api` for the dep add.** Phase 22c-09 already paid the lesson: the Dockerfile build stage pins each runtime dep verbatim in `pip install --prefix=/install`, separate from the pyproject. Skipping the Dockerfile would have left sse-starlette + redis missing from the rebuilt image and broken every downstream plan's `import redis.asyncio`.

3. **`--protected-mode no` (NOT `yes` as the plan's first draft specified).** Rule 1 fix inline (see Deviations). The bridge network IS the security boundary (D-08); the per-process Redis flag was originally designed for the case where Redis binds 0.0.0.0 with no auth on a hostile public network, which doesn't match this deploy.

4. **`depends_on.redis.condition: service_healthy` (not `service_started`).** Plan 22c.3-09's lifespan will instantiate a `redis.asyncio` client that pings on startup; `service_started` would race the redis healthcheck and intermittently fail boot. `service_healthy` matches postgres's existing wiring.

5. **Local override exposes redis on 127.0.0.1:6379 only — never 0.0.0.0.** Host-venv tests + manual `redis-cli` debugging against `localhost:6379` work; nothing on the LAN can reach it. Same shape as the api_server local-override port (also 127.0.0.1:8000).

## Deviations from Plan

### Rule 1 — bug fixed inline

**1. [Rule 1 - Bug] `--protected-mode yes` silently RST-closed api_server's connections from the bridge IP.**

- **Found during:** Task 3 verification — the plan-prescribed first ping from inside api_server returned `redis.exceptions.ConnectionError: Error while reading from redis:6379 : (104, 'Connection reset by peer')`.
- **Issue:** Redis 7's `--protected-mode yes` (the plan's first-pass value) is designed for the case where Redis binds 0.0.0.0 with no auth on a hostile public network — it silently RST-closes any non-loopback client connection. The api_server container connects to redis from its bridge-net IP (172.x.0.x), not loopback, so protected-mode rejected legitimate traffic. The redis-cli ping inside the redis container itself worked because it uses local socket / loopback, hence the healthcheck passed but real clients didn't.
- **Root cause:** Architectural mismatch — protected-mode assumes loopback-only is correct, but the plan's own design comment (lines 244-246) acknowledged that `--bind 127.0.0.1` would prevent api_server from reaching redis. protected-mode makes the same loopback-only assumption at the redis-config layer and conflicts with the bridge-network architecture.
- **Fix:** `command: ["redis-server", "--protected-mode", "no"]`. Justification documented in the prod compose comment: the security boundary is the bridge network itself (D-08 — no `ports:` block in prod compose), not the per-process Redis flag. The local override binds 127.0.0.1 host-loopback only, so still no public exposure either way.
- **Files modified:** `deploy/docker-compose.prod.yml` (the redis `command:` entry + a multi-line comment explaining the tradeoff)
- **Commit:** `e57cac7` (combined with the Task 3 GREEN compose changes since the fix unblocks Task 3's verification gate)

### Other deviations

_None._ The plan's `<read_first>` references (PATTERNS.md MODIFY blocks, RESEARCH §Pinned versions, the existing `oauth_state_secret_fail_loud.py` test analog) all matched verbatim with what shipped. The Action block's exact append-shapes for pyproject.toml + Dockerfile.api + config.py + docker-compose.{prod,local}.yml were used as-is, modulo the protected-mode flip in Task 3.

## Threat Flags

_None._ This plan adds a new stateful service (redis) that lives entirely inside the compose bridge network with no public exposure (D-08 honored — no `ports:` in prod compose; only the local-only override binds 127.0.0.1). No new authentication paths, no new file access, no schema changes at trust boundaries. The `--protected-mode no` choice is documented inline in the compose file with the security-boundary justification; it does not introduce new public attack surface because no port is published to the host or LAN in prod.
