## Recipe-loader compatibility verdict

**No change needed.** `services/recipes_loader.py::load_all_recipes` is permissive: it parses each YAML file via a fresh ruamel `YAML()` instance, validates only that each recipe has a `name` field, and stores the entire dict in `app.state.recipes`. The schema-projection helper `to_summary` reads `channels.*` defensively (`channels_supported = list(channels_raw.keys()) if isinstance(channels_raw, dict) else []`) — so adding a new sibling key under `channels` neither requires a loader change nor breaks the existing telegram surface. Live api_server reload (`docker compose restart api_server`) confirmed this empirically: no `ValueError`, no exception, application startup complete, `(healthy)` within 16s.

## Decisions Made

- **ready_log_regex chosen as the IP/port-bearing form** (not the looser substring): the plan's must_haves.truth #3 mandates this exact form; the production env vars set `API_SERVER_HOST=127.0.0.1` so the bot will bind loopback (matching the regex) even though the spike (which used `0.0.0.0` for host-port mapping reachability) emitted a different bind address. The recipe's regex must match the production-env line, which is the 127.0.0.1 form.
- **`inapp:` block placed BEFORE the trailing `# Discord, Slack, WhatsApp, …` deferred-channels comment**, NOT after it. Reasoning: the deferred-channels comment lists candidates for future channels that come AFTER what's currently empirically validated. inapp IS empirically validated (Wave 0), so it sits with telegram on the validated side of that boundary.
- **auth_mode field NOT set on YAML** (defaults to 'none' on InappRecipeIndex). The dispatcher's openai_compat adapter constructs the Bearer header from `auth_header` template + `container.inapp_auth_token`, regardless of `auth_mode`. The field is informational for v1.
