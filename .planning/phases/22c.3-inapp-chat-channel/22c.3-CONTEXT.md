# Phase 22c.3 — In-App Chat Channel - Context

**Gathered:** 2026-04-29
**Status:** Ready for research / planning

<domain>
## Phase Boundary

Add a new `inapp` channel to the platform so logged-in users can send messages to a running agent and receive replies via the API — instead of going through Telegram. Foundation for the Flutter native app (Phase 23) and the planned web `/dashboard/agents/:id` chat surface.

**In scope:**
- Recipe schema gains `channels.inapp` block (additive with `channels.telegram`)
- Per-recipe inbound IPC declaration (transport + endpoint contract)
- Server-side: `POST /v1/agents/:id/messages` (inbound) + `GET /v1/agents/:id/messages/stream` (SSE outbound)
- Outbound fan-out via Redis Pub/Sub — new infrastructure dependency
- Reuse `agent_events` table with new `kind` values: `inapp_inbound`, `inapp_outbound`
- Auth via existing `ap_session` cookie + `require_user`
- Empirical 5-recipe matrix verification (each recipe's bot can speak the inapp channel)

**Out of scope (deferred to future phases):**
- Temporal substrate refactor (Phase 24 candidate)
- Cross-host / multi-replica scaling story
- Mobile UI (Phase 23 — Flutter native app)
- Web chat UI (different phase — Phase 23.x or 22c.3-web)
- Per-message billing / token metering
- LLM-judge verdicts on chat replies
- Voice / image / file attachments

</domain>

<decisions>
## Implementation Decisions

### Stream protocol

- **D-01:** SSE (Server-Sent Events) for the outbound stream — mirrors MSV exactly. New endpoint `GET /v1/agents/:id/messages/stream` returns `Content-Type: text/event-stream`. Heartbeat every 30s. Reuses the same Phase 22b `event_poll_signals` + `agent_events` plumbing internally.
- **D-02:** Long-poll endpoint `GET /v1/agents/:id/events` (Phase 22b-05) stays as-is for backward compatibility. New SSE endpoint is a wrapper, not a replacement.

### Inbound IPC contract

- **D-03:** **Recipe declares HOW; executor implements.** Each recipe's `channels.inapp` block specifies:
  ```yaml
  channels:
    inapp:
      transport: http_localhost   # initial verb; future verbs may add stdin / docker_exec
      port: 9999
      contract: openai_chat_completions   # OR custom — recipe-author's choice
      endpoint: /v1/chat/completions
      ready_log_regex: "..."             # like channels.telegram has
  ```
- **D-04:** **HTTP-to-localhost adapter is the only validated transport for v1.** Bot exposes HTTP inside its container; runner POSTs to it via `deploy_default` Docker network (container IP discovered from `agent_containers.container_id`). MSV-production-validated.
- **D-05:** **FIFO bind-mount is RULED OUT.** Empirically tested 2026-04-29: FIFO inodes don't propagate read/write semantics across Docker Desktop's macOS gRPC-FUSE/virtiofs boundary (works on Linux Docker daemon, fails on macOS dev). Strikes FIFO from the transport menu.
- **D-06:** **`docker exec` STDIN write is a fallback transport** (deferred — only if a recipe legitimately can't expose HTTP). Not in v1 scope.
- **D-07:** Inbound endpoint shape: `POST /v1/agents/:id/messages` with body `{content: string, idempotency_key?: string}`. Returns `{message_id: string, queued_at: timestamp}`. Reuses existing idempotency middleware.

### Outbound fan-out

- **D-08:** **Add Redis Pub/Sub to the deploy stack.** New service `redis:7-alpine` in `deploy/docker-compose.prod.yml` + `deploy/docker-compose.local.yml`. Bound to localhost only (no public exposure).
- **D-09:** Channel naming convention: `agent:inapp:<agent_id_uuid>`. One channel per agent. SSE handlers subscribe per-stream-connection.
- **D-10:** Multi-tab behavior: Redis Pub/Sub fans out to all subscribers; each browser tab opens its own SSE connection and gets its own copy of the event. Idempotent client-side handling required (use `agent_events.id` as message identity for client-side dedup).
- **D-11:** Redis is also reusable for future phases (rate limiting at scale, session cache, ephemeral state). Adding it now isn't a single-purpose dep — aligns with the CLAUDE.md tech stack which already lists Redis as a v1 dep.

### Persistence

- **D-12:** **Reuse `agent_events` table** — no new chat_messages table. New `kind` values:
  - `inapp_inbound` — payload `{content: string, source: "user", from_user_id: uuid}`
  - `inapp_outbound` — payload `{content: string, source: "agent"}`
- **D-13:** Update `agent_events.kind` CHECK constraint via Alembic migration (add the two new kinds; pre-existing kinds untouched).
- **D-14:** History endpoint: extend existing `GET /v1/agents/:id/events` query to filter by `kind` (`?kinds=inapp_inbound,inapp_outbound&limit=20&before_seq=N`). No separate `/messages` endpoint — events ARE the messages.

### Workflow orchestration

- **D-15:** **Defer Temporal — use direct asyncio.** asyncio task in the inbound handler:
  1. Validate + persist inbound to `agent_events`
  2. Discover bot's container IP/port
  3. POST to bot's HTTP endpoint (with retry-with-exponential-backoff on connection errors; 3 retries, 30s total)
  4. On reply: persist outbound to `agent_events`, PUBLISH to Redis channel `agent:inapp:<agent_id>`
  5. Return success to client
- **D-16:** **Robustness without Temporal:** the persistence-before-action discipline (write inbound to DB BEFORE calling bot; write outbound to DB BEFORE PUBLISH) gives 90th-percentile reliability. If api_server crashes mid-handler, on restart the inbound is in DB and a `kind=inapp_pending` reaper job (separate from this phase) can re-attempt. Mark this as future-work in deferred ideas.
- **D-17:** When Temporal eventually lands (Phase 24 candidate), this handler migrates to a SendInappMessageWorkflow without changing the API contract.

### Auth

- **D-18:** SSE endpoint requires `ap_session` cookie + `require_user` ownership check (improves over MSV which has zero auth on its SSE stream).
- **D-19:** Inbound POST endpoint requires `ap_session` cookie + ownership check via `fetch_agent_instance` (mirrors existing pattern in `routes/agent_lifecycle.py`).

### Recipe schema

- **D-20:** `channels.inapp` is **additive** with `channels.telegram` — a recipe can declare both. `POST /v1/agents/:id/start` accepts `channel: "inapp"` OR `channel: "telegram"` as separate channel choices. Multi-channel-on-one-agent (run telegram AND inapp simultaneously on the same container) is a v2 stretch — phase 22c.3 is single-channel-per-agent-instance.
- **D-21:** Recipe author opts a recipe in by adding the `channels.inapp` block. Recipes without it cannot be deployed in inapp mode. Each of the 5 existing recipes (hermes, picoclaw, openclaw, nullclaw, nanobot) needs a per-recipe assessment + `channels.inapp` block + bot-side HTTP server. This is the bulk of the phase work.

### Dumb-pipe contract (architectural keystone)

- **D-22:** **api_server passes the user's message through to the bot UNMODIFIED.** No history replay. No system-prompt injection at the API layer. No context manipulation. The bot is responsible for its own conversation memory (workspace files / internal config / per-agent persistent state). The api is a dumb pipe. This is the project-level "API is smart, intelligence in API" rule applied correctly: **intelligence about the agent's behavior lives in the agent**, not in the api. The api's smarts are about routing, persistence, auth, and fan-out — NOT about composing the bot's prompt. (User correction 2026-04-29 mid-discussion.)

### Concurrency / turn-taking

- **D-23:** **Pass-through; no api-side queue or in-flight lock.** Each POST /v1/agents/:id/messages forwards immediately to the bot's HTTP endpoint. If user fires 3 messages while bot is replying to message 1, all 3 hit the bot in order. The bot handles concurrent calls (queue, concat, reject) per its own logic. Aligns with D-22 dumb-pipe principle.

### Error event taxonomy

- **D-24:** New `agent_events.kind` values for chat error surfaces:
  - `inapp_outbound_failed` — payload `{error_type: "bot_5xx" | "bot_timeout" | "bot_empty" | "container_dead", message: string, retry_count: int}`
  - Existing `agent_error` kind STAYS for non-chat errors (boot failures, watchdog timeouts, generic agent crashes). Don't conflate.
  - SSE event name mirrors the kind: `event: inapp_outbound_failed\ndata: {...}`. Client-side error UI subscribes to this event type.

### SSE reconnection / replay

- **D-25:** **Standard SSE replay via `Last-Event-Id` header.** SSE handler reads the header on connect:
  - Absent → stream future events only (subscribe to `event_poll_signals` from now)
  - Present (e.g., `Last-Event-Id: 1234`) → replay `SELECT * FROM agent_events WHERE agent_container_id = $1 AND seq > 1234 ORDER BY seq` first, then attach to the live signal stream.
  - Each SSE event includes `id: <seq>` so the client knows what to send back on reconnect.
- **D-26:** Replay is bounded — if `Last-Event-Id` indicates more than 500 missed events, return them all but flag `event: replay_truncated` in the stream so the client knows to refetch via `GET /v1/agents/:id/events?kinds=inapp_outbound&limit=500&before_seq=...` for full backfill. Prevents memory blow-up on long-disconnected clients.

### Per-recipe assessment (research output)

The phase RESEARCH.md (next step after this CONTEXT) will empirically test each recipe to determine:
1. Does the bot already expose an HTTP endpoint? (hermes gateway-daemon may; others probably don't)
2. If not, what's the smallest patch to add one? (sidecar process inside the container, or wrap the bot's main loop in a small HTTP server)
3. What protocol does the bot honor? (OpenAI-compat preferred; recipe-author's call)

### Claude's Discretion

- Exact SSE event envelope shape (e.g., `event: inapp_outbound\ndata: {...}` vs `event: message\ndata: {...}`) — Claude picks during planning, document in PLAN.
- Exponential backoff parameters (base, max, jitter) — Claude picks reasonable defaults.
- Redis client library choice (`redis` vs `aioredis`) — pick whichever is currently maintained.
- The two-step "persist before publish" sequence — Claude lays out the exact transaction boundaries.

## Existing Code Insights

### Reusable assets (zero new infra needed)

- `agent_events` table (Phase 22b alembic) — already has the right shape (id, agent_container_id, seq, kind, payload, ts). Just adds two kinds.
- `app.state.event_poll_signals: dict[UUID, asyncio.Event]` — already in-process pub/sub. SSE handler subscribes to the same Event the watcher fires.
- `app.state.event_poll_locks: dict[UUID, asyncio.Lock]` — D-13 single-poll-per-agent. SSE may need a different lock policy (multiple SSE clients allowed) — flag for research.
- `routes/agent_events.py::events_long_poll()` — full reference for the agent-id ownership check + signal-wait dance. Copy the auth + DB scope pattern.
- `services/watcher_service.py::_get_poll_signal()` — pattern for lazy-init asyncio.Event under `locks_mutex`. Mirror it for any new locks.
- `routes/agent_lifecycle.py::start_persistent()` — channel config pattern (channel_inputs body, channel_type column). New `channel: "inapp"` slots into the same dispatch.
- `crypto/age_cipher.py` — channel_config_enc encryption (Telegram bot tokens). Inapp channel has no secrets to encrypt (no third-party API key) — channel_config_enc stays NULL or stores `{}`.

### Established patterns

- **Persist-before-publish discipline** — every write to `agent_events` is committed BEFORE the watcher fires the signal. SSE handler does the same with Redis PUB.
- **Per-agent ownership check** — every endpoint that touches an agent_id calls `fetch_agent_instance(agent_id, user_id)` which filters by user_id at the SQL layer.
- **Pitfall 4 (DB pool exhaustion)** — never hold a DB connection across `await asyncio.wait_for(signal.wait(), timeout)`. Two-scope pattern.
- **Stripe-shape error envelopes** — `_err()` helper in routes/auth.py and routes/agent_events.py. New endpoints use the same.

### Integration points

- New POST endpoint registered in `main.py` next to existing route imports (routes/agent_messages.py, new file).
- New SSE endpoint same module.
- Alembic migration 007 (next sequential after 006).
- New Pydantic payload models in `models/events.py` (extend `EVENT_KINDS` tuple).
- Redis client added to `app.state.redis` in `main.py` lifespan; closed on shutdown.
- New env var `AP_REDIS_URL` (default `redis://redis:6379/0` in compose; `redis://localhost:6379/0` for host venv tests).

### Anti-patterns to avoid (from prior phases)

- Don't introduce ANONYMOUS_USER_ID-style placeholders. Every chat write requires a real user_id (Phase 22c-06 forcing function still applies).
- Don't write to `runs.detail` strings without `_redact_api_key` if any subprocess output flows in. Chat content is user-supplied — but BYOK keys could leak via prompt injection — pattern-match the `OPENROUTER_API_KEY=` shape in inbound chat content before persisting (defense in depth).
- Don't auto-clear inputs on smoke fail (the 22c.1 fix). Same UX rule applies if Flutter or web client retries a chat send.

### What MSV does that we explicitly DON'T (and why)

- **No Temporal:** MSV uses Temporal SendMessageWorkflow for retry + checkpoint. We use direct asyncio + retry-with-backoff. Defer until Phase 24.
- **No multi-host Redis:** MSV runs Redis cross-host for fan-out across api replicas. We run single api_server process; Redis Pub/Sub still works the same way (one publisher, multi-subscriber pattern), just on one host.
- **No Router service:** MSV has a Node.js Router that handles Telegram bot interface + provisioning. We don't run a separate Router; api_server handles both sides.
- **MSV's SSE has no auth:** zero auth on `/api/notifications/:user_id/stream`. We mount our SSE behind `ap_session` cookie + ownership check. Strict improvement.
