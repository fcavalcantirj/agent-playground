# Phase 22c.3 — In-App Chat Channel - Context

**Gathered:** 2026-04-29
**Status:** Ready for research / planning

<domain>
## Phase Boundary

Add a new `inapp` channel to the platform so logged-in users can send messages to a running agent and receive replies via the API — instead of going through Telegram. Foundation for the Flutter native app (Phase 23) and the planned web `/dashboard/agents/:id` chat surface.

**In scope:**
- Recipe schema gains `channels.inapp` block (additive with `channels.telegram`)
- Per-recipe inbound IPC declaration (transport + endpoint contract)
- Server-side: `POST /v1/agents/:id/messages` (inbound) + `GET /v1/agents/:id/messages/stream` (SSE outbound)
- Outbound fan-out via Redis Pub/Sub — new infrastructure dependency
- Reuse `agent_events` table with new `kind` values: `inapp_inbound`, `inapp_outbound`
- Auth via existing `ap_session` cookie + `require_user`
- Empirical 5-recipe matrix verification (each recipe's bot can speak the inapp channel)

**Out of scope (deferred to future phases):**
- Temporal substrate refactor (Phase 24 candidate)
- Cross-host / multi-replica scaling story
- Mobile UI (Phase 23 — Flutter native app)
- Web chat UI (different phase — Phase 23.x or 22c.3-web)
- Per-message billing / token metering
- LLM-judge verdicts on chat replies
- Voice / image / file attachments

<decisions>
## Implementation Decisions

### Stream protocol

- **D-01:** SSE (Server-Sent Events) for the outbound stream — mirrors MSV exactly. New endpoint `GET /v1/agents/:id/messages/stream` returns `Content-Type: text/event-stream`. Heartbeat every 30s. Reuses the same Phase 22b `event_poll_signals` + `agent_events` plumbing internally.
- **D-02:** Long-poll endpoint `GET /v1/agents/:id/events` (Phase 22b-05) stays as-is for backward compatibility. New SSE endpoint is a wrapper, not a replacement.

### Inbound IPC contract

- **D-03:** **Recipe declares HOW; executor implements.** Each recipe's `channels.inapp` block specifies:
  ```yaml
  channels:
    inapp:
      transport: http_localhost   # initial verb; future verbs may add stdin / docker_exec
      port: 9999
      contract: openai_chat_completions   # OR custom — recipe-author's choice
      endpoint: /v1/chat/completions
      ready_log_regex: "..."             # like channels.telegram has
  ```
- **D-04:** **HTTP-to-localhost adapter is the only validated transport for v1.** Bot exposes HTTP inside its container; runner POSTs to it via `deploy_default` Docker network (container IP discovered from `agent_containers.container_id`). MSV-production-validated.
- **D-05:** **FIFO bind-mount is RULED OUT.** Empirically tested 2026-04-29: FIFO inodes don't propagate read/write semantics across Docker Desktop's macOS gRPC-FUSE/virtiofs boundary (works on Linux Docker daemon, fails on macOS dev). Strikes FIFO from the transport menu.
- **D-06:** **`docker exec` STDIN write is a fallback transport** (deferred — only if a recipe legitimately can't expose HTTP). Not in v1 scope.
- **D-07:** Inbound endpoint shape: `POST /v1/agents/:id/messages` with body `{content: string, idempotency_key?: string}`. Returns `{message_id: string, queued_at: timestamp}`. Reuses existing idempotency middleware.

### Outbound fan-out

- **D-08:** **Add Redis Pub/Sub to the deploy stack.** New service `redis:7-alpine` in `deploy/docker-compose.prod.yml` + `deploy/docker-compose.local.yml`. Bound to localhost only (no public exposure).
- **D-09:** Channel naming convention: `agent:inapp:<agent_id_uuid>`. One channel per agent. SSE handlers subscribe per-stream-connection.
- **D-10:** Multi-tab behavior: Redis Pub/Sub fans out to all subscribers; each browser tab opens its own SSE connection and gets its own copy of the event. Idempotent client-side handling required (use `agent_events.id` as message identity for client-side dedup).
- **D-11:** Redis is also reusable for future phases (rate limiting at scale, session cache, ephemeral state). Adding it now isn't a single-purpose dep — aligns with the CLAUDE.md tech stack which already lists Redis as a v1 dep.

### Persistence

- **D-12:** **Reuse `agent_events` table** — no new chat_messages table. New `kind` values:
  - `inapp_inbound` — payload `{content: string, source: "user", from_user_id: uuid}`
  - `inapp_outbound` — payload `{content: string, source: "agent"}`
- **D-13:** Update `agent_events.kind` CHECK constraint via Alembic migration (add the two new kinds; pre-existing kinds untouched).
- **D-14:** History endpoint: extend existing `GET /v1/agents/:id/events` query to filter by `kind` (`?kinds=inapp_inbound,inapp_outbound&limit=20&before_seq=N`). No separate `/messages` endpoint — events ARE the messages.

### Workflow orchestration

- **D-15:** **Defer Temporal — use direct asyncio.** asyncio task in the inbound handler:
  1. Validate + persist inbound to `agent_events`
  2. Discover bot's container IP/port
  3. POST to bot's HTTP endpoint (with retry-with-exponential-backoff on connection errors; 3 retries, 30s total)
  4. On reply: persist outbound to `agent_events`, PUBLISH to Redis channel `agent:inapp:<agent_id>`
  5. Return success to client
- **D-16:** **Robustness without Temporal:** the persistence-before-action discipline (write inbound to DB BEFORE calling bot; write outbound to DB BEFORE PUBLISH) gives 90th-percentile reliability. If api_server crashes mid-handler, on restart the inbound is in DB and a `kind=inapp_pending` reaper job (separate from this phase) can re-attempt. Mark this as future-work in deferred ideas.
- **D-17:** When Temporal eventually lands (Phase 24 candidate), this handler migrates to a SendInappMessageWorkflow without changing the API contract.

### Auth

- **D-18:** SSE endpoint requires `ap_session` cookie + `require_user` ownership check (improves over MSV which has zero auth on its SSE stream).
- **D-19:** Inbound POST endpoint requires `ap_session` cookie + ownership check via `fetch_agent_instance` (mirrors existing pattern in `routes/agent_lifecycle.py`).

### Recipe schema

- **D-20:** `channels.inapp` is **additive** with `channels.telegram` — a recipe can declare both. `POST /v1/agents/:id/start` accepts `channel: "inapp"` OR `channel: "telegram"` as separate channel choices. Multi-channel-on-one-agent (run telegram AND inapp simultaneously on the same container) is a v2 stretch — phase 22c.3 is single-channel-per-agent-instance.
- **D-21:** Recipe author opts a recipe in by adding the `channels.inapp` block. Recipes without it cannot be deployed in inapp mode. Each of the 5 existing recipes (hermes, picoclaw, openclaw, nullclaw, nanobot) needs a per-recipe assessment + `channels.inapp` block + bot-side HTTP server. This is the bulk of the phase work.

### Dumb-pipe contract (architectural keystone)

- **D-22:** **api_server passes the user's message through to the bot UNMODIFIED.** No history replay. No system-prompt injection at the API layer. No context manipulation. The bot is responsible for its own conversation memory (workspace files / internal config / per-agent persistent state). The api is a dumb pipe. This is the project-level "API is smart, intelligence in API" rule applied correctly: **intelligence about the agent's behavior lives in the agent**, not in the api. The api's smarts are about routing, persistence, auth, and fan-out — NOT about composing the bot's prompt. (User correction 2026-04-29 mid-discussion.)

### Concurrency / turn-taking

- **D-23:** **Pass-through; no api-side queue or in-flight lock.** Each POST /v1/agents/:id/messages forwards immediately to the bot's HTTP endpoint. If user fires 3 messages while bot is replying to message 1, all 3 hit the bot in order. The bot handles concurrent calls (queue, concat, reject) per its own logic. Aligns with D-22 dumb-pipe principle.

### Error event taxonomy

- **D-24:** New `agent_events.kind` values for chat error surfaces:
  - `inapp_outbound_failed` — payload `{error_type: "bot_5xx" | "bot_timeout" | "bot_empty" | "container_dead", message: string, retry_count: int}`
  - Existing `agent_error` kind STAYS for non-chat errors (boot failures, watchdog timeouts, generic agent crashes). Don't conflate.
  - SSE event name mirrors the kind: `event: inapp_outbound_failed\ndata: {...}`. Client-side error UI subscribes to this event type.

### SSE reconnection / replay

- **D-25:** **Standard SSE replay via `Last-Event-Id` header.** SSE handler reads the header on connect:
  - Absent → stream future events only (subscribe to `event_poll_signals` from now)
  - Present (e.g., `Last-Event-Id: 1234`) → replay `SELECT * FROM agent_events WHERE agent_container_id = $1 AND seq > 1234 ORDER BY seq` first, then attach to the live signal stream.
  - Each SSE event includes `id: <seq>` so the client knows what to send back on reconnect.
- **D-26:** Replay is bounded — if `Last-Event-Id` indicates more than 500 missed events, return them all but flag `event: replay_truncated` in the stream so the client knows to refetch via `GET /v1/agents/:id/events?kinds=inapp_outbound&limit=500&before_seq=...` for full backfill. Prevents memory blow-up on long-disconnected clients.

### Durable message orchestration (no in-memory state machine)

User correction 2026-04-29: "no in memory, no mocks, stubs, robust now." D-15..D-17 originally cavaliered orchestration to direct asyncio with no DB-backed durability. Replaced with the following explicit Postgres-backed state machine. Even WITHOUT Temporal, every state transition is persisted before action.

- **D-27:** **New table `inapp_messages`** — durable per-message state machine. Migration 007 (next sequential).
  ```sql
  CREATE TABLE inapp_messages (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    agent_id        UUID NOT NULL REFERENCES agent_instances(id) ON DELETE CASCADE,
    user_id         UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    content         TEXT NOT NULL,
    status          TEXT NOT NULL DEFAULT 'pending'
                      CHECK (status IN ('pending','forwarded','done','failed')),
    attempts        INT  NOT NULL DEFAULT 0,
    last_error      TEXT,
    last_attempt_at TIMESTAMPTZ,
    bot_response    TEXT,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    completed_at    TIMESTAMPTZ
  );
  CREATE INDEX ix_inapp_messages_agent_status ON inapp_messages(agent_id, status);
  CREATE INDEX ix_inapp_messages_status_attempts ON inapp_messages(status, last_attempt_at)
    WHERE status IN ('pending','forwarded');  -- partial index for the reaper
  ```
- **D-28:** **State transitions persisted BEFORE action.** Every transition is committed to Postgres before the next step happens:
  1. `POST /v1/agents/:id/messages` → INSERT status='pending' → return 202 with `{message_id}` to client (within ~50ms — fast ack).
  2. **Background dispatcher** (asyncio task, NOT in the request handler): `SELECT id FROM inapp_messages WHERE status='pending' ORDER BY created_at FOR UPDATE SKIP LOCKED LIMIT 10` → UPDATE status='forwarded', attempts=attempts+1, last_attempt_at=NOW() → POST to bot's HTTP endpoint.
  3. On bot 200: UPDATE status='done', bot_response=<reply>, completed_at=NOW() + INSERT into `agent_events` (kind=inapp_outbound, published=false).
  4. On bot 5xx/timeout/empty: if attempts < 3, UPDATE status='pending' (will be re-picked by next dispatcher tick). If attempts == 3, UPDATE status='failed', last_error=<reason> + INSERT into `agent_events` (kind=inapp_outbound_failed, published=false).
  5. **api_server crash anywhere in this chain** → on restart, the row is in the right state for the dispatcher to resume (status='pending' → re-tries; status='forwarded' → reaper notices it's stuck and re-queues OR marks failed by attempts).
- **D-29:** **`POST /v1/agents/:id/messages` returns 202 Accepted, NOT 200 with reply.** The handler does NOT block on the bot. Client gets `{message_id, status:"pending"}` immediately and listens on the SSE stream for the eventual `inapp_outbound` (or `inapp_outbound_failed`) event. This is the correct dumb-pipe shape per D-22.

### Reaper / restart resilience (no in-memory work)

- **D-30:** **Background asyncio task in api_server** (`reaper_inapp_messages()`) started in the FastAPI lifespan, cancelled on shutdown. Tick every 15s:
  ```sql
  SELECT id FROM inapp_messages
  WHERE status='forwarded' AND last_attempt_at < NOW() - INTERVAL '60 seconds'
  FOR UPDATE SKIP LOCKED LIMIT 50;
  ```
  For each stuck row: if attempts < 3, set status='pending' (re-queue); else status='failed' + last_error='reaper_timeout' + emit inapp_outbound_failed event.
- **D-31:** **API restart resilience.** On lifespan startup, BEFORE the dispatcher pump starts: `UPDATE inapp_messages SET status='pending' WHERE status='forwarded' AND last_attempt_at < NOW() - INTERVAL '5 minutes'` — sweeps any rows that were mid-forward when api crashed. Pumping then resumes them.
- **D-32:** Reaper + dispatcher use `FOR UPDATE SKIP LOCKED` so multiple api_server replicas (Phase 23+) can run them concurrently without double-processing. Single-replica today; future-safe.

### Outbox pattern for Redis PUB (no fire-and-forget)

- **D-33:** **`agent_events.published` BOOLEAN DEFAULT FALSE column** added (Alembic migration 007 alongside the `inapp_messages` table). Every event INSERT starts with `published=false`.
- **D-34:** **Outbox pump** — separate asyncio task (also lifespan-managed) that runs:
  ```sql
  SELECT id, agent_container_id, payload FROM agent_events
  WHERE published=false ORDER BY seq LIMIT 100 FOR UPDATE SKIP LOCKED;
  ```
  For each: `redis.publish('agent:inapp:'||agent_id, payload)` — on success: UPDATE published=true. On Redis failure: roll back transaction; next tick retries. Tick interval: 100ms (sub-second event delivery to SSE clients in the happy path).
- **D-35:** Bounded retry: events that stay unpublished for > 1h are logged + marked `published=true` with `last_error='abandoned_after_1h'` to prevent runaway retries. Clients still recover those events via `Last-Event-Id` replay (D-25) which queries agent_events directly, bypassing Redis.

### Bot port discovery + readiness gate

- **D-36:** **Recipe declares static port** in `channels.inapp.port` — recipe author picks (e.g., 9999). The runner reads this at deploy time when starting the bot's container; no Docker port-mapping discovery needed.
- **D-37:** **`agent_containers.ready_at` is the readiness gate.** Phase 22b's watcher already sets `ready_at` when the bot's `ready_log_regex` matches. The dispatcher (D-28) joins on `agent_containers` and rejects forwarding if:
  - `container_status != 'running'`, OR
  - `ready_at IS NULL` (never reached ready), OR
  - `stopped_at IS NOT NULL` (already torn down).
- **D-38:** **Not-ready handling at POST /messages.** If user POSTs while container is starting/stopped/dead: return `503 Service Unavailable` with `Retry-After: 5`. The inapp_messages row is still inserted (status='pending') — the dispatcher will pick it up when the container becomes ready, OR the reaper will mark failed if it never does within ~3 retries.

### Idempotency

- **D-39:** **Reuse existing `IdempotencyMiddleware` + `idempotency_keys` table.** `POST /v1/agents/:id/messages` honors `Idempotency-Key: <uuid>` header just like `POST /v1/runs` does. Duplicate POSTs return the original `inapp_messages.id`. 24h TTL.

### Bot HTTP retry budget

- **D-40 (REVISED 2026-04-29):** **Per-attempt timeout 600s (10 minutes); max 1 attempt (no auto-retry); reaper threshold matches.** Agent inferences can legitimately take minutes — a 30s bot timeout would cut off real work. Auto-retry is the wrong UX here: if the bot fails or times out, surface the failure to the client immediately and let the user decide whether to send a new message. This honors D-22 dumb-pipe principle: api doesn't second-guess; bot's response (or lack thereof) is the answer.
  - On timeout/5xx/empty: UPDATE status='failed', last_error=<reason>, completed_at=NOW() + INSERT agent_events(kind=inapp_outbound_failed) with `error_type` per D-24.
  - Reaper threshold (D-30) updated: `last_attempt_at < NOW() - INTERVAL '11 minutes'` (10min bot timeout + 1min slack). The reaper exists for genuine stuck rows (api crashed mid-forward), not for normal long inferences.
  - Restart sweep (D-31) updated: `last_attempt_at < NOW() - INTERVAL '15 minutes'` (slightly looser to avoid sweeping inferences mid-flight when api restarts).
  - Client retries by sending a new POST /messages — application-level retry, not framework-level.

### Per-message size cap

- **D-41:** **No API-side size cap on inbound `content`.** Validate non-empty only (matches MSV `req.Content == ""` check). Persist as Postgres `TEXT` (effectively up to ~1GB per row, far beyond practical use). Same for `bot_response` and event payloads. Project-level rate limiting + per-agent throughput naturally bound abuse risk.

### Per-agent rate limit on POST /messages

- **D-42:** **4 messages per minute per (user_id, agent_id) pair.** Aggressive on purpose — chat is human-paced; 4/min covers normal conversation with headroom for retries; anything faster is bot/spam. Implementation: extend existing `RateLimitMiddleware` with a new bucket key derivation: `f"chat:{user_id}:{agent_id}"`. Returns 429 with `Retry-After` header on overflow. **NOT** the global 60/min default.

### Conversation history deletion (separate from agent deletion)

- **D-43:** **New endpoint `DELETE /v1/agents/:id/messages`** — clears chat history WITHOUT deleting the agent or its container. Transactionally:
  ```sql
  BEGIN;
    DELETE FROM inapp_messages WHERE agent_id = $1 AND user_id = $2;
    DELETE FROM agent_events
      WHERE agent_container_id IN (SELECT id FROM agent_containers WHERE agent_instance_id = $1)
        AND kind IN ('inapp_inbound', 'inapp_outbound', 'inapp_outbound_failed');
  COMMIT;
  ```
  Returns 204 No Content. Container keeps running. Bot's internal memory (workspace files / config) NOT touched — that's the bot's domain per D-22; users wanting "fresh start" can stop+restart the agent which gets a fresh container.
- **D-44:** Distinction enforced in CONTEXT/PLAN: `DELETE /v1/agents/:id` (Phase 22c.1) deletes the agent + cascades container + cascades messages via FK. `DELETE /v1/agents/:id/messages` (this phase) deletes ONLY messages and keeps the agent. Different endpoints, different intents.

### Client target

- **D-45:** **Flutter is the v1 client target.** The endpoints are HTTP+JSON+SSE — equivalent web client (browser EventSource + fetch) is trivially compatible and not blocked by Flutter work, but is OUT OF PHASE 22c.3 SCOPE. Web chat UI lives in a future "22c.X-web-chat" phase or as part of Phase 23.x.

### total_runs is NOT touched by chat

- **D-46:** Chat messages **do NOT** bump `agent_instances.total_runs` or `last_run_at`. `total_runs` counts smoke runs (POST /v1/runs) only. If the dashboard "847 messages" stat (per Solvr Labs screenshot) is needed, it computes via `SELECT COUNT(*) FROM inapp_messages WHERE agent_id = $1`. Different concept; separate column or derived view.

- **Pitfall 4 (DB pool exhaustion)** — never hold a DB connection across `await asyncio.wait_for(signal.wait(), timeout)`. Two-scope pattern.
- **Stripe-shape error envelopes** — `_err()` helper in routes/auth.py and routes/agent_events.py. New endpoints use the same.

### What MSV does that we explicitly DON'T (and why)

- **No Temporal:** MSV uses Temporal SendMessageWorkflow for retry + checkpoint. We use direct asyncio + retry-with-backoff. Defer until Phase 24.
- **No multi-host Redis:** MSV runs Redis cross-host for fan-out across api replicas. We run single api_server process; Redis Pub/Sub still works the same way (one publisher, multi-subscriber pattern), just on one host.
- **No Router service:** MSV has a Node.js Router that handles Telegram bot interface + provisioning. We don't run a separate Router; api_server handles both sides.
- **MSV's SSE has no auth:** zero auth on `/api/notifications/:user_id/stream`. We mount our SSE behind `ap_session` cookie + ownership check. Strict improvement.
