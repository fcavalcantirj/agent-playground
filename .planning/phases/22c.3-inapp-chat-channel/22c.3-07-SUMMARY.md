## Pitfall 3 strategy decision recap

RESEARCH §Pitfall 3 surfaced two strategies for handling Redis failure mid-batch:

**Strategy 1 — per-row commit:** open a transaction per row; UPDATE published=true after each successful publish; commit. Pros: never re-publishes a row that succeeded. Cons: doubles PG round-trips per batch (1 SELECT + N UPDATE + N COMMIT instead of 1 SELECT + 1 UPDATE + 1 COMMIT); requires either re-acquiring the row-lock per row OR holding the SKIP LOCKED lock across N commits (which loses the multi-replica disjoint-partition property). Doesn't actually prevent duplicates — Redis Pub/Sub is fire-and-forget; a subscriber that disconnects between two publishes loses the second one entirely; it does NOT see "this was already delivered."

**Strategy 2 — optimistic batch (CHOSEN):** open one outer transaction; iterate publishes; on any RedisError, return early so the outer transaction rolls back the would-be UPDATE; bulk UPDATE all published_ids on success. Pros: 1 SELECT + 1 UPDATE per batch; clean all-or-nothing semantics; SKIP LOCKED disjoint partition preserved. Cons: rows that DID publish before the mid-batch failure get re-published on the retry tick — handled at the SSE layer via D-10 idempotent client-side dedup by seq.

The trade-off is identical at the broker level (Redis is fire-and-forget either way) but Strategy 2 wins on PG load and lock semantics. Decision honored verbatim from RESEARCH §Pattern 3 (lines 658-731).

## Abandon-after-1h semantic

Per D-35 the pump's WHERE clause has `e.ts > NOW() - INTERVAL '1 hour'`. A row with `published=false` and `ts = NOW() - 2h` is excluded from the SELECT. Consequences:

1. The pump emits 0 for that row (and any other old row); the row is NOT marked terminal.
2. Redis NEVER receives a publish for the old row.
3. The row stays `published=false` forever from the pump's perspective.

The recovery path is **Plan 22c.3-08 SSE handler with `Last-Event-Id` replay**. SSE clients reconnect with `Last-Event-Id: 1234`; the handler queries `agent_events` directly (NOT Redis):

```sql
SELECT * FROM agent_events
WHERE agent_container_id = $1 AND seq > 1234
ORDER BY seq
```

This bypasses the outbox pump entirely. Old rows surface to the client via PG read, even though they never made it into Redis. The pump's job is the happy-path live fan-out; replay is the durability layer. Together they give at-least-once delivery (with client-side dedup by seq).

## Decisions Made

- **NOT changing the existing dispatcher/reaper to consume the outbox indirectly.** They INSERT agent_events with the column-default `published=false` and rely on the pump to fan out. This is the correct seam — dispatcher/reaper care about WHEN to insert (per-message state machine); pump cares about WHAT to do with the inserted rows (broker fan-out). Coupling them would re-introduce the fire-and-forget pattern the outbox exists to eliminate.
- **NOT adding a "drain remaining rows on shutdown" step.** Plan 09 lifespan will set the stop_event and await the task, which exits cleanly mid-tick or at the next idle wait. Any unpublished rows survive the restart (they're in PG); the next boot's pump picks them up. The whole point of the outbox is restart-resilience — adding a drain would make shutdown slower without adding any guarantee.
- **NOT decoupling `outbox_pump_loop` into a queue/worker model.** A single asyncio task per replica is sufficient. The work per tick is bounded (PUMP_BATCH_LIMIT=100 rows; redis.publish is sub-millisecond on local). Replicas scale via SKIP LOCKED — adding api_server replicas linearly scales pump throughput.
- **Subscriber-on-separate-client test idiom.** The redis_client fixture is configured for the publisher side; tests open a fresh `redis.asyncio.from_url` for the subscriber and run it as an asyncio.Task. Avoids the in-coroutine pub-sub deadlock and matches production where SSE consumers and the outbox pump live in distinct processes/connections.
- **Test bypass of insert_agent_event.** Tests INSERT directly into agent_events with a caller-supplied seq. Production goes through `event_store.insert_agent_event` (which acquires `pg_advisory_xact_lock` to allocate the next seq). The test bypass is a test-isolation choice — multiple test rows can have well-known seq values without racing the lock — NOT a production pattern.
