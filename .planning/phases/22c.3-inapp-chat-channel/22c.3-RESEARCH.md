## Summary

Phase 22c.3 adds an `inapp` channel transport that lets a logged-in user send a message and receive a reply via the API. The channel is composed of:

1. **Inbound dispatch** — `POST /v1/agents/:id/messages` writes a row to a new `inapp_messages` table (status=`pending`) and returns 202 immediately. A background asyncio dispatcher loop with `FOR UPDATE SKIP LOCKED` picks up the row, transitions it to `forwarded`, calls the bot, transitions it to `done` or `failed`.
2. **Outbound stream** — `GET /v1/agents/:id/messages/stream` is an SSE endpoint (sse-starlette `EventSourceResponse`) that supports `Last-Event-Id` replay against `agent_events`, then attaches to a Redis Pub/Sub subscriber for live fan-out. A 30s heartbeat (sse-starlette default 15s; we tune to 30s to match MSV) keeps middlebox connections alive.
3. **Outbox pump** — `agent_events.published BOOLEAN` column gates Redis publishes; a background task runs `SELECT … WHERE published=false FOR UPDATE SKIP LOCKED LIMIT 100` every 100ms. Crash resilience: events written to PG never lose; Pub/Sub failures don't abandon.
4. **Reaper** — `inapp_messages` rows stuck in `forwarded` past D-40's reaper threshold (11 minutes) are re-queued or marked `failed` based on `attempts`.

The phase's primary unresolved risk — D-21 per-recipe HTTP feasibility — was empirically validated via THREE rounds of docker spikes (2026-04-29 PM and 2026-04-30). All 5 recipes ship `transport: http_localhost` with NATIVE chat HTTP — zero sidecars, zero bridges, zero subprocess gymnastics:

- **3 recipes — OpenAI-compat `/v1/chat/completions`:**
  - **hermes** — `gateway.platforms.api_server` activated by `API_SERVER_ENABLED=true` + `API_SERVER_KEY` env vars; binds port 8642
  - **nanobot** — `nanobot serve` mode (alternative entrypoint to `nanobot gateway`); binds port 8900
  - **openclaw** — gateway binds `/v1/chat/completions` on port 18789 when `gateway.http.endpoints.chatCompletions.enabled = true` + `gateway.auth.token` set + bot runs `openclaw gateway run` (NOT `--allow-unconfigured`). MSV production pattern (`provision_ops.go:380-401`).
- **1 recipe — Google A2A JSON-RPC 2.0:**
  - **nullclaw** — `POST /a2a method=message/send` (sync) + `method=message/stream` (SSE streaming). Activation: `a2a.enabled=true` in config. Empirically PROVED 2026-04-30 with real LLM round-trip in ~3s.
- **1 recipe — native ZeroClaw `{message}→{response,model}`:**
  - **zeroclaw** (NEW, replaces picoclaw) — `POST /webhook` body `{"message":"..."}` returns `{"response":"<reply>","model":"<model>"}`. Built-in `X-Idempotency-Key` (replay returns `{"idempotent":true,"status":"duplicate"}`), built-in `X-Session-Id`, headless mode (`gateway.require-pairing=false`), token-streaming WebSocket at `/ws/chat`, Prometheus `/metrics`. Empirically PROVED 2026-04-30 with real LLM in ~3s. Image: `ghcr.io/zeroclaw-labs/zeroclaw:latest` (Rust, distroless, ~50 MB).

**picoclaw deferred** per user direction (2026-04-30) — leave for last. `recipes/picoclaw.yaml` stays in repo for backward compatibility with the smoke suite, but is OUT of Phase 22c.3 inapp scope.

This means the planner implements ONE transport verb (`http_localhost` via `httpx.AsyncClient` with 600s timeout per D-40) with THREE per-contract adapters (~50 LOC each):
- `contract: openai_compat` (hermes, nanobot, openclaw) — `{model, messages: [{role, content}]}` → `{choices:[{message:{content}}]}`
- `contract: a2a_jsonrpc` (nullclaw) — `{jsonrpc, method:"message/send", params:{message:{role,parts}}}` → `{result:{artifacts:[{parts:[{text}]}]}}`
- `contract: zeroclaw_native` (zeroclaw) — `{message}` → `{response, model}` (cleanest envelope of the 5)

**Primary recommendation:** Single-verb `http_localhost` dispatcher with per-recipe `contract:` declaration. Recipes own their activation (env-flag for hermes, alt entrypoint for nanobot, config-flag for openclaw, A2A enable for nullclaw, onboard --quick for zeroclaw). Dispatcher POSTs JSON, parses reply per the `contract` adapter, period. No sidecars, no docker_exec_cli, no subprocess fork.

## Architectural Responsibility Map

| Capability | Primary Tier | Secondary Tier | Rationale |
|------------|-------------|----------------|-----------|
| Inbound POST /messages handler | API / Backend | — | Auth, idempotency, persist-before-action all live in Postgres; D-29 fast-ack 202 keeps client UX snappy |
| Outbound SSE stream | API / Backend | Browser / Client | Client connects via `EventSource`; backend is the auth boundary + Redis subscriber; the client is dumb per project rules |
| Bot HTTP/exec dispatch | API / Backend | Bot Container | Backend's dispatcher loop owns the state machine; bot container honors the recipe-declared transport contract |
| Redis Pub/Sub fan-out | Cross-cutting (Redis service in deploy stack) | API / Backend | Service is independent; backend publishes via outbox + subscribes per SSE connection |
| Durable per-message state machine | Database / Storage | API / Backend | `inapp_messages` Postgres table is the source of truth; asyncio loops are stateless workers |
| Conversation memory & context | Bot Container | — | D-22: dumb pipe — bot owns its own memory (workspace files, internal config); api never composes prompts |

### Pattern 1: SSE Handler with Last-Event-Id Replay

**What:** SSE endpoint that replays missed events from Postgres before attaching to Redis live stream.
**When to use:** Every outbound stream — D-25 mandates Last-Event-Id support.

```python
# routes/agent_messages.py
from sse_starlette.sse import EventSourceResponse, ServerSentEvent
from fastapi import Request, Header
from uuid import UUID
import asyncio
import json
import logging

from ..auth.deps import require_user
from ..services.event_store import fetch_events_after_seq
from ..services.run_store import fetch_agent_instance

_log = logging.getLogger("api_server.agent_messages")
INAPP_KINDS = {"inapp_inbound", "inapp_outbound", "inapp_outbound_failed"}
REPLAY_HARD_CAP = 500   # D-26


async def messages_stream(
    request: Request,
    agent_id: UUID,
    last_event_id: str | None = Header(default=None, alias="Last-Event-Id"),
):
    """SSE outbound stream — D-01, D-25, D-26, D-09, D-18.

    1. require_user + fetch_agent_instance (ownership check at SQL layer).
    2. Resolve container_row_id (URL agent_id IS agent_containers.id per
       Spike B 2026-04-19, same convention as routes/agent_events.py:377).
    3. Replay agent_events seq > Last-Event-Id (capped at 500; D-26 emits
       replay_truncated event when over budget).
    4. Subscribe to redis channel agent:inapp:<agent_id>.
    5. Heartbeat every 30s via sse-starlette ping_message_factory.
    """
    # Auth + ownership ----------------------------------------------------
    sess = require_user(request)
    from fastapi.responses import JSONResponse
    if isinstance(sess, JSONResponse):
        return sess
    user_id: UUID = sess

    pool = request.app.state.db
    async with pool.acquire() as conn:
        agent = await fetch_agent_instance(conn, agent_id, user_id)
    if agent is None:
        return JSONResponse(status_code=404, content={"error": "agent not found"})

    # Parse Last-Event-Id (D-25: integer seq) -----------------------------
    since_seq = 0
    if last_event_id:
        try:
            since_seq = max(0, int(last_event_id))
        except ValueError:
            since_seq = 0  # malformed → start from now

    redis_client = request.app.state.redis  # populated by lifespan
    channel = f"agent:inapp:{agent_id}"

    async def event_generator():
        # Step 1 — replay missed events (D-25, capped at 500 per D-26)
        async with pool.acquire() as conn:
            rows = await fetch_events_after_seq(
                conn, agent_id, since_seq, INAPP_KINDS,
            )
        truncated = False
        if len(rows) > REPLAY_HARD_CAP:
            rows = rows[:REPLAY_HARD_CAP]
            truncated = True
        for r in rows:
            payload = r["payload"]
            if isinstance(payload, str):
                payload = json.loads(payload)
            yield ServerSentEvent(
                id=str(r["seq"]),
                event=r["kind"],
                data=json.dumps({
                    "seq": int(r["seq"]),
                    "kind": r["kind"],
                    "payload": payload,
                    "ts": r["ts"].isoformat(),
                }),
            )
        if truncated:
            yield ServerSentEvent(
                event="replay_truncated",
                data=json.dumps({"hint": "use GET /v1/agents/:id/events for full history"}),
            )

        # Step 2 — subscribe to Redis Pub/Sub for live events.
        # async with auto-cleans subscriber on disconnect (HIGH confidence;
        # see redis.readthedocs.io asyncio_examples).
        async with redis_client.pubsub() as pubsub:
            await pubsub.subscribe(channel)
            while True:
                # client disconnected? sse-starlette also enforces this
                # internally — defensive belt-and-braces here.
                if await request.is_disconnected():
                    return
                msg = await pubsub.get_message(
                    ignore_subscribe_messages=True,
                    timeout=1.0,   # 1s poll → responsive disconnect detection
                )
                if msg is None:
                    continue
                # msg["data"] is the JSON payload published by outbox_pump
                data = msg["data"]
                if isinstance(data, (bytes, bytearray)):
                    data = data.decode("utf-8")
                try:
                    parsed = json.loads(data)
                    yield ServerSentEvent(
                        id=str(parsed["seq"]),
                        event=parsed["kind"],
                        data=data,
                    )
                except Exception:
                    _log.exception("phase22c3.sse.malformed_redis_message")
                    continue

    # ping_message_factory emits a comment line every 30s — keeps middleboxes
    # from killing idle connections; sse-starlette docs §heartbeat.
    return EventSourceResponse(
        event_generator(),
        ping=30,
    )
```

### Pattern 2: Dispatcher Loop with FOR UPDATE SKIP LOCKED

**What:** Background asyncio task that pulls pending messages and forwards them to bots.
**When to use:** Single instance v1; multi-replica safe via SKIP LOCKED.

import asyncio
import json

from datetime import datetime, timezone

import asyncpg
import httpx

_log = logging.getLogger("api_server.inapp_dispatcher")

DISPATCHER_TICK_S = 0.0          # immediate next-tick on activity
DISPATCHER_IDLE_TICK_S = 0.5     # poll cadence when idle
BATCH_LIMIT = 10                 # rows per tick


async def dispatcher_loop(app_state, stop_event: asyncio.Event) -> None:
    """Pull pending → forwarded → done|failed. D-28.

    Loop tick policy: when the SELECT returns 0 rows, sleep
    DISPATCHER_IDLE_TICK_S; when it returns rows, immediately re-tick after
    handling. Avoids fixed-rate polling (battery on dev) while keeping
    busy-throughput high.

    Pitfall 4 — DB connection lifetime: the SELECT-and-mark-forwarded
    transaction releases its connection BEFORE the bot HTTP/exec call. The
    second commit (status=done|failed) acquires a fresh connection. Holding
    a connection across a 600s `await client.post()` would exhaust the pool
    under modest message-fanout. Mirror routes/agent_events.py:248-265
    Pitfall 4 discipline.
    """
    pool = app_state.db
    http_client: httpx.AsyncClient = app_state.bot_http_client

    while not stop_event.is_set():
        picked = await _pick_and_mark_forwarded(pool)
        if not picked:
            try:
                await asyncio.wait_for(stop_event.wait(), DISPATCHER_IDLE_TICK_S)
            except asyncio.TimeoutError:
                pass
            continue
        # Concurrent forwarding within the batch — each row's bot call is
        # independent. asyncio.gather lets a slow bot for one user not block
        # other users' messages. Still capped at BATCH_LIMIT per tick.
        await asyncio.gather(*[
            _forward_one(app_state, http_client, row, pool)
            for row in picked
        ], return_exceptions=False)
        # Yield to give the event loop a tick — gather already does, this is
        # a defensive re-yield in case BATCH_LIMIT grows in the future.
        await asyncio.sleep(0)


async def _pick_and_mark_forwarded(pool) -> list[asyncpg.Record]:
    """Atomic pick: SELECT FOR UPDATE SKIP LOCKED + UPDATE in one tx."""
    async with pool.acquire() as conn:
        async with conn.transaction():
            rows = await conn.fetch(
                """
                SELECT m.id, m.agent_id, m.user_id, m.content, m.attempts,
                       c.id AS container_row_id, c.container_id, c.container_status,
                       c.ready_at, c.stopped_at, c.recipe_name, c.channel_type,
                       c.inapp_auth_token  -- new column for http_localhost auth
                FROM inapp_messages m
                JOIN agent_containers c ON c.agent_instance_id = m.agent_id
                WHERE m.status = 'pending'
                ORDER BY m.created_at ASC
                FOR UPDATE OF m SKIP LOCKED
                LIMIT $1
                """,
                BATCH_LIMIT,
            )
            if not rows:
                return []
            ids = [r["id"] for r in rows]
            await conn.execute(
                """
                UPDATE inapp_messages
                SET status='forwarded',
                    attempts=attempts+1,
                    last_attempt_at=NOW()
                WHERE id = ANY($1::uuid[])
                """,
                ids,
            )
            return rows


async def _forward_one(app_state, http_client, row, pool) -> None:
    """Dispatch one row to its bot per the recipe's transport verb."""
    recipe = app_state.recipes.get(row["recipe_name"])
    if recipe is None:
        await _mark_failed(pool, row, "recipe_missing")
        return

    # D-37/D-38: readiness gate
    if (row["container_status"] != "running"
        or row["ready_at"] is None
        or row["stopped_at"] is not None):
        await _mark_failed(pool, row, "container_not_ready")
        return

    inapp_block = recipe.channels.get("inapp")
    if inapp_block is None:
        await _mark_failed(pool, row, "recipe_no_inapp_channel")
        return

    transport = inapp_block.transport
    if transport != "http_localhost":
        await _mark_failed(pool, row, f"unknown_transport:{transport}")
        return
    try:
        reply = await _dispatch_http_localhost(
            http_client, row, inapp_block,
            _resolve_container_ip(app_state, row["container_id"]),
        )
    except httpx.TimeoutException:
        await _mark_failed(pool, row, "bot_timeout")
        return
    except httpx.HTTPStatusError as e:
        await _mark_failed(pool, row, f"bot_5xx:{e.response.status_code}")
        return
    except Exception:
        _log.exception("phase22c3.dispatch.unexpected", extra={"id": str(row["id"])})
        await _mark_failed(pool, row, "internal")
        return

    if not reply or not reply.strip():
        await _mark_failed(pool, row, "bot_empty")
        return

    await _mark_done(pool, row, reply)


async def _dispatch_http_localhost(http_client, row, inapp_block, ip: str) -> str:
    url = f"http://{ip}:{inapp_block.port}{inapp_block.endpoint}"

    body = {
        "model": inapp_block.contract_model_name or "agent",

### Pattern 3: Outbox Pump (Postgres → Redis PUB)

**What:** Background task that publishes `agent_events` rows where `published=false` to Redis.
**When to use:** Always — D-33/D-34 mandate exactly-once-on-commit semantics.

```python
# services/outbox_pump.py
import asyncio
import json
import logging
from datetime import timedelta

import redis.asyncio as redis_async

_log = logging.getLogger("api_server.outbox_pump")

PUMP_TICK_S = 0.1                   # 100ms cadence — D-34
PUMP_IDLE_TICK_S = 0.5              # back off when no rows
PUMP_BATCH_LIMIT = 100              # D-34
ABANDON_AFTER = timedelta(hours=1)  # D-35


async def outbox_pump_loop(app_state, stop_event) -> None:
    pool = app_state.db
    redis_client: redis_async.Redis = app_state.redis

    while not stop_event.is_set():
        published_n = await _pump_once(pool, redis_client)
        if published_n == 0:
            try:
                await asyncio.wait_for(stop_event.wait(), PUMP_IDLE_TICK_S)
            except asyncio.TimeoutError:
                pass
        else:
            await asyncio.sleep(PUMP_TICK_S)


async def _pump_once(pool, redis_client) -> int:
    """One pass: pick up to 100 unpublished, publish to Redis, mark done."""
    async with pool.acquire() as conn:
        async with conn.transaction():
            rows = await conn.fetch(
                """
                SELECT e.id, e.seq, e.kind, e.payload, e.correlation_id,
                       e.ts, e.agent_container_id,
                       c.agent_instance_id
                FROM agent_events e
                JOIN agent_containers c ON c.id = e.agent_container_id
                WHERE e.published=false
                  AND e.created_at > NOW() - $1::interval
                ORDER BY e.id
                FOR UPDATE OF e SKIP LOCKED
                LIMIT $2
                """,
                ABANDON_AFTER, PUMP_BATCH_LIMIT,
            )
            if not rows:
                return 0
            published_ids = []
            for r in rows:
                payload = r["payload"]
                if isinstance(payload, str):
                    payload = json.loads(payload)
                channel = f"agent:inapp:{r['agent_instance_id']}"
                msg = json.dumps({
                    "seq": int(r["seq"]),
                    "kind": r["kind"],
                    "payload": payload,
                    "correlation_id": r["correlation_id"],
                    "ts": r["ts"].isoformat(),
                })
                try:
                    await redis_client.publish(channel, msg)
                    published_ids.append(r["id"])
                except redis_async.RedisError:
                    # Don't mark published — next tick retries. Transaction
                    # rolls back the would-be UPDATE because we never reach it.
                    _log.exception("phase22c3.outbox.redis_publish_failed")
                    return 0  # abort batch; let caller back off
            if published_ids:
                await conn.execute(
                    "UPDATE agent_events SET published=true WHERE id = ANY($1::bigint[])",
                    published_ids,
                )
            return len(published_ids)
```

### Pattern 4: Lifespan Wiring

**What:** Attach dispatcher + reaper + outbox tasks to FastAPI lifespan; cancel cleanly on shutdown.
**When to use:** Single source for all 3 background tasks — extends `main.py:58-187` lifespan.

```python
# main.py — extend the existing lifespan
import redis.asyncio as redis_async
import httpx

@asynccontextmanager
async def lifespan(app: FastAPI) -> AsyncIterator[None]:
    settings = app.state.settings
    app.state.db = await create_pool(settings.database_url)
    # ... existing init (recipes, locks, signals, watchers) ...

    # Phase 22c.3 — Redis client (single instance, async)
    app.state.redis = redis_async.from_url(
        settings.redis_url,                     # AP_REDIS_URL env, default redis://redis:6379/0
        decode_responses=False,                  # we push bytes, decode in handler
        max_connections=20,
    )

    try:
        await app.state.redis.ping()
    except Exception:
        _log.exception("phase22c3.redis.ping_failed_boot")
        raise  # boot failure — Redis is required

    # httpx.AsyncClient for bot http_localhost dispatch (single instance,
    # connection pool shared across messages)
    app.state.bot_http_client = httpx.AsyncClient(
        timeout=httpx.Timeout(600.0, connect=5.0),  # D-40
        limits=httpx.Limits(max_connections=50, max_keepalive_connections=20),
    )

    # Phase 22c.3 — restart sweep (D-31): mark stale 'forwarded' rows as
    # 'pending' so dispatcher re-picks; threshold 15min per D-40 revision.
    async with app.state.db.acquire() as _conn:
        swept = await _conn.execute(
            """
            UPDATE inapp_messages
            SET status='pending'
            WHERE status='forwarded'
              AND last_attempt_at < NOW() - INTERVAL '15 minutes'
            """
        )
        _log.info("phase22c3.restart_sweep", extra={"swept": swept})

    # Phase 22c.3 — start the three background tasks
    from .services.inapp_dispatcher import dispatcher_loop
    from .services.inapp_reaper import reaper_loop
    from .services.outbox_pump import outbox_pump_loop

    app.state.inapp_stop = asyncio.Event()
    app.state.inapp_tasks = [
        asyncio.create_task(dispatcher_loop(app.state, app.state.inapp_stop),
                            name="inapp_dispatcher"),
        asyncio.create_task(reaper_loop(app.state, app.state.inapp_stop),
                            name="inapp_reaper"),
        asyncio.create_task(outbox_pump_loop(app.state, app.state.inapp_stop),
                            name="inapp_outbox"),
    ]

    try:
        yield
    finally:
        # ... existing watcher drain ...

        # Phase 22c.3 — graceful cancel
        app.state.inapp_stop.set()
        if app.state.inapp_tasks:
            _, pending = await asyncio.wait(
                app.state.inapp_tasks, timeout=5.0,
            )
            for p in pending:
                p.cancel()
        try:
            await app.state.bot_http_client.aclose()
        except Exception:
            pass
        try:
            await app.state.redis.aclose()
        except Exception:
            pass
        await close_pool(app.state.db)
```

### Anti-Patterns to Avoid

- **Don't subscribe to Redis Pub/Sub inside the route handler with no `async with`.** Leaks subscribers on disconnect. Always `async with redis.pubsub() as pubsub:`. [VERIFIED: redis docs §asyncio_examples]
- **Don't hold an asyncpg connection across `await http_client.post()`.** Pitfall 4 already documented in `routes/agent_events.py:31-39` — same rule applies to dispatcher. Two distinct connections: one for SELECT-and-mark, another for UPDATE-and-event-INSERT.
- **Don't fire-and-forget `redis.publish()` after a DB INSERT.** That's the classic broken outbox; if the API crashes between INSERT and publish, the event is lost from Pub/Sub forever. Use `published=false` + outbox pump (D-33/D-34).
- **Don't build a custom `ServerSentEvent` formatter when sse-starlette ships one.** Hand-rolling `data: …\n\n` strings drops `id:`, `event:`, `retry:` field handling and reinvents reconnect semantics. [VERIFIED: sse-starlette README] [DON'T HAND-ROLL]
- **Don't use redis Pub/Sub for at-least-once delivery.** Pub/Sub is fire-and-forget; D-25 Last-Event-Id replay against `agent_events` is the durability layer. If a subscriber misses a publish, replay handles it.
- **Don't mutate `app.state.recipes[name].channels.inapp` at runtime.** Recipes are loaded once at boot (`load_all_recipes(settings.recipes_dir)` at `main.py:80`); the dispatcher reads them as immutable.

## Common Pitfalls

### Pitfall 1: SSE `Last-Event-Id` race (event arrives between query end + signal subscribe)

**What goes wrong:** Client reconnects with `Last-Event-Id: 1234`. Generator queries `seq > 1234` from PG, returns rows up to seq 1240. Between the SELECT completing and the Redis `pubsub.subscribe()` call, the dispatcher persists seq 1241 + outbox publishes to Redis. Subscriber attaches AFTER the publish. Client never sees seq 1241.

**Why it happens:** Redis Pub/Sub is "best effort while subscribed" — messages published before subscription are lost.

**How to avoid:** After the replay query, BEFORE the subscribe, capture `current_max_seq = SELECT MAX(seq) FROM agent_events WHERE …`. After subscribing, run a DIFFERENTIAL second replay: `SELECT … WHERE seq > replay_max AND seq <= current_max`. Then start consuming live messages. Mirrors the watcher_service's "clear-then-fetch" discipline at `routes/agent_events.py:239-247` (Step 3b comment).

**Warning signs:** Integration test that sends a flurry of inapp messages during a client reconnect and observes a missing seq.

### Pitfall 2: asyncpg pool exhaustion on long bot HTTP forward (Pitfall 4 redux)

**What goes wrong:** Dispatcher acquires a connection, runs `SELECT … FOR UPDATE`, marks rows forwarded, then awaits `await http_client.post(url, timeout=600)`. Connection held the entire 600s. Pool size = 20; 25 simultaneously-running messages = 5 dispatcher tasks waiting on a free connection. Throughput stalls.

**Why it happens:** Same as `routes/agent_events.py:31-39` Pitfall 4 — DB pool is bounded; awaits on non-DB I/O must release the connection.

**How to avoid:** Two-scope pattern: scope 1 (`async with pool.acquire()`) does SELECT-and-mark-forwarded inside one transaction, then releases. Bot HTTP/exec call happens with NO connection held. Scope 2 acquires a fresh connection for UPDATE-status + INSERT-event. The dispatcher pattern in §Pattern 2 follows this discipline.

**Warning signs:** Test with `max_connections=2` + 5 in-flight messages — should still complete; if it deadlocks, Pitfall 2 fired.

### Pitfall 3: Outbox double-publish on transaction retry

**What goes wrong:** Outbox pump SELECTs 100 rows, publishes 50 to Redis, then a `RedisConnectionError` fires on row 51. Code path catches and aborts. UPDATE was never run for the 50 already-published rows. Next tick re-publishes them. Subscribers see duplicates.

**Why it happens:** No transaction encloses the publish + update.

**How to avoid:** Two strategies — pick one:

1. **Per-row tx (slow but bulletproof):** `for row in rows: BEGIN; redis.publish(); UPDATE; COMMIT;` — 100 round-trips per tick.
2. **Optimistic batch (faster, accepts at-most-N-duplicate-on-failure):** outer transaction holds `FOR UPDATE`. Iterate publishes in a list. On any Redis error, abort batch (no UPDATE runs; tx rolls back; row stays locked-then-released-then-retryable). On success, run UPDATE on the published_ids list as ONE statement, then COMMIT.

§Pattern 3 implements (2) — accepts that a Redis failure mid-batch retries the WHOLE batch, possibly publishing a few rows twice. SSE clients dedupe by `agent_events.id` (D-10). Strict choice for D-33 single-instance simplicity; revisit if multi-replica + multi-Redis-broker.

**Warning signs:** Subscribers receive same `id:` twice — confirm dedup path on client side.

### Pitfall 4: Redis Pub/Sub silent loss on broker bounce

**What goes wrong:** Redis container restarts. Existing pubsub subscribers' connection drops. Messages published during the gap are lost. Subscriber's `get_message()` returns None forever (or raises).

**Why it happens:** Pub/Sub has no persistence; if no subscriber is listening, messages evaporate. Reconnect doesn't replay.

**How to avoid:** SSE generator catches `redis.RedisConnectionError` / `redis.RedisError` in the `async for` loop, RE-SUBSCRIBES, then runs an "I might have missed something" replay against `agent_events` — `SELECT WHERE seq > last_seen_seq` (the last seq we yielded to the client). Continues. Mirrors D-25's Last-Event-Id pattern but triggered by detected reconnect rather than HTTP reconnect.

**Warning signs:** Integration test that bounces Redis with `docker restart redis-1` while a subscriber is active — subscriber should recover within ~5s with no missed events.

### Pitfall 5: Bot HTTP call hangs without timeout

**What goes wrong:** httpx default timeout is 5s; without override, a slow bot causes 5s timeout but D-40 mandates 600s. ALSO: if you set `timeout=600` only at request time, connect-timeout might still default to 5s — fine for our case, but mention.

**Why it happens:** httpx has separate connect / read / write / pool timeouts.

**How to avoid:** Use `httpx.Timeout(600.0, connect=5.0)` — fast-fail on container DNS but allow long inferences. Set on `app.state.bot_http_client` at lifespan (Pattern 4). Don't override per-request.

**Warning signs:** A bot that takes 6 minutes to reply gets cut off at 5 minutes — request-level timeout is interfering with client default.

### Pitfall 6: Per-contract adapter mis-parse of bot reply path

**What goes wrong:** With three contract envelopes (openai_compat, a2a_jsonrpc, zeroclaw_native), each adapter must parse the reply text from a DIFFERENT JSON path:
- openai_compat: `body["choices"][0]["message"]["content"]`
- a2a_jsonrpc: `body["result"]["artifacts"][0]["parts"][0]["text"]`
- zeroclaw_native: `body["response"]`

If a recipe declares the wrong `contract:` field OR the dispatcher's adapter dispatch table has the wrong recipe→adapter mapping, the dispatcher will get a `KeyError` or read the wrong field, and `inapp_messages.bot_response` will be empty / malformed.

**Why it happens:** Three envelopes is one more than copy-paste-friendly. Easy to mismap "nullclaw uses A2A" in one place and "nullclaw is openai_compat" in another.

**How to avoid:** Dispatcher's adapter dispatch is a single switch on `recipe.channels.inapp.contract:` with explicit unknown→reject:

```python
match inapp.contract:
    case "openai_compat":
        reply = body["choices"][0]["message"]["content"]
    case "a2a_jsonrpc":
        reply = body["result"]["artifacts"][0]["parts"][0]["text"]
    case "zeroclaw_native":
        reply = body["response"]
    case other:
        await _mark_failed(pool, row, f"unknown_contract:{other}")
        return
```

Plus per-adapter unit tests with a fake bot HTTP server (respx) that emits each envelope shape — verifies parsing extracts the right reply text. **Warning signs:** `inapp_messages.bot_response` is null/empty for one specific recipe while others succeed; KeyError exceptions in dispatcher logs name the recipe.

**Warning signs:** Bridge logs show subprocess exited 0 but `len(stdout) == 0`; `_mark_failed(bot_empty)` events with attempts=1.

### Pitfall 7: Rate-limit bucket key collision with /v1/runs

**What goes wrong:** D-42 says 4/min for chat. If `RateLimitMiddleware` derives bucket key as `f"{user_id}:{path}"`, the chat endpoint and /v1/runs share buckets and both 4/min.

**Why it happens:** Default middleware bucketing.

**How to avoid:** Add a path predicate in `RateLimitMiddleware` — when path matches `/v1/agents/{}/messages`, bucket key is `f"chat:{user_id}:{agent_id}"` with a 4/min limit; everything else uses the existing 60/min bucket. Explicit dispatch table inside the middleware. Does NOT require D-42 to be changed.

**Warning signs:** /v1/runs starts hitting 429 after 4 calls/min during E2E tests.

### Pitfall 8: SSE client + Cloudflare/middlebox 60s idle disconnect

**What goes wrong:** sse-starlette's default ping is 15s. We chose 30s to mirror MSV. Some middleboxes (Cloudflare's 100s default, nginx's 60s default) will close connections that go silent longer than their idle timeout. 30s is safely under 60s.

**Why it happens:** Reverse proxies don't know SSE; an idle TCP socket is fair game for cleanup.

**How to avoid:** `EventSourceResponse(generator, ping=30)`. Confirmed against MSV's 30s heartbeat (CONTEXT canonical_refs: notification_sse_handler.go).

**Warning signs:** Browser EventSource silently reconnects every 60s with `error` event, repeatedly.

### Known Threat Patterns for FastAPI + asyncpg + Redis

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| SQL injection via `kinds=` query param | Tampering | Existing V13 whitelist defense (parse against `VALID_KINDS` BEFORE binding). Reuse pattern in routes/agent_events.py:215-226 for the new DELETE/messages endpoint and any other inapp-kind filter |
| Cross-tenant chat access | Information Disclosure | `fetch_agent_instance(agent_id, user_id)` filters by user_id at SQL — defense-in-depth even if route fails to validate ownership |
| BYOK key leakage via prompt injection in chat content | Information Disclosure | Pre-INSERT scrubber: regex-match `(?:OPENROUTER_API_KEY|ANTHROPIC_API_KEY|sk-[a-zA-Z0-9]{40,})` in inbound `content`; if matched, reject with 400 INVALID_REQUEST. Mirrors the pattern in code_context anti-patterns. |
| Idempotency-Key collision across users | Spoofing | Existing IdempotencyMiddleware namespaces by user_id |
| SSE connection exhaustion DoS | DoS | Per-user concurrent-connection cap (Pitfall not covered in D-decisions — flag as plan-level concern; recommend `max=5` SSE per (user_id, agent_id)) |
| Redis published payload XSS reflected to client | XSS (web-only) | Client treats SSE `data:` field as JSON, not HTML. Flutter has no HTML interp. If web client emerges, escape on render. |
| Bot spoofing reply via injected control characters in stdout | Tampering | Pydantic `extra="forbid"` on InappOutboundPayload + `min_length=1` rejects empty replies; reply_extract_regex bounds the captured content; `bot_response` stored as plain TEXT (no eval) |

## Open Questions

1. **A5: Does `nanobot serve` require Authorization?**
   - What we know: Hermes does (`API_SERVER_KEY` env activates Bearer auth)
   - What's unclear: Nanobot's `nanobot serve` CLI has no auth flags surfaced in `--help`
   - Recommendation: 5-line Wave 0 spike — `curl -m 3 -X POST http://127.0.0.1:8900/v1/chat/completions -H 'Content-Type: application/json' -d '{"model":"x","messages":[{"role":"user","content":"hi"}]}'` against a running serve container. If 401 → bake auth proxy; if 200 (or 4xx for bad model rather than auth) → no auth.

2. **Per-user SSE concurrent-connection cap?**
   - What we know: D-10 says Redis Pub/Sub fans out to all subscribers. Multi-tab is supported and idempotent.
   - What's unclear: Is there a max number of SSE connections per (user, agent)?
   - Recommendation: Plan-level concern; document `max_sse_per_user_agent = 5` in the SSE handler; reject 6th with 429 + `Retry-After`.

3. **Dispatcher batching tradeoff at high message rate**
   - What we know: BATCH_LIMIT = 10, asyncio.gather across rows.
   - What's unclear: Does parallel forwarding of 10 messages overload the bot? (Bots are designed for one user — concurrent prompts may interfere.)
   - Recommendation: Per-(agent_id) serialization within a tick — group rows by agent_id, run sequentially per group, parallel across groups. Plan-level decision; flag.

4. **`agent_events.published` index strategy**
   - What we know: Partial index on `WHERE published=false` keeps the outbox SELECT fast.
   - What's unclear: At scale, the partial index will be small (rows transition quickly); but on a multi-day backlog (e.g., Redis broken), it grows.
   - Recommendation: Migration includes the partial index; monitor at deploy.

5. **Multi-replica safety of the dispatcher**
   - What we know: D-32 says `FOR UPDATE SKIP LOCKED` is multi-replica safe.
   - What's unclear: When >1 api_server replica runs simultaneously, the SSE subscriber's chosen replica isn't the same as the dispatcher's — but Redis Pub/Sub fans out across replicas. Verified semantics, untested at scale.
   - Recommendation: Plan-level note; v1 is single-replica per CONTEXT (deferred).
