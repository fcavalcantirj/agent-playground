## Contract-specific quirks surfaced during the run

### openai_compat (hermes / nanobot / openclaw)

- **hermes — auth_mode field absence (Rule 1 fix)**: recipe declared `auth_header` but not `auth_mode`. Parser defaults to `none`, so the dispatcher didn't send the bearer token and the bot returned 401. Added `auth_mode: bearer` to recipes/hermes.yaml.
- **nanobot — model id rejection**: bot only accepts the literal configured model id (returns 400 `Only configured model 'X' is available` on the placeholder `"agent"` value the dispatcher sends per `inapp.contract_model_name or "agent"`). Worked around in test fixture via per-test contract_model_name override.
- **openclaw — model id format**: only recognizes `anthropic/claude-haiku-4-5` (dash) not `anthropic/claude-haiku-4.5` (dot); verified against the spike. Test fixture selects the dash variant for openclaw.
- **openclaw — upstream Anthropic auth surfaced in bot response**: the dispatcher and OpenAI envelope work end-to-end (status='done', non-empty bot_response, OpenAI envelope parsed correctly), but the *content* of the reply is `"HTTP 401 authentication_error: invalid x-api-key"`. The OpenRouter key isn't valid against api.anthropic.com directly. This is exactly the documented spike behavior ("The content surfaced an upstream Anthropic billing error... faithfully relayed by the bot. The contract — openai_compat — is empirically validated."). The gate counts this as PASS because the contract switch is what's under test; provider-side credential routing for openclaw's anthropic-direct path is a separate concern documented in `recipes/openclaw.yaml::known_quirks.openrouter_provider_plugin_silent_fail` and `provider_compat.deferred=[openrouter]`.

### a2a_jsonrpc (nullclaw)

- Required `host=0.0.0.0` + `allow_public_bind=true` in the recipe's heredoc-written config.json (already present in the recipe — no fix needed).
- Required `chmod 0777` on the bind-mounted data dir for the `nobody` container user; the fixture handles this in the volume-mount setup.
- `nullclaw` uses the `openrouter/$MODEL` prefix in its config.json (with the `openrouter/` prefix), per the recipe heredoc — a deliberate departure from openclaw's bare `${MODEL}`.

### zeroclaw_native (zeroclaw)

- Required pre_start_commands to run BEFORE the persistent daemon (zeroclaw is distroless — no shell to chain `onboard && daemon`). Refactored fixture to run pre_start as separate `docker run --rm` invocations sharing the bind-mounted `/zeroclaw-data` volume.
- Built-in `X-Idempotency-Key` and `X-Session-Id` headers — the dispatcher's zeroclaw_native adapter sends both; the gate doesn't exercise the idempotency-replay path (covered by Plan 22c.3-13's spike).

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] recipes/hermes.yaml missing `auth_mode: bearer`**
- **Found during:** Task 2 first hermes run (got `bot_5xx:401`)
- **Issue:** Hermes recipe declared `auth_header: "Authorization: Bearer ${INAPP_AUTH_TOKEN}"` but not `auth_mode: bearer`. The InappRecipeIndex parser reads `auth_mode` (defaults to `none`); the dispatcher's openai_compat adapter only sends the Authorization header when `auth_mode='bearer'`. Without the fix the dispatcher reached the bot but didn't send the bearer token, returning 401.
- **Fix:** Added `auth_mode: bearer` to recipes/hermes.yaml line 341 (additive YAML — same shape as openclaw which already has it).
- **Files modified:** recipes/hermes.yaml (+6 lines)
- **Commit:** 9600e76

### Rule-3 (Blocking) — macOS Docker Desktop bridge-IP routing limitation

**3. [Rule 3 - Blocking] Host process can't reach docker bridge IPs on Docker Desktop**

- **Found during:** Task 2 first nullclaw run (got httpx.ConnectError)
- **Issue:** The dispatcher's InappRecipeIndex.get_container_ip returns the bridge-net IPv4 address from docker inspect; the test process running on host can't route to that IP through Docker Desktop's gRPC-FUSE/virtiofs boundary.
- **Fix:** Wrap the recipe_index fixture so get_container_ip returns "127.0.0.1" and get_inapp_block.port returns the per-test HOST_PORT; publish the recipe's port to host via `-p 127.0.0.1:HOST_PORT:CONTAINER_PORT`; force the bot to bind 0.0.0.0 inside the container (override ENV or rewrite argv).
- **Files modified:** api_server/tests/e2e/conftest.py (recipe_index wrapper + recipe_container_factory port-publish + per-recipe bind override)
- **Commit:** 9600e76
- **Note:** On Linux Docker daemon paths the bridge-IP routing works (the production code path is unchanged); the wrapper is isolated to the test fixture.

### Rule-3 — nanobot model placeholder rejection

**4. [Rule 3 - Blocking] nanobot rejects "agent" model placeholder**

- **Found during:** Task 2 first nanobot run (got `bot_5xx:400`)
- **Issue:** Recipe declares `contract_model_name: "agent"` but nanobot's OpenAI-compat surface only accepts the literal model id from its config.json (returns 400 `Only configured model 'X' is available`).
- **Fix:** recipe_index fixture wrapper substitutes the actual configured model when serving InappChannelConfig for nanobot.
- **Files modified:** api_server/tests/e2e/conftest.py (recipe_index wrapper + factory contract_model_name override)
- **Commit:** 9600e76
- **Tracked as deferred:** dispatcher's openai_compat adapter could fall back to `row['model']` when `contract_model_name` is the "agent" placeholder. Single-line dispatcher change; out of Plan 15 scope.

### Rule-3 — zeroclaw distroless pre_start ordering

**5. [Rule 3 - Blocking] zeroclaw pre_start_commands need to run BEFORE the persistent daemon**

- **Found during:** Task 2 first zeroclaw run (daemon failed with `Unknown provider: default`)
- **Issue:** Zeroclaw is distroless (no shell). Pre_start_commands as docker-exec calls fail because the daemon needs config.toml in place at start. Recipe pattern is "share a named volume across two `docker run` invocations".
- **Fix:** Refactored factory to run pre_start_commands as separate `docker run --rm` invocations sharing the bind-mounted volume BEFORE the persistent `docker run -d`.
- **Files modified:** api_server/tests/e2e/conftest.py
- **Commit:** 9600e76
