## contract_model_name="openclaw" decision recap

The dispatcher's `openai_compat` adapter (Plan 22c.3-05, `services/inapp_dispatcher.py`) constructs the request body for hermes/nanobot/openclaw uniformly:

```python
body = {
    "model": inapp.get("contract_model_name") or "agent",
    "messages": [{"role": "user", "content": <user message verbatim>}],
}
```

For hermes (plan 10) and nanobot (plan 11) the recipe omits the field, so the dispatcher sends `model="agent"`. For openclaw the recipe sets `contract_model_name: "openclaw"`, so the dispatcher sends `model="openclaw"`.

This is empirically required: openclaw's gateway returns HTTP 400 `Invalid model. Use openclaw or openclaw/<agentId>` for any other value, including raw upstream model ids like `openrouter/anthropic/claude-haiku-4.5` (verified spike lines 113-117). The bot internally maps `model="openclaw"` to the configured `agents.defaults.model.primary` (e.g. `anthropic/claude-haiku-4-5` from our heredoc) and routes the upstream provider call from there.

**No special-case code in the dispatcher** — just a generic recipe-driven rewrite. Adding a 4th recipe with a different `contract_model_name` is a single YAML edit; the dispatcher already reads `inapp.get('contract_model_name') or 'agent'` (Plan 22c.3-05 line: `body["model"] = (inapp.contract_model_name or "agent")`).

## `rm -f openclaw.json5` defense recap

The persistent_argv_override sh-chain runs `rm -f /home/node/.openclaw/openclaw.json5` AFTER writing `openclaw.json` and BEFORE `exec openclaw gateway run`. This defends against:

- A leftover JSON5 config from a previous container restart (per the existing telegram-channel persistent.spec.argv comment at line 341: "remove any leftover .json5 so the runtime doesn't try to reconcile").
- A baked-in JSON5 default in the image that the runtime might prefer over our hand-written .json (verified empirically in the existing telegram path; same defense applies here).

Without this `rm`, openclaw's config loader can pick up the older .json5 file and ignore our chatCompletions/auth/model overrides — silently. The 1-line defense is cheap and the empirical evidence is the existing telegram path's persistent.spec.argv (line 329) which carries the same line for the same reason.

## Decisions Made

- **Provider-prefix discipline (Rule 1 deviation from plan template):** Heredoc uses `"primary": "${MODEL}"` (no `openrouter/` prefix). The plan template at line 130 specified `"openrouter/${MODEL}"`, but this contradicts (a) must_haves.key_links requirement that the heredoc mirror the spike's verbatim config, (b) the Wave 0 spike line 27 (`"primary": "anthropic/claude-haiku-4-5"`), (c) the recipe's `provider_compat.deferred=[openrouter]` + `provider_compat.supported=[anthropic]`, and (d) `known_quirks.openrouter_provider_plugin_silent_fail` documenting that the openrouter plugin is upstream-broken. Shipping `openrouter/${MODEL}` would silently produce 0-token completions per the documented bug.
- **activation_env declares ANTHROPIC_API_KEY (not OPENROUTER_API_KEY):** Same Rule 1 alignment.
