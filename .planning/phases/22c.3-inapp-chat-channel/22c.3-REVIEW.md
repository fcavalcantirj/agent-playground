# Phase 22c.3: Code Review Report

**Reviewed:** 2026-04-30
**Depth:** standard
**Files Reviewed:** 41
**Status:** issues_found

## Summary

Phase 22c.3 lands a substantial in-app chat channel: 3 new HTTP routes, an asyncio dispatcher
with 3 contract adapters (openai_compat / a2a_jsonrpc / zeroclaw_native), a transactional outbox
pump (PG → Redis Pub/Sub), a reaper, two Alembic migrations, 5 recipe YAML extensions, and a
5-cell e2e gate. Overall the design is rigorous: ownership filters live at the SQL layer
(matching the project's defense-in-depth pattern), all SQL uses asyncpg parameterization
(no string interpolation observed), the SSE replay → subscribe → differential-replay race
window (Pitfall 1) is handled, persist-before-action discipline holds across dispatcher
and reaper, and Redis is correctly compose-internal-only in `docker-compose.prod.yml` (no host port).

The 5/5 e2e gate currently shows `passed: true` end-to-end against real OpenRouter, but
**one cell (`openclaw`) returns `bot_response_excerpt: "HTTP 401 authentication_error: invalid x-api-key"`** —
this is a true bot-layer 200 response carrying an upstream-billing/credential error verbatim
("dumb-pipe" behavior per D-22). It is NOT a code defect, but it exposes a soft-spot in the gate's
`bot_response_excerpt` semantics and is worth flagging.

The most important correctness finding is **CR-01**: a Dockerfile/runtime version skew between
`tools/Dockerfile.api` (`redis>=5.2,<7`) and `api_server/pyproject.toml` (`redis>=5.2,<8`) — the
built API image will install redis 5.x or 6.x while the test/dev environment uses redis 7.x.
This is exactly the kind of drift that produces a "works on my laptop, breaks at deploy" surprise
during the next prod roll.

The remaining issues are split between operational hardening (rate-limit/idempotency fail-open
behavior on chat creates an attack window for throttle bypass; the SSE generator's defensive
re-subscribe loop has unbounded retry cost on a flapping Redis), defense-in-depth gaps in the
SSE handler's ownership assumption around stale containers, and a few minor code-quality items.
None of them block ship; CR-01 should be resolved before the next deploy and the warning items
queued before going multi-tenant at scale.

## Critical Issues

### CR-01: Dockerfile.api pins `redis<7` while pyproject.toml pins `redis<8` — version skew between built image and test env

**File:** `tools/Dockerfile.api:27`
**Issue:** The Dockerfile build stage installs runtime deps via an explicit `pip install` list that
pins `'redis>=5.2,<7'`, but `api_server/pyproject.toml:60` pins `"redis>=5.2,<8"` (and the
PR description even explicitly says the upper bound was bumped to `<8` for testcontainers'
`testcontainers[redis]>=4.14.2` transitive requirement). The result is that the built API
container will ship redis-py 5.x (likely 5.2-5.x — whichever is the latest <7), while every
test path (host venv + testcontainers) installs redis-py 7.x. The redis 5.x → 7.x asyncio API
is mostly source-compatible but the Pub/Sub helper calls used by `inapp_outbox._pump_once`
(`redis_client.publish`) and `routes/agent_messages.messages_stream` (`redis_client.pubsub()`,
`pubsub.subscribe`, `pubsub.get_message(ignore_subscribe_messages=...)`) have shifted minor
behaviors across the major boundary. Even if today they happen to work, two truths from this
review process are worth honoring: (a) the comment in `pyproject.toml:54-58` explicitly documents
why the upper bound was loosened (Plan 22c.3-07 deviation Rule 3), and (b) the Dockerfile is the
artifact that ships to prod. Letting them diverge means **prod runs a different redis-py major

designed to prevent.

```dockerfile
      'asgi-correlation-id>=4.3.4,<5' 'ruamel.yaml>=0.17.21' 'jsonschema>=4.23' \
      'pyrage>=1.2' 'cryptography>=42' 'docker>=7.0,<8' \
      'authlib>=1.6.11' 'itsdangerous>=2.2.0' 'httpx>=0.27' \
```

Better still, drive the Dockerfile install from `pyproject.toml` (`pip install

## Warnings

### WR-01: SSE handler trusts `agent_id` ownership but resolves an old stopped container that may belong to a deleted agent

**File:** `api_server/src/api_server/routes/agent_messages.py:312-321`
**Issue:** The SSE endpoint resolves `agent_id` (= `agent_instances.id`) to a `container_row_id`
via `ORDER BY (stopped_at IS NULL) DESC, created_at DESC LIMIT 1`. When the agent has NO
`running` container but has a stopped one, the handler still subscribes (with stale data) and
streams replay events — which is fine. The defense-in-depth concern is the second branch: if a
new admin/owner action ever transfers an `agent_instances` row across users (today it cannot —
`fetch_agent_instance` filters by `user_id` first), and the old `agent_containers` row remains
keyed to a previous `user_id`, the SSE consumer would be reading the prior user's container
events. There is no transfer flow today, so this is theoretical, but the SQL contract should be
tightened to require the container's `user_id` matches the session user as well, mirroring the
discipline established in `inapp_messages_store.fetch_by_id`.

**Fix:**
```python
container_row_id = await conn.fetchval(
    """
    SELECT id FROM agent_containers
    WHERE agent_instance_id = $1
      AND user_id = $2
    ORDER BY (stopped_at IS NULL) DESC, created_at DESC
    LIMIT 1
    """,
    agent_id, user_id,
)
```
Same fix applies to the `delete_history` route at line 230 (the `DELETE FROM agent_events ...
WHERE agent_container_id IN (SELECT id FROM agent_containers WHERE agent_instance_id = $1)`
subselect would benefit from `AND user_id = $2`).

### WR-02: SSE `pubsub_get_message_failed` defensive branch can busy-loop on a flapping Redis

**File:** `api_server/src/api_server/routes/agent_messages.py:425-464`
**Issue:** When `pubsub.get_message(...)` raises (Redis connection drop, transient error), the
handler logs, unsubscribes, re-subscribes, runs a PG catch-up query, and `continue`s the loop.
There is no backoff. If Redis is in a flapping state where every `get_message` call raises,
the loop will hammer Postgres with `fetch_events_after_seq` queries at full CPU until either
the client disconnects (which is detected only on the next iteration's
`request.is_disconnected()` check, *after* the catch-up query runs) or Redis recovers. With
hundreds of concurrent SSE clients, this becomes an outage amplifier.

**Fix:** Add an exponential backoff sleep on the exception path, capped at ~5s, and check
`request.is_disconnected()` BEFORE running the catch-up query:
```python
except Exception:
    _log.exception("phase22c3.sse.pubsub_get_message_failed")
    if await request.is_disconnected():
        return
    # Backoff before re-subscribe + PG catch-up
    await asyncio.sleep(min(0.5 * (1 + _consecutive_failures), 5.0))
    _consecutive_failures += 1
    try:
        await pubsub.unsubscribe(channel)
    except Exception:
        pass
    await pubsub.subscribe(channel)
    # ... existing PG catch-up ...
    continue
# On any successful get_message (msg is None or msg has data), reset:
_consecutive_failures = 0
```

### WR-03: Rate-limit middleware fail-open semantics let an attacker bypass the chat 4/min cap during PG hiccups

**File:** `api_server/src/api_server/middleware/rate_limit.py:170-180`
**Issue:** The middleware fails open on any `Exception` from
`check_and_increment(conn, subject, bucket, limit, window_s)`. The comment correctly identifies
the trade-off (T-19-05-06 — "deliberate"), and for `runs` / `lint` / `get` buckets where
abuse is rate-bounded by upstream cost (LLM calls) this is fine. The chat path is different:
its bucket is 4/min (D-42, very tight precisely because the chat fast-ack path is cheap), AND
the chat dispatcher loop will pull and forward all queued messages regardless of rate-limit
state. A user (or attacker) who triggers a brief PG outage can flood `inapp_messages` with
hundreds of pending rows in seconds, all of which the dispatcher will attempt to forward (and
which OpenRouter will charge for through the bot's container env-injected key). The
fail-open posture is right for `runs` (LLM cost is the real throttle) but probably wrong for
`chat` (where the per-message cost is the throttle).

**Fix:** Bucket-specific fail policy. For `chat`, fail closed with a 503 + Retry-After:
```python
except Exception:
    _log.exception("rate_limit backend error")
    if bucket == "chat":
        # Chat path: fail closed — D-42 4/min is the cost throttle.
        body = json.dumps(make_error_envelope(
            ErrorCode.RATE_LIMITED,
            "rate-limit backend unavailable; please retry",
            param=bucket,
        )).encode()
        await send({
            "type": "http.response.start", "status": 503,
            "headers": [
                (b"content-type", b"application/json"),
                (b"retry-after", b"30"),
                (b"content-length", str(len(body)).encode()),
            ],
        })
        await send({"type": "http.response.body", "body": body})
        return
    # All other buckets: fail open per T-19-05-06.
    await self.app(scope, receive, send)
    return
```

### WR-04: Idempotency middleware caches 202 + replays 200 — a replayed chat POST returns wrong status code

**File:** `api_server/src/api_server/middleware/idempotency.py:215-219, 258`
**Issue:** Plan 22c.3-08 widens the cache predicate from `{200}` to `{200, 202}` (line 258),
which is correct for caching the chat fast-ack body. But the cache replay path on line 219 always
emits status 200 (`await _send_json(send, 200, cached or {})`), regardless of whether the
original response was 200 (runs) or 202 (chat). A client that retries with the same Idempotency-Key
and previously got back a 202 will now see 200 with an unchanged body. SDK clients keying off
HTTP status will see two different statuses for the same logical operation, which violates
Stripe-style idempotency semantics. The fix requires storing the original status in
`idempotency_keys` and replaying it.

**Fix:**
1. Add a `status_code` column (or recycle an existing JSONB column) to `idempotency_keys`.
2. Update `services.idempotency.write_idempotency` and `check_or_reserve` to round-trip the
   status code.
3. Replay using the stored code:
```python
if tag == "hit":
    cached_status = (cached or {}).get("__status__", 200)
    cached_body = {k: v for k, v in (cached or {}).items() if k != "__status__"}
    await _send_json(send, cached_status, cached_body)
    return
```
Lower-effort interim fix: special-case the chat path in the replay branch to emit 202 since
that's the only non-200 status currently in the cache predicate. Less general but works today:
```python
if tag == "hit":
    is_chat = _AGENT_MESSAGES_PATTERN.match(path) is not None
    await _send_json(send, 202 if is_chat else 200, cached or {})
    return
```

### WR-05: `_handle_row` silently masks `bot_invalid_response` as `internal` to fit the Pydantic enum

**File:** `api_server/src/api_server/services/inapp_dispatcher.py:404-414, 305-325`
**Issue:** Plan 04 declares 9 canonical `error_type` enum values for `InappOutboundFailedPayload`
(line 184-191 of `models/events.py`). But the dispatcher emits `bot_invalid_response`
(line 413, on `httpx.RequestError` / `ValueError` / JSON-decode failure) which is NOT in the enum;
`_truncate_error_type` collapses it silently to `internal` (line 325 default branch). The
`inapp_messages.last_error` column receives the rich `bot_invalid_response` string, but the
SSE outbound `agent_events.payload.error_type` only sees `internal`. Operators reading the
SSE/dashboard cannot tell apart "bot returned non-JSON" from "dispatcher had an unclassified
internal bug" without correlating against the underlying `inapp_messages` row — which is doable
but defeats the purpose of having an enum.

**Fix:** Either (a) extend the `InappOutboundFailedPayload.error_type` Pydantic pattern to add
`bot_invalid_response`, or (b) rename the dispatcher's emission to `bot_empty` (which is in the
enum, but means something subtly different — empty 200 vs. non-JSON 200). (a) is the correct fix:
```python
# In api_server/src/api_server/models/events.py:184-191:
error_type: str = Field(
    ...,
    pattern=(
        r"^(bot_5xx|bot_timeout|bot_empty|bot_invalid_response|"
        r"container_dead|recipe_no_inapp_channel|container_not_ready|"
        r"recipe_missing|reaper_timeout|internal)$"

# And in inapp_dispatcher.py:_KNOWN_ERROR_TYPES set, add "bot_invalid_response".
```

### WR-06: `inapp_messages_store.restart_sweep` and other `int(res.split()[-1])` parsers are fragile

**File:** `api_server/src/api_server/services/inapp_messages_store.py:289-292, 354-357`
**Issue:** `restart_sweep` and `delete_history_for_agent_user` parse the integer count out of
asyncpg's command-tag string ("UPDATE N" / "DELETE N") via `int(res.split()[-1])` with a try/except
fallback to 0. The current asyncpg returns this format reliably for simple statements, but the
behavior is undocumented at the asyncpg level (it's a passthrough of the Postgres protocol's
CommandComplete tag) and can produce non-numeric tags for compound commands. More importantly,
this hides a bug case: if the tag format ever changes, `restart_sweep` will report 0 rows
silently and the lifespan log will report a misleading "swept: 0" — the sweep is opportunistic
so this is non-fatal, but the metric for "did we recover stuck rows on this restart?" is
silently broken. Same fragility on the DELETE.

**Fix:** Use `RETURNING id` (or `RETURNING 1`) and `len()` of the result:
```python
async def restart_sweep(conn, threshold_minutes: int = 15) -> int:
    rows = await conn.fetch(
        """
        UPDATE inapp_messages
        SET status='pending'
        WHERE status='forwarded'
          AND last_attempt_at < NOW() - make_interval(mins => $1)
        RETURNING id
        """,
        threshold_minutes,
    )
    return len(rows)
```

### WR-07: Recipe YAML `persistent_argv_override` heredocs interpolate `${MODEL}` and `${OPENROUTER_API_KEY}` via sh — shell-injection surface

**File:** `recipes/openclaw.yaml:817-850`, `recipes/nullclaw.yaml:460-473`,
`recipes/nanobot.yaml`, `recipes/zeroclaw.yaml:286-301`, `recipes/hermes.yaml:347-356`
**Issue:** Three of the five recipe YAMLs ship `persistent_argv_override.argv` as `sh -c` chains
with heredocs that interpolate `${MODEL}`, `${INAPP_AUTH_TOKEN}`, `${OPENROUTER_API_KEY}`, and
similar at sh expansion time inside the container. Today the runner passes these via Docker env
(`-e`) and the values originate from server-side mints (UUID hex tokens, validated provider
keys in DB) — so there is no immediate injection vector. But the values are ALSO substituted
into the JSON heredoc body unquoted; if any one of these env values ever contains a `}`,
`"`, backtick, or `$(...)`, the resulting `openclaw.json` / `config.json` becomes
malformed — at best a parse error at bot boot, at worst (for a shell-metachar-laden value) a
shell injection inside the container. The recipe layer is the place where validation is
weakest because it is YAML-as-code by design. Worth tightening before BYOK accepts free-form
provider-key strings.

**Fix:** Constrain the substitution layer (the e2e helper `render_placeholders` and the future
production runner that consumes it) to URL-encode or JSON-escape values that flow into a heredoc
body, vs. left-alone for plain env injection. Concretely, in
`api_server/tests/e2e/_helpers.py:render_placeholders`, when the value will be embedded in a
heredoc the substitutions should pass through `json.dumps(value)[1:-1]` (JSON string-escape
without surrounding quotes). Alternative: stop using sh-heredoc for config writes and have the
runner write `openclaw.json` to a bind-mounted dir directly (the runner already has full data
on what to write, and bypassing sh removes the injection surface entirely). Lowest-friction fix
for now: reject any provider-key / model string that contains `[$"\`{}\\\\]` at registration
time (current keys/IDs are alphanumeric + dashes + slashes, so this is safe to enforce).

### WR-08: Outbox pump's "duplicate-publish on retry" tradeoff (Pitfall 3 strategy 2) relies on SSE-client de-dup that the production HTML/JS client doesn't yet implement

**File:** `api_server/src/api_server/services/inapp_outbox.py:79-160`, `routes/agent_messages.py:472-485`
**Issue:** The outbox `_pump_once` documents that on a mid-batch Redis `RedisError` it abandons
the bulk UPDATE — meaning the rows that DID successfully publish before the error get re-published
on the next tick. The SSE handler's live consumer dedupes via `last_yielded_seq` (line 472-485),
which works fine for one connected client. But the outbox publishes ONCE per tick to a Redis
channel that may have multiple subscribers (multi-tab SSE for the same user). If a downstream
JS client is naive (not seq-aware) it will render the duplicate. The CONTEXT.md D-10 invariant
documents the requirement ("idempotent client-side handling"), but the front-end is a separate
phase and this expectation is enforced via documentation, not via a contract check. Worth
flagging for the front-end review.

**Fix:** None on the backend side — the strategy is sound and explicitly documented. Add a
linter / type-gen check in the front-end phase that the SSE consumer carries a `Map<seq,
event>` dedup buffer or refuses to render `event.id <= lastSeenId`.

### IN-01: SSE generator: dropping `await asyncio.wait_for(stop_event.wait(), …)` symmetry

```python
async def event_generator():

        if await request.is_disconnected():
            return
        if stop is not None and stop.is_set():
            return
        # ...
```

### IN-02: Migration 008 reverses safely only if no chat-path rows have been written

**File:** `api_server/alembic/versions/008_idempotency_keys_relax_run_fk.py:60-75`
**Issue:** `downgrade()` re-adds the FK + NOT NULL but the docstring correctly notes that any
chat-path cache row (where `run_id` holds a `message_id` that isn't in `runs`) will fail FK
reinstatement. This is correctly disclosed as a documentation issue, not a code bug. Worth
adding a guard so the downgrade fails loud with a helpful message:

**Fix:**
```python
def downgrade() -> None:

        DO $$
        DECLARE n int;
        BEGIN
            SELECT COUNT(*) INTO n FROM idempotency_keys

    op.create_foreign_key(
        "idempotency_keys_run_id_fkey",
        "idempotency_keys", "runs", ["run_id"], ["id"],
    )
```

### IN-03: e2e gate: openclaw cell PASSes with a 401 in `bot_response_excerpt`

**File:** `api_server/tests/e2e/e2e-report.json:25-33`
**Issue:** The openclaw cell records `"status": "PASS"` and `"bot_response_excerpt": "HTTP 401
authentication_error: invalid x-api-key"`. The dispatcher correctly forwards the bot's HTTP 200
response verbatim (D-22 dumb-pipe), and the bot's content happened to be an upstream API
authentication error from Anthropic — this is a real round-trip that exercised the contract
adapter correctly. The test passes because all it checks is `len(bot_response) > 0`. The
recipe author has already documented this in `recipes/openclaw.yaml:881-887` ("content surfaced
the developer's zero-credit ANTHROPIC_API_KEY billing error verbatim through the bot — NOT a
contract failure"). It's still worth flagging because future code reviewers will look at this
file once and assume "PASS" means "bot answered the prompt" — which here it does NOT. The gate's
PASS criterion is "the dispatcher delivered a non-empty 200-bot-body to PG", which is the
correct contract-level criterion.

**Fix:** Add a soft gate that warns (does not fail) when `bot_response_excerpt` matches a
canonical "looks-like-an-upstream-error" regex (`/HTTP \d{3}|401|429|invalid.*api.?key/i`), so the
gate at least flags the cell for human review. Or carry the openclaw exception into a
`recipe_known_quirks` list that surfaces in the report. Not a code fix — a reviewability fix.

### IN-04: `_BYOK_LEAK_RE` (`sk-[a-zA-Z0-9]{40,}`) misses Anthropic's `sk-ant-` shape (longer dash separator)

**File:** `api_server/src/api_server/routes/agent_messages.py:61-63`
**Issue:** The regex matches `sk-` followed by 40+ alphanumerics — but Anthropic API keys take
the shape `sk-ant-api03-...` which contains dashes inside the suffix. The regex's `[a-zA-Z0-9]+`
class will fail to match the embedded `-`, so the run-on `sk-ant-api03-` prefix doesn't reach
40 alphanumerics in a row. Anthropic keys would not be detected by this guard. OpenRouter keys
(`sk-or-v1-...`) similarly have dashes inside, but the test at line 273 passes a key that
already has 40+ continuous alphanumerics after the last dash, so the test happens to match.

**Fix:**
```python
_BYOK_LEAK_RE = re.compile(
    r"(OPENROUTER_API_KEY|ANTHROPIC_API_KEY|OPENAI_API_KEY|"
    r"sk-(?:[a-zA-Z0-9_-]{40,}|ant-[a-zA-Z0-9_-]{40,}|or-[a-zA-Z0-9_-]{40,}))"
)
```
Add a regression test exercising a real-shape Anthropic and OpenRouter key (synthetic, not

### IN-05: `inapp_recipe_index._load_from_disk` swallows YAML parse errors and caches `None`

**File:** `api_server/src/api_server/services/inapp_recipe_index.py:285-306`
**Issue:** When ruamel raises on a malformed YAML, the function catches `Exception`, logs, and
returns `_CachedEntry(config=None, mtime=...)`. Subsequent `get_inapp_block` calls see the
cached `None` and dispatch the corresponding inapp_message to `_terminal_failure` with
`recipe_lacks_inapp_channel` — which is misleading; the user saw a malformed recipe
counted against "this recipe doesn't support inapp" rather than "the recipe file on disk is
broken". The cached entry remains until mtime changes again. For a recipe file that's broken
upstream and never edited, this means every chat to that agent fails with a wrong reason.

**Fix:** Either (a) raise on YAML parse error (fail-loud at first access — preferable, since
recipes are admin-curated), or (b) introduce a third state (`_CachedEntry.error: str | None`)
and have the dispatcher emit `recipe_missing` (in the enum) when `error` is set:
```python
@dataclass
class _CachedEntry:
    config: InappChannelConfig | None
    mtime: float
    error: str | None = None
# ...
except Exception as e:
    return _CachedEntry(config=None, mtime=path.stat().st_mtime, error=f"yaml_parse:{e}")
```
Then surface `error` to the dispatcher via a new `get_inapp_block_or_error()` method.

### IN-06: `inapp_dispatcher.dispatcher_loop` doesn't accept the `stop_event` argument the lifespan passes for the reaper / outbox loops

```python
# In inapp_dispatcher.py:449:
async def dispatcher_loop(state: Any, stop_event: asyncio.Event) -> None:
    pool = state.db
    while not stop_event.is_set():
        # ...
```
Drop the `getattr(state, "inapp_stop", None) or asyncio.Event()` fallback at line 479.
