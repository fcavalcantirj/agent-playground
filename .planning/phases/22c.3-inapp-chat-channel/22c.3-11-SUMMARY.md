## A5 answer used + auth_header value chosen

**A5 = no_auth.** Source: `spikes/recipe-nanobot.md` line 95 ("No built-in auth gate — the recipe binds 127.0.0.1 in production and the api_server's dispatcher reaches it via Docker bridge networking; auth is enforced upstream by `require_user`.").

`auth_header: "none"` set in the YAML. The dispatcher's `openai_compat` adapter (Plan 22c.3-05) skips the Authorization header when `auth_mode='none'` (the default; not set on YAML, parser falls back to 'none').

## v1 acceptable risk note (A5=no_auth)

The bot's HTTP listener binds 127.0.0.1 inside the container's network namespace. The api_server dispatcher (also on the deploy_default Docker bridge network) reaches the bot via the bridge IP, NOT through host-port mapping. External access from outside the bridge is blocked at the Docker network boundary. Auth on the user-facing surface is enforced upstream by `require_user` on `POST /v1/agents/:id/messages` (Plan 22c.3-08). If a future hardening phase adds an auth gate (e.g., a small aiohttp proxy in front of `nanobot serve`), the YAML changes are minimal: auth_header flips to "Authorization: Bearer ${INAPP_AUTH_TOKEN}" and activation_env gains the bearer env var; no schema change.

## Decisions Made

- **persistent_argv_override.host=127.0.0.1** chosen over the spike's `--host 0.0.0.0` for parity with the hermes pattern (Plan 22c.3-10): production binds loopback inside the container's network namespace; the spike used 0.0.0.0 for host-port-mapping reachability during the docker run probe. The api_server dispatcher reaches the bot via the bridge IP using the same `containers.get(...).attrs['NetworkSettings']['Networks'][network_name]['IPAddress']` lookup (`InappRecipeIndex.get_container_ip`), so loopback-bound listeners inside the container's namespace are reachable IF the bot's HTTP server explicitly accepts connections on 127.0.0.1 from within the same network namespace. (If e2e Plan 22c.3-15 surfaces a connectivity issue here, the fix is changing `--host` to `0.0.0.0` and re-running — single-line change.)
- **auth_header="none"** per A5=no_auth (spike line 95). The contract-level adapter shape stays the same; the dispatcher already handles `auth_mode='none'` by skipping the Authorization header. No INAPP_AUTH_TOKEN injection required for nanobot today; runner side is unchanged for this recipe.
- **`{}` (empty map) on activation_env** — nanobot serve inherits the OPENROUTER_API_KEY from `runtime.process_env.api_key` via the same channel-input mechanism the gateway path uses. No inapp-specific env var injection needed.
- **`inapp:` block placed BEFORE the trailing `# Discord, Slack, ...` deferred-channels comment** — same convention as plan 22c.3-10 hermes: empirically validated channels live with telegram, deferred candidates come after.

## Deviations from Plan

None — plan executed exactly as written.

The plan flagged a possible scope-expansion if the InappRecipeIndex parser strict-validated the new `persistent_argv_override` key (which would force a parser extension). It does not — the parser only reads the keys it knows about (transport / port / endpoint / contract / contract_model_name / request_envelope / response_envelope / auth_mode / idempotency_header / session_header) and silently ignores everything else. The runner-side concern (consuming persistent_argv_override at deploy time) is out of scope for this plan and is flagged as future work in the plan 22c.3-10 SUMMARY's "Next Phase Readiness" section (the same INAPP_AUTH_TOKEN-mint extension to `start_persistent` applies; nanobot's auth_header="none" means INAPP_AUTH_TOKEN injection is a no-op for this recipe regardless).
