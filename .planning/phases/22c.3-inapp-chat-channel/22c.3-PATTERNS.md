# routes/agent_lifecycle.py lines 226-260 (canonical POST handler shape)
@router.post("/agents/{agent_id}/start")
async def start_agent(
    request: Request,
    agent_id: UUID,
    body: AgentStartRequest,
    authorization: str = Header(default=""),
):
    # --- Step 1a: Session cookie → user_id (require_user gate) ---
    session_result = require_user(request)
    if isinstance(session_result, JSONResponse):
        return session_result
    user_id: UUID = session_result

    # --- Step 1b: Authorization header → provider_key (memory only) ---
    if not authorization.startswith("Bearer "):
        return _err(401, ErrorCode.UNAUTHORIZED, "Bearer token required",
                    param="Authorization")
    # ...

    # --- Step 2: resolve agent_instance by (agent_id, user_id) ---
    pool = request.app.state.db
    async with pool.acquire() as conn:
        agent = await fetch_agent_instance(conn, agent_id, user_id)
    if agent is None:
        return _err(404, ErrorCode.AGENT_NOT_FOUND,

**Apply to D-07 / D-29:** Same auth + ownership prologue. Phase 22c.3 differs in step 1b: NO `Authorization` Bearer required (chat is session-only — D-18 says SSE uses `ap_session` cookie alone; POST follows the same precedent — drop the bearer-parse block). Then `INSERT inapp_messages(status='pending')` and return `202 Accepted` with `{message_id}`.

    status: int,
    code: str,
    message: str,

#### SSE handler — closest precedent is `agent_events.py:228-270` two-scope DB pattern + Pitfall 4

```python
# routes/agent_events.py lines 238-270 — Pitfall 4 two-scope DB discipline
async with poll_lock:
    # --- Step 3b: CLEAR the wake signal BEFORE any fetch ---
    signal = _get_poll_signal(request.app.state, agent_id)
    signal.clear()

    # --- Step 4: DB scope 1 (fast path), wake signal armed ---
    async with pool.acquire() as conn:
        rows = await fetch_events_after_seq(
            conn, agent_id, since_seq, kinds_set
        )
    # --- Step 5: immediate return if rows exist ---
    if rows:
        return _project(rows, since_seq, agent_id, timed_out=False)

    # --- Step 6: NO DB held during the wait (Pitfall 4) ---
    try:
        await asyncio.wait_for(signal.wait(), timeout=timeout_s)
    except asyncio.TimeoutError:
        return _project([], since_seq, agent_id, timed_out=True)

    # --- Step 8: DB scope 2 (re-query after signal fires) ---
    async with pool.acquire() as conn:
        rows = await fetch_events_after_seq(
            conn, agent_id, since_seq, kinds_set
        )
```

**Apply to D-25 SSE replay:** the SSE generator's first phase is the same "DB scope 1 → release → wait/subscribe → DB scope 2" rhythm. Difference: the wait is on `redis_client.pubsub().get_message(timeout=1.0)` instead of `signal.wait()`. **Critical:** never hold a Postgres connection across the Redis subscribe loop — same Pitfall 4 reasoning.

**For Pitfall 1 race window** (event arrives between PG replay end and Redis subscribe attach): mirror the clear-then-fetch idiom at `agent_events.py:239-247` — capture `current_max_seq` AFTER replay, attach Redis subscriber, then differentially re-query for `seq > replay_max AND seq <= current_max` before yielding live messages.

#### `_get_poll_lock` / `_get_poll_signal` mutex idiom — extract from `watcher_service.py:41-63`

```python
# services/watcher_service.py lines 41-63 — locks_mutex pattern
async def _get_poll_lock(app_state, agent_id: UUID) -> asyncio.Lock:
    """Return (creating if needed) the per-agent long-poll ``asyncio.Lock`` (D-13)."""
    async with app_state.locks_mutex:
        lock = app_state.event_poll_locks.get(agent_id)
        if lock is None:
            lock = asyncio.Lock()
            app_state.event_poll_locks[agent_id] = lock
    return lock


def _get_poll_signal(app_state, agent_id: UUID) -> asyncio.Event:
    """Return (creating if needed) the per-agent watcher→handler wake signal."""
    signal = app_state.event_poll_signals.get(agent_id)
    if signal is None:
        signal = asyncio.Event()
        app_state.event_poll_signals[agent_id] = signal
    return signal
```

**Apply to dispatcher / SSE handler:** if Phase 22c.3 needs any per-agent in-process state (e.g., a "concurrent SSE connection cap" counter — RESEARCH Open Question #2), use this `locks_mutex`-guarded pattern. **Note:** the existing dict `app.state.event_poll_signals` is keyed on `container_row_id` per Spike B 2026-04-19 (URL `agent_id` IS `agent_containers.id`). RESEARCH Pattern 1 SSE handler at line 311 uses `agent_id` as the same key — confirm the dispatcher emits inapp_outbound events with the matching key (`row["container_row_id"]`).

### NEW: `services/inapp_outbox.py` — **GREENFIELD: no existing analog**

The codebase has **no transactional-outbox pattern**. The only Redis usage anywhere in `api_server` is **net-new** in this phase. RESEARCH §Pattern 3 is the source-of-truth code sketch (lines 658-731 in 22c.3-RESEARCH.md). The planner MUST inline the full sketch into the task action — there's no analog file to point at.

**Closest precedent for the outbox loop's *control flow*:** `services/watcher_service.py::run_watcher` consumer block at `watcher_service.py:526-567` (asyncio.Queue consumer with batch flush + flush trigger). The watcher's batcher → `insert_agent_events_batch` → `_get_poll_signal(...).set()` is structurally similar to outbox's batch-pick → `redis.publish(...)` → UPDATE published=true. Difference: outbox reads from Postgres (not from an asyncio.Queue) and writes to Redis (not Postgres).

**Greenfield risk callouts for the planner:**
- **Pitfall 3 double-publish on tx retry:** RESEARCH §Pitfall 3 lays out two strategies; the recommended one is "optimistic batch with single UPDATE on success." Planner must decide explicitly and document.
- **Pitfall 4 Redis broker bounce:** RESEARCH §Pitfall 4 — outbox loop catches `redis.RedisError`, aborts the batch's UPDATE, retries on next tick. SSE generator is the consumer-side complement (catches the same exception, re-subscribes, runs `seq > last_seen_seq` PG replay).

## Anti-Patterns (from prior phases — DO NOT replicate)

### From `routes/agent_events.py:31-39` Pitfall 4

> Holding a Postgres connection across `await asyncio.wait_for(signal.wait(), timeout)` exhausts the pool. **Two-scope discipline.**

**Apply to:** dispatcher's bot HTTP `await client.post(timeout=600)`; SSE handler's `redis.pubsub().get_message(timeout=1.0)` loop; outbox pump's `redis.publish()`.

### From CLAUDE.md (project-level rule, Phase 22c-06 forcing function)

> "Don't introduce ANONYMOUS_USER_ID-style placeholders. Every chat write requires a real user_id."

The `users` table no longer has a seed-anonymous row (`006_purge_anonymous.py` truncated it). The new `inapp_messages` FK to `users(id) ON DELETE CASCADE` is enforced at the schema level — there's no escape hatch.

### From `models/events.py:6-10` (D-06 watcher leak prevention)

> `ConfigDict(extra="forbid")` rejects payloads carrying any non-declared field BEFORE INSERT.

**Apply to all 3 new payload classes** (`InappInboundPayload` etc.) — keep the `model_config = ConfigDict(extra="forbid")` line. Per D-22 dumb-pipe, the chat content IS the user-visible message — the `content` field is declared and allowed. `extra="forbid"` is still the defense against accidental field additions.

### From `routes/runs.py` (BYOK redaction discipline)

`agent_lifecycle.py::_redact_creds` (lines 111-133) — apply the same shape if any subprocess output flows into `inapp_messages.content` or `bot_response` columns. CONTEXT.md anti-pattern note: "BYOK keys could leak via prompt injection — pattern-match the `OPENROUTER_API_KEY=` shape in inbound chat content before persisting (defense in depth)."

### From RESEARCH §Anti-Patterns

- Don't subscribe to Redis Pub/Sub without `async with redis.pubsub() as pubsub:` — leaks subscribers on disconnect.
- Don't fire-and-forget `redis.publish()` after a DB INSERT — that's a broken outbox; use `published=false` + outbox pump.
- Don't build a custom `ServerSentEvent` formatter — sse-starlette ships one. RESEARCH §Don't Hand-Roll has the table.
