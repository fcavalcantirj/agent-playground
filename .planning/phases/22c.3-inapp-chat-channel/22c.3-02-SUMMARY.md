## Decisions Made

1. **Backfill agent_events.published=TRUE for pre-007 rows.** Without this, the Plan 22c.3-07 outbox pump would re-publish every historical Phase 22b event on first start (D-33 says "every event INSERT starts published=false" — that's about NEW events; the column add must seed existing rows to TRUE so they're not picked up as un-published work). Backfill lands as `op.execute("UPDATE agent_events SET published = TRUE")` in the upgrade body, after `add_column` and before `create_index` — same alembic transaction as the schema change.

2. **Partial index on (id) WHERE published=false (not (seq) WHERE published=false).** The pump's SELECT uses `ORDER BY seq` per D-34, but the partial index narrowing is the cheap part — id is the BIGSERIAL PK that scales linearly. Indexing on `id` keeps the partial index narrow (just the unpublished row PKs) while letting Postgres use the existing seq-based composite for the ORDER BY.

3. **CHECK extension via DROP + CREATE under the same name.** Same approach 004 used to author the constraint originally. Drops the old 4-kind CHECK then creates the 7-kind under the same `ck_agent_events_kind` name — preserves identity for downstream tooling that might reference the constraint name (currently nothing does, but future-safe).

4. **Round-trip test cleans up new-kind rows before downgrade.** Test isolation refinement, not a migration bug. The migration's downgrade contract is schema-only: after a downgrade, post-007 rows with the new kinds *should* fail the rebuilt 4-kind CHECK — that's correct prod behavior (don't silently lose schema invariants over data). The test must clean those rows to exercise the schema-only contract cleanly.

5. **Live apply via docker cp + docker exec, not host-side venv.** Plan 22c-02 already established this (commit `ec19e7f`) — Docker Desktop on macOS doesn't bridge the deploy_default network's container IPs to the host, so attempts to connect to `172.18.0.3:5432` from the host venv time out. The api_server container has bridge access to `postgres:5432` natively; copying the migration file in and running alembic from inside the container is the canonical path until CI rebuilds the image.

## Deviations from Plan

### Rule 1 — bug fixed inline

**1. [Rule 1 - Test isolation bug] Round-trip test failed when run after the DDL test under shared `pg` fixture.**

- **Found during:** Task 1 GREEN — first run after writing the migration, both tests collected together
- **Issue:** `test_migration_007_creates_table_and_columns` inserts an `agent_events` row with `kind='inapp_inbound'` to assert the new CHECK accepts it. When the round-trip test then runs `alembic downgrade -1` against the SAME shared module-scoped container, the downgrade rebuilds `ck_agent_events_kind` as the 4-kind shape and the existing inapp_inbound row trips the CHECK. Alembic exits 1.
- **Root cause:** Migration is correct. The downgrade *should* fail loud if rows with new-kind values exist — that's the correct schema-invariant-preservation behavior in prod. The bug is in the test sharing a fixture between an INSERT-ing test and a DOWNGRADE-ing test.
- **Fix:** Round-trip test now `DELETE`s rows with the 3 new kinds before invoking `alembic downgrade -1`. Documented inline so the next reader understands this is test isolation, not a migration weakness.
- **Files modified:** `api_server/tests/test_migration_007.py` (round-trip test body)
- **Commit:** `b9b5004` (combined with the GREEN migration commit since the fix unblocks the migration's verification gate)
