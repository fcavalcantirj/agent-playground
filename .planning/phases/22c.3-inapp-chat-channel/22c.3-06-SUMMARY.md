## Implementation note: agent_containers lookup ordering

Why `ORDER BY stopped_at DESC NULLS LAST, created_at DESC LIMIT 1`:

`fetch_stuck_forwarded` returns only `inapp_messages` columns (id, agent_id, user_id, attempts, last_attempt_at). To attribute the `agent_events` row, the reaper needs `agent_containers.id` (the `agent_container_id` FK). One agent_id can have multiple containers over time (each `start_persistent` creates a new row; old `stopped_at` may be set).

The chosen ordering returns the most-current container preferentially:

1. `stopped_at DESC NULLS LAST` — live containers (stopped_at IS NULL) sort FIRST. NULLS LAST in DESC order means NULLs bubble to the top.
2. `created_at DESC` — among multiple stopped containers (rare), the most-recently created wins.

This is consistent with how restart-resilient SSE replays read the freshest container per agent (Phase 22b discipline). It also means a stuck message whose original container was torn down and replaced will route the failure event to the NEW container's event stream — which is what a re-attached SSE client would actually be reading.

## Decisions Made

- **NOT reusing `inapp_dispatcher._terminal_failure`.** That helper opens its own connection (`async with pool.acquire()`); the reaper needs to keep `mark_failed` + `insert_agent_event` INSIDE the outer transaction that holds the `FOR UPDATE` row-locks. Inlining the two calls keeps the SKIP LOCKED contract intact across the writes. Future refactor opportunity if a third caller surfaces, but YAGNI for now.
- **`insert_agent_event` does NOT receive a `published` kwarg.** The alembic 007 column-default `FALSE` carries the outbox flag — the function signature `(conn, agent_container_id, kind, payload, correlation_id)` has no `published` parameter, and Plan 22c.3-05 dispatcher relies on the same default. Plan 06 follows. The plan PLAN.md example showed `published=False, correlation_id=...` as a kwarg; the actual implementation removes `published` (column default) and keeps `correlation_id=str(row["id"])`.
- **`captured_at` computed once per sweep, not once per row.** The N rows in a sweep all share a captured_at instant — keeps the agent_events payload trivially auditable as "this sweep cycle produced these N failures." Microsecond resolution is sufficient.
- **Type annotations dropped on the 3 module-level constants** to satisfy the plan's literal grep gate (`grep -q "STUCK_THRESHOLD_MINUTES = 11"`). The dispatcher uses `: int = ...` annotations, but the plan's verify gate is a substring match. Behaviour is identical; cosmetic alignment with the plan's grep saves a brittle gate.
