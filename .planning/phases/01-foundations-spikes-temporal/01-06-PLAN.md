    **Spike 2: chat_io.mode per agent**

    For each agent, determine how it exposes chat I/O:

    1. Read agent entrypoint scripts and documentation
    2. Determine: Does the agent read from stdin/write to stdout? Can it be driven via named pipes? Does it expose an API? Does it use WebSocket?

    **Spike 3: tmux + named-pipe round-trip latency**

    Measure the latency of the named-pipe chat architecture (the design chosen for the chat surface):

    1. Create a Docker container with tmux installed
    2. Inside the container:
       ```bash
       mkfifo /work/.ap/chat.in /work/.ap/chat.out
       # In a tmux session, run a simple echo responder:
       # cat /work/.ap/chat.in | while read line; do echo "echo: $line"; done > /work/.ap/chat.out
       ```
    3. From outside or inside the container, write timestamped messages to chat.in and read from chat.out
    4. Measure round-trip 100 times
    5. Report p50, p95, p99 latency
    6. Pass criterion: p99 < 50ms

    If Docker is not available locally for testing, document the test methodology and mark as "pending host access" -- but attempt it locally first with `docker run -it alpine sh`.
