<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser -> Next.js -> Go API | Frontend calls Go API via fetch with credentials; cookie is the trust mechanism |
| Next.js middleware | Client-side route protection; not a security boundary (server-side auth is in Go) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1-10 | Spoofing | Client-side auth check | accept | Client-side /api/me check is for UX, not security. Server-side auth middleware in Go is the real gate. Acceptable that a user could manipulate JS to see the empty dashboard shell -- no sensitive data is displayed. |
| T-1-11 | Information Disclosure | API proxy in next.config.ts | accept | Proxy only forwards to localhost:8080 in dev; production will use same-origin deployment. No secrets in proxy config. |
| T-1-12 | Tampering | CSRF on /api/dev/login | mitigate | Cookie is SameSite=Lax; POST from cross-origin is blocked by browser. Dev-only endpoint further reduces risk. |
</threat_model>
