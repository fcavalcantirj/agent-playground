# Phase 22c.3.1: Runner-side inapp wiring — Research

**Researched:** 2026-04-30
**Domain:** Plumbing — port the e2e harness's `_factory` substitution+minting+pre_start discipline into the production route+runner codepath, then close the loop by swapping `_factory` to call the route.
**Confidence:** HIGH (single-file reference impl; substitution dict, minting, pre_start loop all already work in `_factory` against real infra)

---

## Summary

CONTEXT.md already locks 28 decisions. The code-grounded shape of the work is small and well-scoped: ~80 LOC into `tools/run_recipe.py::run_cell_persistent` (channel-aware argv override + activation_env merge + pre_start loop + placeholder substitution), ~50 LOC into `start_agent` (token mint + post-running UPDATE), and a partial rewrite of `tests/e2e/conftest.py::_factory` to issue an `httpx` POST against an in-process FastAPI ASGI app instead of shelling out to docker. The pieces all individually exist and run green today — they're just on the wrong side of the route boundary.

The residual gaps the planner needs are (a) the `started_api_server` fixture **does not exist yet** — the e2e suite has never spawned a real api_server; conftest.py builds containers directly via subprocess without going through any HTTP layer, so Task 3 is not "swap one call" but "introduce an ASGI/lifespan-driven HTTP harness for the e2e directory and rewire 5/5 cells through it"; (b) several real production failure modes the e2e harness currently masks (channel-input encryption with empty dict, watcher_service spawn for inapp channels, run_watcher's `chat_id_hint` lookup against a non-Telegram input shape, OpenRouter key resolution from session vs. env, log redaction of the new INAPP_AUTH_TOKEN); (c) Macos Docker Desktop bind-host quirks the harness papers over (`_PER_RECIPE_E2E_BIND_OVERRIDES`, the `127.0.0.1 → 0.0.0.0` argv rewrite, the host-port-publish workaround) — these need an opinionated decision because the production route does NOT publish ports (dispatcher reaches via bridge IP) so the harness's macOS-only workaround cannot be ported verbatim.

**Primary recommendation:** Wave 0 = introduce a session-scoped `started_api_server` fixture in `tests/e2e/conftest.py` that boots `create_app()` via `app.router.lifespan_context` against the e2e docker network + already-spawned testcontainers PG/Redis, exposes an `httpx.AsyncClient` over `ASGITransport`, and seeds an authenticated session cookie. Then Task 1 (runner) → Task 2 (route) → Task 3 (rewire `_factory` to POST through the new fixture). The phase is mechanical once that scaffold exists.

## Risks & Pitfalls

> Code-grounded risks the planner must not paper over. Each cites a file:line.

### 1. `data_dir` lifetime mismatch — pre_start needs the volume BEFORE persistent run
**What:** Current `run_cell_persistent` allocates `data_dir = Path(tempfile.mkdtemp(...))` at line 1098, AFTER env-file assembly at line 1075. The pre-start loop must mount that same `data_dir` because zeroclaw's 4 pre-starts write to `/zeroclaw-data/.zeroclaw/config.toml` and the daemon reads it.
**Why it matters:** Naively inserting the loop where `_factory` does (`conftest.py:483`) will reference an unallocated `data_dir`.
**Mitigation:** Hoist `data_dir = mkdtemp(...)` and `env_file.write_text(...)` to BEFORE the new pre-start loop. Order:
  1. Validate recipe + channel
  2. Build substitutions dict; render argv/entrypoint/activation_env
  3. Allocate data_dir
  4. Write env-file (legacy + cred + activation_env overlay)
  5. **NEW:** Run pre_start_commands loop (each docker run --rm with same env-file + same data_dir mount)
  6. docker run -d for the persistent container
  7. Poll ready_log_regex
  8. Health check + return

### 2. Pre-start cleanup on failure
**What:** When a pre-start command fails (D-25/D-26), the persistent container has not started — but `data_dir` and `env_file` ARE allocated, and previous pre-start commands may have completed (e.g., zeroclaw's onboard succeeds, then `config set` fails because flag name changed). State on disk is non-empty.
**Why it matters:** D-25 says "fail-fast" but doesn't specify cleanup. If `_cleanup(env_file, data_dir)` is not called, /tmp leaks.

### 3. Substitution leakage into telegram path

### 4. `start_agent` call ordering for the post-running UPDATE
**What:** The natural place to UPDATE `agent_containers.inapp_auth_token` is AFTER `write_agent_container_running` (line 432-438) but BEFORE the watcher spawn (line 495-508). However, the route is already at "Step 8b" with no DB connection acquired between Step 8 and Step 9. Adding a third `pool.acquire()` block is fine but introduces a window where (a) the row is `running`, (b) `inapp_auth_token` is NULL, (c) the watcher MAY spawn before the UPDATE lands.

### 6. env-file overlap when activation_env declares OPENROUTER_API_KEY
**What:** The runner currently writes `f"{api_key_var}={api_key_val}"` as line 1 (`run_recipe.py:1076`). D-24 says "activation_env wins"; verify with a unit test that asserts the rendered file's LAST line for a colliding key is the activation_env value.

### 7. INAPP_AUTH_TOKEN as a fresh secret in redaction surfaces

### 8. Boot timeout budget vs. pre-start budget

**Why it matters:** D-25's 120s/cmd is per-command. Total pre-start time can be up to `4 * 120 = 480s` for zeroclaw — far over the 180s default boot_timeout_s.
**Mitigation:** Either (a) reset `t0` after the pre-start loop completes (clean — boot_timeout_s applies only to the persistent container's ready_log poll), OR (b) document that callers must pass `boot_timeout_s >= sum(pre_start_timeouts) + persistent_boot_budget`. Option (a) is correct. Update boot_wall_s reporting to either separate `pre_start_wall_s` + `persistent_boot_wall_s` or sum them — CONTEXT.md doesn't dictate, planner must decide.

### 9. The `--user` flag and pre-start commands
**What:** zeroclaw's distroless image has USER 65534. Pre-start commands need to write to `/zeroclaw-data/.zeroclaw/config.toml` — they MUST run as the same UID or the volume isn't writable. `_factory` handles this via `if user_override: pre_run_cmd += ["--user", str(user_override)]` at `conftest.py:502-503`, but ONLY when `inapp.persistent_argv_override.user_override` is set. The recipe declares `user_override` at the recipe-level `persistent.spec.user_override` (read at `_factory` line 378) and possibly inapp-overrides it. **zeroclaw.yaml does NOT declare `user_override` at any level** — the distroless image's USER 65534 is the default.
**Mitigation:** Pre-start `docker run` commands inherit the image's default USER unless `--user` is passed. For zeroclaw this is correct (65534). For a future recipe that needs root for pre-start (chown, mkdir etc.) and 65534 for the daemon, the recipe would need a per-pre-command user override — out of scope for this phase per D-22 (no recipe changes). **Document this limitation in the PLAN.**

### 10. Image build-cache trap on first start
**What:** `execute_persistent_start` holds `tag_lock` for the entire boot, including `ensure_image` (the first start on a cold cache builds/pulls). For inapp recipes that use `image_pull` (zeroclaw, nullclaw — see `recipes/zeroclaw.yaml:24-25`) the pull is ~10s; for `upstream_dockerfile` (hermes, openclaw, nanobot) it's a `git clone --depth=1 + docker build` that can take 10+ minutes uncached.

**Mitigation:** No code change. PLAN should call out: e2e gate must run on a runner with images pre-pulled, OR run a warm-up step before the matrix. The CI configuration is out of scope for this phase per D-22.

## Production Failure Modes the E2E Harness Masks

> What can break in production that 5/5 PASS today does NOT catch. The planner should ensure Task 2 tests cover these explicitly.

### 1. Watcher spawn AttributeError on `event_log_regex`-less inapp recipes
**Path:** `start_agent` line 497-508 calls `run_watcher(..., recipe=recipe, channel=body.channel)`. The watcher at `watcher_service.py:516-523` calls `_compile_regexes(recipe, channel)` which reads `recipe.channels.<channel>.event_log_regex`. For all 5 inapp recipes this field is absent — currently the watcher emits a warning and idles (line 519-523).
**Why the harness masks it:** The harness never calls the route; the watcher never spawns.
**Risk:** If a future refactor of `_compile_regexes` raises (KeyError, etc.) instead of returning empty, every inapp deploy fails. Defensive — but the warning means inapp containers spawn 1 idle asyncio.Task each. Operators should see this as `inapp watcher idle for container <id>` in logs.
**Coverage needed:** Task 2 should assert the route returns 200 (i.e. watcher spawn doesn't propagate exception); idle-watcher behavior is documented, not tested.

### 2. `mark_agent_container_stopped` does not clear `inapp_auth_token`
**Path:** `run_store.py::mark_agent_container_stopped` (lines 318-348) writes `container_status`, `stopped_at`, `last_error` only. It does NOT touch `inapp_auth_token`.
**Why the harness masks it:** `_factory` does its own DB seed via `_helpers.py::seed_inapp_message`; it never calls `mark_agent_container_stopped`.
**Risk:** Per D-09 "Cleared by `mark_stopped`". Currently false. A stopped container row retains its inapp_auth_token. The dispatcher's row fetch JOINs on `agent_instance_id` and respects `container_status='running'` — so a stopped row never dispatches anyway. But D-09 says cleared; either honor D-09 (extend `mark_agent_container_stopped` with a `SET inapp_auth_token = NULL` clause) or amend D-09 in CONTEXT to say "the token becomes inert when the row transitions to stopped (read paths gate on container_status)". Recommendation: extend the function — cheap, makes the column an audit-clean signal.
**Coverage needed:** Task 2 should test stop→token-cleared.

### 3. BYOK key resolution: provider_key → ANTHROPIC_API_KEY for openclaw
**Path:** Existing route at `agent_lifecycle.py:312` calls `_resolve_api_key_var(recipe, agent.model)` — for `agent.model="anthropic/...", recipe=openclaw`, this returns `ANTHROPIC_API_KEY`. The Bearer token is then written under that env var.
**Why the harness masks it:** `_factory` at `conftest.py:305-317` reads `os.environ.get("ANTHROPIC_API_KEY")` separately and falls back with a loud warning. Production has NO such fallback — it just writes the user's Bearer (which is OpenRouter today) under `ANTHROPIC_API_KEY`, producing an Anthropic 401.
**Risk:** Production openclaw inapp deploy will silently produce a 401-relay reply (the dumb-pipe behavior CONTEXT.md acknowledges as out-of-scope per D-21 / 22c.3 deferred-items.md "Per-recipe BYOK key resolution").
**Coverage needed:** Document — DO NOT fix here (deferred). But Task 2 SHOULD include a test that asserts: `start_agent(recipe=openclaw, channel=inapp, body.bearer=<openrouter_key>)` succeeds (deploy lands), even if the actual reply is a 401-relay. The phase ships dumb-pipe; the BYOK fix is a separate phase.

### 4. Lifespan pool exhaustion under serial pre-starts
**Path:** `execute_persistent_start` holds `app_state.run_semaphore` for the entire boot (`runner_bridge.py:189-190`). Pre-starts add up to 4× `docker run --rm` in series. Default semaphore size is `AP_MAX_CONCURRENT_RUNS=2` in dev, higher in prod.
**Why the harness masks it:** `_factory` holds NO semaphore; concurrent starts are unbounded.
**Risk:** With the current 2-cap, if a user starts a zeroclaw container (~70s pre-starts + 0.3s daemon boot), one of the 2 slots is held for 70+s. A second concurrent start with a different user is fine; a third blocks. Not broken, but observably slower than the harness suggests.
**Coverage needed:** None (existing behavior). Document in PLAN.

### 5. Channel config encryption with empty inputs
**Path:** `agent_lifecycle.py:322-325`: `config_plain = {"channel": "inapp", "inputs": {}}`; encrypted via `encrypt_channel_config(user_id, config_plain)`. The `age_cipher` library handles empty dicts cleanly (round-trip-tested at 22a).
**Why the harness masks it:** `_factory` doesn't go through the route, so no encryption happens.
**Risk:** Defensive — passes today. Confirm with Task 2's `test_start_inapp_empty_channel_inputs_accepted` test that the encrypted blob round-trips and `decrypt_channel_config` returns `{"channel": "inapp", "inputs": {}}`.

### 6. Race: dispatch arrives before `inapp_auth_token` UPDATE lands
**Path:** See Risks §4 above. The window is microseconds in practice (the UPDATE happens ~100ms after `write_agent_container_running` returns) but the watcher spawn at line 497 is `asyncio.create_task(...)` (fire-and-forget). The watcher itself doesn't call dispatch; the dispatcher loop (lifespan task) polls `inapp_messages` — for THAT loop to send before the UPDATE lands, a message would have to be POSTed to `/v1/agents/:id/messages` between `write_agent_container_running` returning and the UPDATE landing. Unlikely but possible.
**Mitigation:** Per Risks §4, fold the `inapp_auth_token` write into `write_agent_container_running` to close the window entirely.

### 7. Pre-start container leakage on TimeoutExpired
**Path:** D-25's `subprocess.run(..., timeout=120)` raises `subprocess.TimeoutExpired` after 120s. The `--rm` flag should auto-remove on container exit, but a hung container won't exit until `docker kill` is sent. The TimeoutExpired path needs to also `docker kill` the container.
**Why the harness masks it:** `_factory` doesn't enforce a timeout (line 505: `subprocess.run(pre_run_cmd, capture_output=True, text=True, check=False)` — no timeout kwarg).
**Mitigation:** Each pre_start command must use a `--cidfile` (mirroring `run_cell` at line 792) so the timeout handler can `docker kill <cid>` from the file. OR generate a unique container name (`f"ap-prestart-{uuid.uuid4().hex[:8]}"`) per pre-start and `docker kill` by name.

            "--entrypoint", pre_entry,
        ]
        if user_override:
            pre_cmd += ["--user", str(user_override)]
        pre_cmd += [image_tag, *pre_args]
        try:
            ex = subprocess.run(
                pre_cmd, timeout=120, capture_output=True, text=True, check=False,
            )
        except subprocess.TimeoutExpired:
            # Reap the hung container via cidfile (mirror run_cell line 853-870).
            try:
                cid = pre_cidfile.read_text().strip()
                if cid:
                    subprocess.run(["docker", "kill", cid], timeout=10, check=False)
                    subprocess.run(["docker", "rm", "-f", cid], timeout=10, check=False)
            except OSError:
                pass
            _cleanup(env_file, data_dir)
            raise RuntimeError(
                f"pre_start_command timed out after 120s: {pre_entry} "
                f"{' '.join(pre_args)[:100]}"
            )
        finally:
            try:
                pre_cidfile.unlink(missing_ok=True)
            except OSError:
                pass
        if ex.returncode != 0:
            tail = _redact_channel_creds(
                ex.stderr or "", api_key_var, api_key_val,
                required_inputs, optional_inputs, channel_creds,
            )
            _cleanup(env_file, data_dir)

                f"pre_start_command failed rc={ex.returncode}: {pre_entry} "

## Open Questions

1. **Should `mark_agent_container_stopped` clear `inapp_auth_token`?**
   - What we know: D-09 says yes ("Cleared by `mark_stopped`"); current code does not.
   - What's unclear: Whether the planner is expected to extend `mark_agent_container_stopped` (1-line SQL change) or document as deferred.
   - Recommendation: Extend. It's cheap, it honors D-09 verbatim, and it makes audit reasoning trivial.

2. **`started_api_server` fixture scope: session vs. function?**
   - What we know: Function-scoped is simpler and matches `async_client`; session-scoped saves ~15s across the 5-cell matrix.
   - What's unclear: Whether the e2e suite's startup cost target is sensitive to this.
   - Recommendation: Function-scoped. The 15s is dwarfed by the 5×30-60s container boot wall.

3. **Should pre-start `boot_wall_s` be reported separately from persistent boot?**
   - What we know: D-25 mandates fail-fast; doesn't specify telemetry.
   - What's unclear: Whether the route's response model `AgentStartResponse.boot_wall_s` should now be `pre_start_wall_s + persistent_boot_wall_s` (sum) or just persistent.
   - Recommendation: Sum, for backwards compat. Add an optional `pre_start_wall_s: float | None` field to the response — non-breaking.

4. **Cidfile vs. unique container name for pre-start reaping?**
   - What we know: `run_cell` (line 792) uses cidfile. Either approach works.
   - What's unclear: Naming convention. Cidfile mirrors existing pattern; container name is more grep-friendly in `docker ps -a`.
   - Recommendation: Cidfile — matches existing code, and pre-start containers are short-lived enough that grep visibility is irrelevant.
