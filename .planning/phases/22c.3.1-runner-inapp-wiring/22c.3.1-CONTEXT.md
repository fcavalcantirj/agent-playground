# Phase 22c.3.1 — Runner-side inapp wiring - Context

**Gathered:** 2026-05-01
**Status:** Ready for planning (skip discuss-research; deferred-items.md already enumerates the scope)

<domain>
## Phase Boundary

Close the residual runner-side gap from Phase 22c.3 — the route-handler codepath (`POST /v1/agents/:id/start` with `body.channel == "inapp"`) does not yet wire the inapp deploy flow. The dispatcher contract layer + Redis fan-out + SSE replay all proven via 5/5 e2e PASS in 22c.3, but Plan 15 took Route B: the e2e gate's `tests/e2e/conftest.py::_factory` builds containers via direct docker calls (mints `INAPP_AUTH_TOKEN`, substitutes placeholders, runs pre_start_commands) instead of calling the real route handler. This phase ports the substitution + minting discipline from `_factory` into `tools/run_recipe.py::run_cell_persistent` + `api_server/src/api_server/routes/agent_lifecycle.py::start_agent`, then swaps `_factory` to call the route handler so `make e2e-inapp` becomes the route-path proof.

**In scope:**
- `tools/run_recipe.py::run_cell_persistent` extended to consume `channels.{channel_id}.persistent_argv_override` (entrypoint + argv + pre_start_commands) and `channels.{channel_id}.activation_env` when set, falling back to global `recipe.persistent.spec.argv` when unset.
- Activation-time placeholder substitution: `${INAPP_AUTH_TOKEN}`, `${INAPP_PROVIDER_KEY}`, `${OPENROUTER_API_KEY}`, `${ANTHROPIC_API_KEY}`, `$MODEL`, `{agent_name}`, `{agent_url}` — rendered before docker invocations.
- `pre_start_commands` execution as `docker run --rm` per command, before the persistent container starts, with the workspace volume mounted (zeroclaw distroless requirement).
- `start_agent` route mints per-session `INAPP_AUTH_TOKEN` (UUID hex) when `body.channel == "inapp"`, passes through `execute_persistent_start`, persists via `UPDATE agent_containers SET inapp_auth_token = $1 WHERE id = $2` after `write_agent_container_running`.
- `tests/e2e/conftest.py::_factory` modified in place: replace docker-direct calls with a `httpx` POST to `/v1/agents/:id/start` against the spawned api_server. `make e2e-inapp` asserts 5/5 PASS via the route handler — phase exit gate.

**Out of scope (deferred to future phases):**
- Frontend "deploy inapp" button (Phase 23 Flutter or earlier web UI).
- Token rotation policy beyond never-until-stop (rotation on restart, on message, etc.).
- BYOK per-recipe key resolution (when openclaw needs ANTHROPIC_API_KEY but openrouter is the user's default — currently the e2e harness aliases via env, which the route layer would inherit; deeper plumbing is a separate phase).
- Multi-channel-per-instance (D-20 keeps the single-channel-per-instance discipline).
- New routes; new schema migrations; new redis topology — the column already exists (migration 007's `agent_containers.inapp_auth_token`).

</domain>

<decisions>
## Implementation Decisions

### Verification

- **D-01:** Modify `tests/e2e/conftest.py::_factory` in place. Replace the docker-direct invocations with a `httpx.AsyncClient` POST to `/v1/agents/:id/start` against the spawned api_server (the same `started_api_server` fixture already in use). The 5-cell matrix in `test_inapp_5x5_matrix.py` is unchanged. `make e2e-inapp` becomes the route-path proof.
- **D-02:** No new test file. Single harness, single Makefile target. The historical Route B docker-direct path is replaced — not preserved alongside.
- **D-03:** **Phase exit criterion:** `e2e-report.json` shows `passed=true` with 5/5 PASS entries AFTER `_factory` swap, AND `git grep -c "docker_client.containers.run" tests/e2e/conftest.py` returns 0 (the docker-direct invocations are gone). The 4 currently-PASS cells (hermes/nanobot/nullclaw/zeroclaw) must stay PASS; openclaw remains the documented honest-unfunded cell unless `ANTHROPIC_API_KEY` is funded at run time.

### Pre-start commands

- **D-04:** Each `channels.{channel_id}.persistent_argv_override.pre_start_commands` entry executes as `docker run --rm -v <workspace_volume>:/data <image_tag> <argv>` BEFORE the persistent container starts. State accumulates in the shared workspace volume — the persistent container reads the workspace at boot and finds it pre-configured.
- **D-05:** When the recipe declares no `pre_start_commands` (the common case — only zeroclaw uses them today), the loop is a no-op. No conditional branch in the recipe; the absence of the field is the signal.
- **D-06:** `pre_start_commands` invocations inherit the same env-file as the persistent container (so `${OPENROUTER_API_KEY}` etc. are resolved). Failures bubble up as a `RuntimeError` with the failing command in the message — the persistent container never starts.
- **D-07:** No `docker exec` post-start option. Race-prone (daemon may bind port before pre-commands finish). Rejected.
- **D-08:** No `sh -c` chain in the persistent argv. Doesn't work for distroless images (zeroclaw has no shell). Recipes that *want* sh-c (openclaw) are free to declare their argv that way explicitly, but the runner doesn't impose it.

### Token rotation

- **D-09:** `INAPP_AUTH_TOKEN` minted ONCE at `start_agent` (UUID hex, 128 bits). Persisted to `agent_containers.inapp_auth_token`. Cleared by `mark_stopped` (when the container row transitions to `stopped`). Mirrors `agent_containers.config_enc` durability.
- **D-10:** No rotation on restart, on message, or on time-window. Rotation policy is a separate phase if/when needed (loopback-only attack surface today; rotation buys little).
- **D-11:** Token format: `uuid.uuid4().hex` (32 hex chars). NOT prefixed (no `inapp_` etc.) — the `inapp_auth_token` column name carries the channel context.

### Placeholder substitution

- **D-12:** **Activation-time placeholders** (rendered before docker invocations): `${INAPP_AUTH_TOKEN}`, `${INAPP_PROVIDER_KEY}`, `${OPENROUTER_API_KEY}`, `${ANTHROPIC_API_KEY}`, `$MODEL`, `{agent_name}`, `{agent_url}`. The substitution dict is built by `start_agent` (mints token, reads bearer = INAPP_PROVIDER_KEY, derives agent_name + agent_url from the agent row) and passed to `run_cell_persistent` via a new kwarg.
- **D-13:** **Per-message placeholders** (`{message_id}`, `{prompt}`) — already handled by the dispatcher (Plan 22c.3-05). NOT in this phase's scope. The runner-side substitution only touches activation-time placeholders.
- **D-14:** Substitution helper: lift `tests/e2e/_helpers.py::render_placeholders` verbatim into `tools/run_recipe.py` (or a new `tools/_placeholders.py` helper if it benefits other recipes). Same regex, same dict-driven substitution.
- **D-15:** Both shell-style `${VAR}` and Python-format-style `{key}` substitutions supported (the existing harness uses both). Order: `${VAR}` first (envvar-style, no escapes), then `{key}` (only if `${...}` didn't consume the placeholder).

### Channel input shape

- **D-16:** For `channel: "inapp"`, `body.channel_inputs` is allowed to be `{}`. The recipes' `channels.inapp.required_user_input` is empty (verified across all 5 recipes). The existing `required_user_input` validation passes vacuously.
- **D-17:** `body.channel_inputs` for `inapp` is NOT used to populate `INAPP_AUTH_TOKEN` (platform-minted) or `INAPP_PROVIDER_KEY` (Bearer header). The user provides nothing beyond the Bearer (provider key) and the URL `agent_id`.
- **D-18:** Future-compat: if a future inapp recipe declares `required_user_input` (e.g., a per-session `MODEL_OVERRIDE`), the existing flow handles it without code changes. This phase doesn't preempt that.

### Plan structure

- **D-19:** ONE plan: `22c.3.1-01-PLAN.md`. Three tasks:
  - **Task 1:** Extend `tools/run_recipe.py::run_cell_persistent` with `channel_id`-aware argv override + `activation_env` merge + `pre_start_commands` loop + placeholder substitution. ### Boundary respect

- **D-22:** This phase does NOT touch:
  - Recipe YAML files (all 5 inapp recipes already declare the right shape — phase 22c.3 plans 10-14 baked it in).
  - Migration files (007 already has the column).
  - Dispatcher / outbox / reaper / SSE / route handlers other than `start_agent` and `_factory`.
  - The 5 recipe spike artifacts.
- **D-23:** No new alembic migration. No new compose service. No new env vars beyond what's already declared.

### env-file precedence + activation_env merge

- **D-24:** **activation_env wins (overlay).** Render `channels.{channel_id}.activation_env` AFTER the legacy `api_key_var=api_key_val` line + after the `channel_creds` entries, so duplicate keys from the recipe's explicit declaration take precedence. Same-value collisions are harmless; different-value collisions surface the recipe author's intent (e.g., a future recipe wanting to inject a different `OPENROUTER_API_KEY` value via activation_env). When `channels.{channel_id}.activation_env` is absent (the telegram common case), no overlay happens — env-file shape is byte-identical to current behavior.

### Pre-start command timeout + failure semantics

- **D-25:** **120s timeout per pre_start_command, fail-fast.** Each pre-command runs as `docker run --rm` with a `subprocess.run(..., timeout=120)`. Non-zero exit OR `subprocess.TimeoutExpired` aborts `start_agent`: the persistent container never starts, the volume is cleaned up, and the route handler returns 502 `INFRA_UNAVAILABLE` with `detail=f"pre_start_command failed: {argv[0]} {argv[1]} ... rc={rc} stderr={tail[:200]}"`. zeroclaw's `onboard --quick` is the long-pole pre-command (cold OpenRouter call, observed up to ~70s on macOS Docker Desktop); 120s gives ~50s headroom.
- **D-26:** Pre-command stderr is captured (`capture_output=True`) and the last 200 chars surface in the 502 response detail (with channel_creds + api_key_val redacted via `_redact_creds`, same discipline as the existing route handler at line 386-414). Full stderr lands in the structlog event so operators can grep without exposing it to the API caller.

### Telegram path byte-identical hard invariant

- **D-27:** The new code path is gated on `channels.{channel_id}.persistent_argv_override is not None` — when absent, fall-through to current `recipe.persistent.spec.argv` + current env-file shape with zero diffs. This is enforced by an explicit regression test in Plan 22c.3.1-01: snapshot the docker run command + env-file content for `recipes/openclaw.yaml channel: telegram` BEFORE the change (commit `f28ff5f` baseline), assert byte-identical AFTER. The test is added in Task 1 RED phase (it MUST pass on current main, which proves the snapshot baseline is correct, THEN the telegram path code change passes the same snapshot).

### agent_name + agent_url derivation (only nullclaw consumes these)

- **D-28:** `agent_name` = `agent_instances.name` column value (already populated when the user creates the agent; NOT NULL per migration 002). `agent_url` = `f"http://{agent_name}.local"` — matches the e2e harness pattern verbatim. Both are metadata only; nullclaw's A2A agent-card advertises them but never resolves the URL. Other 4 recipes don't reference these placeholders today; the substitution happens regardless (cheap), so future recipes can opt in without code changes.

### Researcher-surfaced gaps (post-RESEARCH.md)

- **D-29:** **`mark_agent_container_stopped` MUST clear `inapp_auth_token`.** Extend `api_server/src/api_server/services/run_store.py:mark_agent_container_stopped` (line ~336) with `SET inapp_auth_token = NULL` in the same UPDATE. Honors D-09's "cleared by mark_stopped" semantics verbatim. 1-line SQL change.
- **D-30:** **`started_api_server` fixture is function-scoped.** Mirrors the existing `async_client` fixture in `api_server/tests/conftest.py:207-263` (also function-scoped). The ~15s spawn overhead per test is acceptable; session-scoping would couple test isolation across cells. Used by Task 3's `_factory` swap and any new route-handler tests in Task 2.
- **D-31:** **`boot_wall_s` includes pre_start time AS A SUM**, with an optional `pre_start_wall_s` sub-field for diagnostics. The persistent container's ready-poll budget (`boot_timeout_s`, default 180s) is RESET at `t0` AFTER the pre_start_commands loop completes — so zeroclaw's ~70s pre-start doesn't steal from the 180s ready budget. The total `boot_wall_s` returned in the response includes both phases (pre_start + persistent ready) so operators see end-to-end latency.
- **D-32:** **Pre-start command cleanup uses cidfile pattern.** `docker run --rm --cidfile <tmpfile>` writes the container ID to a temp file at start. On `subprocess.TimeoutExpired`, the cidfile is read and `docker kill <cid>` is invoked, then the cidfile is unlinked. Mirrors `tools/run_recipe.py:run_cell` line 792 — same pattern, same lifecycle. Skip the kill if the cidfile is empty (container exited before writing).
- **D-33:** **INAPP_AUTH_TOKEN write is FOLDED INTO `write_agent_container_running`** (extend its kwargs with optional `inapp_auth_token: str | None = None`) — NOT a separate post-running UPDATE. Closes the microsecond race window where a chat POST between the running-UPDATE and the token-UPDATE would dispatch unauthenticated. Single transaction, atomic.
- **D-34:** **`data_dir` allocation is HOISTED to before pre_start_commands.** Currently `tools/run_recipe.py:run_cell_persistent` allocates `data_dir = Path(tempfile.mkdtemp(...))` at line 1098, AFTER env-file assembly. The new flow allocates `data_dir` AND `env_file` upfront, then runs pre_start_commands (which need both volumes mounted), then runs the persistent container. Cleanup on failure unwinds in reverse order: kill-persistent → cleanup pre_start cidfiles → unlink env_file → rmtree data_dir.
