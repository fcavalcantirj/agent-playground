## Deferred Issues

### AC-01 BLOCKER — macOS Docker Desktop networking

The e2e gate (`make e2e-inapp`) cannot reach 5/5 PASS on macOS Docker Desktop without further harness work. Three concrete root causes were investigated during Task 3:

1. **Import path bug** (Rule-1 fix landed in commit `dc69a36`): `tools/run_recipe.py::from _placeholders import render_placeholders` failed when `runner_bridge` loaded the module via `importlib.util.spec_from_file_location` because `tools/` was not on `sys.path`. Fixed by inserting `tools/` dir at the top of `run_recipe.py`.

2. **AMD-37 gate condition bug** (Rule-1 fix landed in commit `dc69a36`): the original gate `(override AND substitutions)` excluded hermes (which declares `activation_env` but no `persistent_argv_override`). Without the gate opening, hermes daemons booted with the legacy argv and never received `API_SERVER_ENABLED=true` → ready_log timeout. Extended to `(override OR activation_env) AND substitutions`. D-27 invariant preserved (telegram has neither).

3. **Network reachability** (Rule-4 architectural — out of plan scope): With both fixes above, hermes now boots correctly via the route. But the dispatcher running INSIDE the test process on the host cannot reach the bot's bridge IP (macOS Docker Desktop limitation). The original e2e harness solved this with `-p 127.0.0.1:HOST_PORT:CONTAINER_PORT` host-port-publishing. The route-driven runner does NOT publish ports (production Linux uses bridge IP from one container to another, no publish needed). Three follow-up options:
   - (a) Add an `app_state.publish_ports` flag the runner honors when present, defaulted off in prod.
   - (b) Run the test process inside a container so it shares the docker bridge.
   - (c) Use Docker Desktop's `host.docker.internal` + `docker network connect`.

Recommendation: option (a). The runner gains an opt-in `--p` block driven by a new `app_state.publish_ports: bool` set by the e2e fixture. Production Linux doesn't set it. Keeps the harness honest while keeping prod clean.

## Deviations from Plan

### Auto-fixed Issues (Rule 1 — bug fixes)

**1. [Rule 1 — Bug] Runner import path broken in production runner_bridge import**
- **Found during:** Task 3 e2e gate exercise
- **Issue:** `from _placeholders import render_placeholders` at the top of `tools/run_recipe.py` raised `ModuleNotFoundError: No module named '_placeholders'` because `runner_bridge._import_run_recipe_module()` loads the file via `importlib.util.spec_from_file_location("run_recipe", <path>)`, and `tools/` is NOT on `sys.path` in the api_server runtime context. The module loaded fine via the test harness because `tools/tests/conftest.py` adds `tools/` to `sys.path`, but the production path doesn't.
- **Fix:** At the top of `tools/run_recipe.py`, insert `Path(__file__).resolve().parent` onto `sys.path` BEFORE the import. Same trick the test conftest uses.
- **Files modified:** `tools/run_recipe.py`
- **Commit:** `dc69a36`

**2. [Rule 1 — Bug] AMD-37 gate excludes activation_env-only recipes (hermes inapp)**
- **Found during:** Task 3 e2e gate exercise (after fix #1 landed)
- **Issue:** AMD-37 was specified as `(persistent_argv_override is not None) AND (activation_substitutions is not None)`. But `recipes/hermes.yaml` declares `channels.inapp.activation_env` WITHOUT `persistent_argv_override`. Under the strict gate, the legacy path runs for hermes → activation_env is dropped on the floor → daemon never gets `API_SERVER_ENABLED=true` → ready_log_regex never matches → 240s timeout → 502. The plan's AMD-37 was conservative for safety but inadvertently broke the e2e contract for hermes.
- **Fix:** Widen the gate to `(persistent_argv_override OR activation_env) AND activation_substitutions`. When override is absent + activation_env is present, runner uses `recipe.persistent.spec.argv` (legacy substitute_argv path) but injects rendered activation_env into the env-file via `_build_env_file_content`. Telegram path STILL falls through (it has neither override nor activation_env in the channel block) — D-27 invariant preserved (3/3 telegram-invariant tests still PASS).
- **Files modified:** `tools/run_recipe.py`
- **Commit:** `dc69a36`

### Auto-fixed Issues (Rule 2 — operability)

**3. [Rule 2 — Operability] Route's execute_persistent_start logger drops exception traceback**
- **Found during:** Task 3 e2e gate exercise — `_log.error("execute_persistent_start raised", extra={...})` did NOT pass `exc_info=True`, so the actual `RuntimeError` / `ImportError` text was opaque in CI logs. This made root-cause analysis of #1 and #2 above harder than it needed to be.
- **Fix:** Add `exc_info=True` to the log.error call.
- **Files modified:** `api_server/src/api_server/routes/agent_lifecycle.py`
- **Commit:** `dc69a36`

### Architectural deferral (Rule 4 — would need user input)

**4. [Rule 4 — Architectural] macOS Docker Desktop port reachability**
- **Found during:** Task 3 e2e gate exercise (after fixes #1 and #2 landed)
- **Issue:** With the runner now correctly extending hermes inapp + the import bug fixed, hermes containers DO boot successfully via the route. But the test process running on the host can't reach the bot's docker bridge IP (macOS Docker Desktop loopback limitation). The original e2e harness side-stepped this with `-p 127.0.0.1:HOST_PORT:CONTAINER_PORT` host-port publishing. Production Linux doesn't need it (the dispatcher is in another container on the same bridge), so the route-driven runner doesn't `-p`. Cross-OS divergence requires either runner option flag, dockerized test harness, or `host.docker.internal` plumbing.
- **Documented as follow-up.** Three options outlined in "Deferred Issues" above. Recommended: opt-in `--p` block driven by `app_state.publish_ports: bool` flag.
